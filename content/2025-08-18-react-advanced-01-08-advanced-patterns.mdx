# React Router 가이드 - 고급 패턴

> React Router v6를 기반으로 한 라우팅 학습 자료  
> 출처: [React Router Documentation](https://reactrouter.com/)

## 고급 패턴

고급 패턴은 “API를 아는 수준”에서 “구조를 설계하는 수준”으로 넘어갈 때 필요한 패턴 모음이다.

앞선 글에서 라우팅의 기본, 중첩, 파라미터/쿼리, 네비게이션, 가드, 로딩/에러 경계를 다뤘다. 이 글은 그것들을 조합해 “프로젝트 규모가 커졌을 때도 버티는 라우팅 구조”를 만든다.

---

## 1. 라우트 기반 코드 스플리팅을 어디에 둘지 결정한다

코드 스플리팅은 “페이지 단위”로 하는 편이 기본이다. 하지만 `Suspense`를 앱 최상단에 두면 전환 시 매번 전체 fallback이 떠서 UX가 흔들린다.

실무에서는 다음을 자주 쓴다.

- 최상단: 전역 프레임(헤더/사이드바)은 즉시 렌더링한다.
- `Outlet` 영역: 라우트 전환이 일어나는 본문만 `Suspense`로 감싼다.

### Suspense와 Lazy Loading

```jsx
import { Suspense, lazy } from "react";
import { BrowserRouter, Routes, Route } from "react-router-dom";
const Home = lazy(() => import("./pages/Home"));
const About = lazy(() => import("./pages/About"));
const Dashboard = lazy(() => import("./pages/Dashboard"));
export default function App() {
  return (
    <BrowserRouter>
      <Suspense fallback={<LoadingSpinner />}>
        <Routes>
          <Route path="/" element={<Home />} />
          <Route path="/about" element={<About />} />
          <Route path="/dashboard" element={<Dashboard />} />
        </Routes>
      </Suspense>
    </BrowserRouter>
  );
}
function LoadingSpinner() {
  return (
    <div className="loading-spinner">
      <div>Loading...</div>
    </div>
  );
}
```

## 2. 에러 바운더리는 “복구 경로”를 포함해야 한다

에러 바운더리는 “에러를 보여준다”로 끝나면 실무에서 쓸모가 줄어든다. 사용자는 다음 중 하나를 할 수 있어야 한다.

- 다시 시도한다(리트라이)
- 안전한 화면으로 이동한다(홈/목록)
- 상태를 초기화한다(로그아웃 등)

### 에러 바운더리

```jsx
import { Component } from "react";
class ErrorBoundary extends Component {
  constructor(props) {
    super(props);
    this.state = { hasError: false, error: null };
  }
  static getDerivedStateFromError(error) {
    return { hasError: true, error };
  }
  componentDidCatch(error, errorInfo) {
    console.error("Error caught by boundary:", error, errorInfo);
  }
  render() {
    if (this.state.hasError) {
      return (
        <div>
          <h1>Something went wrong</h1>
          <p>{this.state.error?.message}</p>
          <button onClick={() => window.location.reload()}>Reload</button>
        </div>
      );
    }
    return this.props.children;
  }
}
// 사용
<BrowserRouter>
  <ErrorBoundary>
    <Suspense fallback={<LoadingSpinner />}>
      <Routes>
        <Route path="/" element={<Home />} />
      </Routes>
    </Suspense>
  </ErrorBoundary>
</BrowserRouter>;
```

## 3. 404와 리소스 없음(도메인 404)을 분리한다

라우트 404는 `path="*"`로 끝난다. 하지만 실무에서 더 자주 만나는 것은 “라우트는 맞는데 리소스가 없다”는 케이스다.

- `/users/999`는 라우트는 유효하지만 API가 404일 수 있다.
- 이때는 라우터가 아니라 데이터 레이어에서 “not found UI”를 보여줘야 한다.

### 404 Not Found

```jsx
<Routes>
  <Route path="/" element={<Home />} />
  <Route path="/about" element={<About />} />
  <Route path="*" element={<NotFound />} />
</Routes>;
function NotFound() {
  return (
    <div>
      <h1>404 - Page Not Found</h1>
      <Link to="/">Go Home</Link>
    </div>
  );
}
```

## 4. 데이터 로딩 패턴을 라우트 경계로 정리한다

데이터 로딩은 “페이지 단위”로 경계를 잡으면 단순해진다.

- 목록 페이지: 필터/정렬/페이지네이션은 쿼리 스트링으로 고정한다.
- 상세 페이지: 파라미터를 캐시 키에 포함한다.
- 전환 UX: 이전 데이터를 유지할지, skeleton으로 비울지 결정한다.

아래 코드는 “비동기 상태를 한 컴포넌트가 전부 책임지는 형태”다. 규모가 커지면 React Query로 옮기는 편이 낫다.

### 데이터 로딩 패턴

```jsx
function UserProfile() {
  const { userId } = useParams();
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);
  useEffect(() => {
    setLoading(true);
    fetch(`/api/users/${userId}`)
      .then((res) => {
        if (!res.ok) throw new Error("User not found");
        return res.json();
      })
      .then((data) => {
        setUser(data);
        setLoading(false);
      })
      .catch((err) => {
        setError(err);
        setLoading(false);
      });
  }, [userId]);
  if (loading) return <LoadingSpinner />;
  if (error) return <ErrorMessage error={error} />;
  if (!user) return <NotFound />;
  return (
    <div>
      <h1>{user.name}</h1>
      <p>{user.email}</p>
    </div>
  );
}
```

## 5. 404, 에러, 로딩의 우선순위를 규칙으로 만든다

실무에서 가장 나쁜 상태는 “에러인데 로딩이 보인다” 같은 모순이다. 우선순위를 팀 규칙으로 고정한다.

- 라우트 미스매치: 404
- 데이터 404: not found UI
- 네트워크/서버 오류: error UI + retry
- 로딩: skeleton/fallback

## Vue Router와 비교

Vue도 결국 같은 문제를 푼다. 차이는 “구성이 어디에 모이는가”다.

| 관심사 | Vue Router | React Router |
| --- | --- | --- |
| 코드 스플리팅 | 라우트에 `() => import()` | `React.lazy`/lazy route 구성 |
| 에러 경계 | 전역 핸들러 + 컴포넌트 `errorCaptured` | Error Boundary |
| 404 | 라우트 마지막에 catch-all | `path="*"` |
| 데이터 로딩 | 컴포넌트/스토어에서 로딩 상태 | 컴포넌트/React Query에서 로딩 상태 |
