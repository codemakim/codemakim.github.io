# 실전: E-Commerce 상태 설계

상태 관리 시리즈의 마지막 글이다. 앞선 5편에서 React Query, Zustand, Jotai, Redux Toolkit의 개념과 패턴을 각각 다뤘다. 이 글에서는 그것들을 하나의 실제 앱에 녹여내는 방법을 다룬다. 도구를 하나씩 익히는 것과, 실무 화면에서 역할 경계를 긋는 것은 전혀 다른 문제다.

---

## 1. 오늘 풀 문제

E-Commerce 앱을 처음 설계할 때 흔히 나타나는 실수가 있다. 상태 관리 도구를 하나로 통일하려는 시도다.

상품 목록도 Redux, 장바구니도 Redux, 필터도 Redux, 체크아웃 계산도 Redux. 처음에는 "단일 소스 오브 트루스"처럼 보인다. 그런데 몇 주 지나면 다음 문제들이 동시에 터진다.

- 상품 목록을 새로고침할 시점을 Redux가 모른다. 포커스 회복, 윈도우 재활성화, 백그라운드 갱신 — 전부 직접 구현해야 한다.
- 장바구니 필터를 URL에 넣으면 뒤로가기가 동작하지 않는다. store는 URL과 독립적으로 살아있기 때문이다.
- 체크아웃 합계 계산이 여러 컴포넌트에 흩어진다. 세금, 배송비, 쿠폰이 얽히면 파생 상태 정합성을 수동으로 맞춰야 한다.

문제의 공통 원인은 "상태의 성격"을 무시하고 도구를 통일했기 때문이다. 해결책은 역할에 맞는 도구를 각각 배치하는 것이다.

---

## 2. 정신 모델 — 도메인별 역할 분리

먼저 앱의 상태를 성격별로 분류한다.

| 도메인 | 상태 성격 | 도구 | 이유 |
|--------|-----------|------|------|
| 상품 목록/상세 | 서버 상태 | React Query | 캐시/리페치 자동화 |
| 재고 확인 | 서버 상태 | React Query | 짧은 staleTime + refetchInterval |
| 장바구니 | 전역 클라이언트 | Zustand + persist | 영속 필요, 서버 미동기화 |
| 상품 필터 | URL 상태 | useSearchParams | 공유/뒤로가기 |
| 체크아웃 계산 | 파생 상태 | Jotai atom | 의존 그래프 |
| 결제 폼 입력 중 | 로컬 상태 | useState | 임시 값, 제출 전 소멸 |

이 표를 보면 각 도구의 적용 기준이 명확해진다.

- **서버 상태**: 원본이 서버에 있고, 최신성·리페치·캐시 무효화가 필요하면 React Query.
- **전역 클라이언트 상태**: 여러 화면이 공유하고, 서버가 진실의 주체가 아닌 경우 Zustand.
- **URL 상태**: 링크 공유, 뒤로가기, 북마크가 필요하면 searchParams.
- **파생 상태**: 다른 상태로부터 계산되는 값이고 의존 그래프가 명확하면 Jotai.
- **로컬 상태**: 단일 컴포넌트에서만 필요하고, 제출 또는 언마운트와 함께 사라지면 useState.

## 3. 최소 코드 — 역할 분리 한눈에

아래는 체크아웃 페이지 컴포넌트가 5가지 레이어에서 각각 상태를 가져오는 최소 예제다.

```typescript
// 이 예제는 react-router-dom 기준이다. Next.js에서는 next/navigation을 사용한다.
import { useSearchParams } from 'react-router-dom'
import { useQuery } from '@tanstack/react-query'
import { useState } from 'react'
import { useCartStore } from '@/store/cart'
import { useAtomValue } from 'jotai'
import { totalAtom } from '@/atoms/checkout'

function CheckoutPage() {
  // URL 상태 — 카테고리 필터, 뒤로가기/공유 가능
  const [searchParams] = useSearchParams()
  const category = searchParams.get('category') ?? 'all'

  // 서버 상태 — React Query가 캐시/리페치 관리
  const { data: products, isPending, isError } = useQuery({
    queryKey: ['products', category],
    queryFn: () => fetchProducts(category),
    staleTime: 60_000,
  })

  // 전역 클라이언트 상태 — Zustand, localStorage에 영속
  const cartItems = useCartStore(s => s.items)

  // 파생 상태 — Jotai, cartStore + 쿠폰 + 배송비 의존
  const total = useAtomValue(totalAtom)

  // 로컬 상태 — 폼 제출 전 임시 값
  const [note, setNote] = useState('')

  if (isPending) return <div>불러오는 중...</div>
  if (isError) return <div>상품을 불러오지 못했다. 다시 시도한다.</div>

  return (
    <div>
      <ProductGrid products={products} />
      <CartSummary items={cartItems} total={total} />
      <textarea value={note} onChange={e => setNote(e.target.value)} />
    </div>
  )
}
```

5줄짜리 import가 역할 분리의 전부다. 각 레이어는 서로를 모른다. 장바구니가 초기화돼도 React Query 캐시는 유지된다. 필터가 URL에서 바뀌면 React Query만 리페치된다.

## 4. 실무 코드 — 각 레이어 구현

### 4-1. Zustand cartStore (immer + persist)

장바구니는 `immer`로 불변 업데이트를 단순하게 유지하고, `persist`로 localStorage에 직렬화한다. Next.js SSR 환경에서는 hydration 불일치를 방지하는 처리가 필요하다(5절에서 다룬다).

```typescript
// store/cart.ts
import { create } from 'zustand'
import { persist } from 'zustand/middleware'
import { immer } from 'zustand/middleware/immer'

interface CartItem {
  productId: string
  qty: number
  price: number
  name: string
}

interface CartStore {
  items: CartItem[]
  addItem: (item: Omit<CartItem, 'qty'>) => void
  removeItem: (productId: string) => void
  updateQty: (productId: string, qty: number) => void
  clear: () => void
}

export const useCartStore = create<CartStore>()(
  persist(
    immer((set) => ({
      items: [],

      addItem: (item) =>
        set((s) => {
          const existing = s.items.find(i => i.productId === item.productId)
          if (existing) {
            existing.qty += 1
          } else {
            s.items.push({ ...item, qty: 1 })
          }
        }),

      removeItem: (productId) =>
        set((s) => {
          s.items = s.items.filter(i => i.productId !== productId)
        }),

      updateQty: (productId, qty) =>
        set((s) => {
          const item = s.items.find(i => i.productId === productId)
          if (item) {
            if (qty <= 0) {
              s.items = s.items.filter(i => i.productId !== productId)
            } else {
              item.qty = qty
            }
          }
        }),

      clear: () => set({ items: [] }),
    })),
    { name: 'cart' }
  )
)
```

`updateQty`에서 `qty <= 0`이면 아이템을 제거한다. 수량 입력 중 0 이하가 입력되는 경계 조건을 store에서 처리하지 않으면 렌더링 측에서 방어 코드가 중복된다.

### 4-2. React Query — 상품과 재고

상품 목록과 상세는 캐시 staleTime을 다르게 가져간다. 재고처럼 짧은 주기로 변하는 데이터는 `refetchInterval`로 자동 폴링한다.

```typescript
// hooks/useProducts.ts
import { useQuery, useQueryClient } from '@tanstack/react-query'

interface Product {
  id: string
  name: string
  price: number
  category: string
}

interface ProductDetail extends Product {
  description: string
  imageUrl: string
  stockCount: number
}

async function fetchProducts(category: string): Promise<Product[]> {
  const res = await fetch(`/api/products?category=${category}`)
  if (!res.ok) throw new Error(`상품 목록 요청 실패: ${res.status}`)
  return res.json()
}

async function fetchProduct(id: string): Promise<ProductDetail> {
  const res = await fetch(`/api/products/${id}`)
  if (res.status === 404) throw new Error('존재하지 않는 상품이다')
  if (!res.ok) throw new Error(`상품 상세 요청 실패: ${res.status}`)
  return res.json()
}

async function fetchStock(productId: string): Promise<{ count: number }> {
  const res = await fetch(`/api/stock/${productId}`)
  if (!res.ok) throw new Error(`재고 요청 실패: ${res.status}`)
  return res.json()
}

export function useProducts(category: string) {
  return useQuery({
    queryKey: ['products', category],
    queryFn: () => fetchProducts(category),
    staleTime: 60_000,      // 1분간 신선
    gcTime: 5 * 60_000,     // 5분 후 가비지 컬렉션
  })
}

export function useProduct(id: string) {
  return useQuery({
    queryKey: ['product', id],
    queryFn: () => fetchProduct(id),
    staleTime: 30_000,
    enabled: Boolean(id),   // id가 없으면 쿼리 비활성화
  })
}

// 재고 확인 — 5초마다 자동 갱신
export function useStock(productId: string) {
  return useQuery({
    queryKey: ['stock', productId],
    queryFn: () => fetchStock(productId),
    staleTime: 0,              // 항상 stale
    refetchInterval: 5_000,    // 5초마다 폴링
    refetchIntervalInBackground: false, // 탭이 비활성화되면 폴링 중단
    enabled: Boolean(productId),
  })
}

// 상품 추가 후 목록 캐시 무효화에 사용
export function useProductsQueryClient() {
  const queryClient = useQueryClient()
  return {
    invalidateProducts: (category?: string) => {
      queryClient.invalidateQueries({
        queryKey: category ? ['products', category] : ['products'],
      })
    },
  }
}
```

`refetchInterval: 5_000`이 재고 5초 자동 갱신을 담당한다. `refetchIntervalInBackground: false`를 설정해두지 않으면 탭이 숨겨진 상태에서도 폴링이 계속 된다. 불필요한 요청을 방지하려면 명시적으로 꺼야 한다.

### 4-3. Jotai — 체크아웃 파생 상태

체크아웃 화면의 최종 금액은 장바구니 소계, 배송비, 쿠폰 할인의 합성이다. 각 요소가 독립 atom이고 최종값은 파생 atom으로 표현된다.

Zustand store를 atom 밖에서 읽을 때는 `useCartStore.getState()`를 사용한다. 이 방법은 Jotai atom이 React 렌더링 사이클 밖에서 store 값을 읽는 표준 패턴이다.

```typescript
// atoms/checkout.ts
import { atom } from 'jotai'
import { useCartStore } from '@/store/cart'

// 기본 입력 atom
export const couponDiscountAtom = atom<number>(0)
export const shippingMethodAtom = atom<'standard' | 'express'>('standard')

// 배송비 파생 atom
const shippingCostAtom = atom((get) => {
  const method = get(shippingMethodAtom)
  if (method === 'express') return 5_000
  return 0
})

// Zustand store를 atom 밖에서 읽기
// getState()는 React 훅이 아니므로 서버/클라이언트 모두에서 호출 가능
const cartSubtotalAtom = atom(() => {
  const items = useCartStore.getState().items
  return items.reduce((sum, i) => sum + i.price * i.qty, 0)
})

// 최종 금액 파생 atom
export const totalAtom = atom((get) => {
  const subtotal = get(cartSubtotalAtom)
  const shipping = get(shippingCostAtom)
  const discount = get(couponDiscountAtom)
  return Math.max(0, subtotal + shipping - discount)
})

// 체크아웃 요약 파생 atom (UI 표시용)
export const checkoutSummaryAtom = atom((get) => {
  const subtotal = get(cartSubtotalAtom)
  const shipping = get(shippingCostAtom)
  const discount = get(couponDiscountAtom)
  const total = get(totalAtom)
  return { subtotal, shipping, discount, total }
})
```

`totalAtom`이 `cartSubtotalAtom`, `shippingCostAtom`, `couponDiscountAtom`에 의존한다. 쿠폰을 바꾸면 `totalAtom`만 재계산된다. 이 의존 그래프를 컴포넌트 안에서 `useMemo`로 구현하면, 의존 배열 실수로 stale 값을 읽는 버그가 자주 발생한다.

### 4-4. URL 필터 — useSearchParams

필터 상태를 URL에 넣으면 뒤로가기, 링크 공유, 새로고침 후 복원이 무료로 제공된다.

```typescript
// hooks/useProductFilter.ts
// react-router-dom 기준. Next.js는 'next/navigation'에서 useSearchParams, useRouter를 가져온다.
import { useSearchParams } from 'react-router-dom'
import { useCallback } from 'react'

type SortOrder = 'price_asc' | 'price_desc' | 'newest'

interface ProductFilter {
  category: string
  sort: SortOrder
  minPrice: number | null
  maxPrice: number | null
}

export function useProductFilter() {
  const [searchParams, setSearchParams] = useSearchParams()

  const filter: ProductFilter = {
    category: searchParams.get('category') ?? 'all',
    sort: (searchParams.get('sort') as SortOrder) ?? 'newest',
    minPrice: searchParams.get('minPrice') ? Number(searchParams.get('minPrice')) : null,
    maxPrice: searchParams.get('maxPrice') ? Number(searchParams.get('maxPrice')) : null,
  }

  const setFilter = useCallback(
    (partial: Partial<ProductFilter>) => {
      // stale closure 방지: 함수형 업데이트 사용
      setSearchParams((prev) => {
        const next = new URLSearchParams(prev)
        Object.entries(partial).forEach(([key, value]) => {
          if (value == null || value === '') {
            next.delete(key)
          } else {
            next.set(key, String(value))
          }
        })
        return next
      })
    },
    [setSearchParams]
  )

  const resetFilter = useCallback(() => {
    setSearchParams(new URLSearchParams())
  }, [setSearchParams])

  return { filter, setFilter, resetFilter }
}
```

`setSearchParams`에 함수형 업데이트를 사용하지 않으면 빠르게 연속 호출할 때 이전 상태가 덮어써지는 stale closure 버그가 발생한다.

### 4-5. 결제 완료 후 초기화 순서

결제가 완료되면 세 레이어를 순서에 맞게 초기화해야 한다.

```typescript
// hooks/useCheckoutComplete.ts
import { useQueryClient } from '@tanstack/react-query'
import { useSetAtom } from 'jotai'
import { useCartStore } from '@/store/cart'
import { couponDiscountAtom, shippingMethodAtom } from '@/atoms/checkout'
import { useCallback } from 'react'

export function useCheckoutComplete() {
  const queryClient = useQueryClient()
  const clearCart = useCartStore(s => s.clear)
  const setCoupon = useSetAtom(couponDiscountAtom)
  const setShipping = useSetAtom(shippingMethodAtom)

  const handleComplete = useCallback(
    async (orderId: string) => {
      // 1. 장바구니 먼저 비운다
      //    이후 컴포넌트 리렌더링이 발생해도 빈 장바구니가 표시되도록
      clearCart()

      // 2. Jotai atom 초기화
      //    체크아웃 파생 상태가 빈 장바구니 기준으로 재계산되도록
      setCoupon(0)
      setShipping('standard')

      // 3. React Query 캐시 무효화
      //    주문 내역, 재고 등을 서버에서 새로 가져오도록
      await queryClient.invalidateQueries({ queryKey: ['orders'] })
      await queryClient.invalidateQueries({ queryKey: ['stock'] })

      // 4. 주문 상세 페이지로 이동 (라우터 의존 — 호출 측에서 처리)
      return orderId
    },
    [clearCart, setCoupon, setShipping, queryClient]
  )

  return { handleComplete }
}
```

초기화 순서가 중요하다. 장바구니를 비우기 전에 캐시를 무효화하면, 무효화로 인한 리렌더 시점에 여전히 이전 장바구니 데이터가 화면에 보인다. 장바구니 → atom → 캐시 순서로 처리하면 각 리렌더 시점에 항상 일관된 상태가 표시된다.

## 5. 망하는 패턴

### 장바구니를 React Query로 관리하기

```typescript
// 잘못된 예
function addToCart(item) {
  queryClient.setQueryData(['cart'], (old) => [...old, item])
}
```

React Query는 서버 상태를 위한 도구다. 장바구니는 서버가 진실의 주체가 아니다. `invalidateQueries`를 호출할 시점이 불명확해지고, 네트워크가 없을 때 장바구니가 사라진다. 장바구니는 Zustand + persist가 적합하다.

### 필터를 Zustand store에 넣기

```typescript
// 잘못된 예
const useFilterStore = create(() => ({
  category: 'all',
  sort: 'newest',
}))
```

store에 있는 필터는 URL에 반영되지 않는다. 링크를 공유하면 필터 없는 페이지가 열린다. 뒤로가기를 누르면 필터가 초기화된다. 필터 상태는 URL의 searchParams가 적합하다.

### 체크아웃 계산을 컴포넌트 안에서 하기

```typescript
// 잘못된 예
function CheckoutSummary({ items, coupon }) {
  const subtotal = items.reduce(...)
  const shipping = subtotal > 50000 ? 0 : 3000
  const total = subtotal + shipping - coupon
  // ...
}
```

같은 계산이 `CheckoutStep1`, `CheckoutStep2`, `OrderConfirmPage`에 중복된다. 배송비 정책이 바뀌면 세 컴포넌트를 모두 수정해야 한다. 파생 상태는 Jotai atom에 한 번만 정의한다.

### 결제 완료 후 초기화 순서 없음

```typescript
// 잘못된 예
async function handlePaymentSuccess() {
  queryClient.invalidateQueries({ queryKey: ['orders'] })
  clearCart()  // 순서 바뀜 — invalidate 리렌더 시점에 이전 장바구니 표시
}
```

`invalidateQueries`가 리페치를 트리거하면 해당 쿼리에 의존하는 컴포넌트가 즉시 리렌더된다. 이 시점에 장바구니가 아직 남아있으면 이전 데이터가 잠깐 표시된다. 장바구니 초기화를 먼저 해야 한다.

---

## 6. Vue/Nuxt 비교

같은 E-Commerce를 Vue/Nuxt로 구현할 때의 대응 관계는 다음과 같다.

| 도메인 | React 스택 | Vue/Nuxt 스택 |
|--------|-----------|---------------|
| 상품 목록 | React Query | `useAsyncData` / `useFetch` |
| 장바구니 | Zustand + persist | Pinia + `useLocalStorage` (VueUse) |
| 필터 | useSearchParams | `useRoute().query` |
| 체크아웃 파생 | Jotai atom | `computed` 체이닝 |
| 결제 폼 | useState | `ref` / `reactive` |

Vue의 `computed`는 Jotai 파생 atom과 역할이 동일하다. 의존 추적이 자동으로 이루어지고, 의존 값이 바뀔 때만 재계산된다. Nuxt의 `useAsyncData`는 React Query의 서버 캐시 레이어에 대응하며, SSR 중 데이터를 payload로 직렬화해 클라이언트에 전달하는 hydration 처리가 내장되어 있다. 두 생태계 모두 서버 상태와 클라이언트 상태의 경계를 명확히 긋는 방향으로 수렴하고 있다.

## 7. 마스터 체크

아래 세 질문에 답할 수 있으면 이 시리즈의 핵심을 실무에 적용할 수 있다.

**결제 완료 후 장바구니, React Query 캐시, Jotai atom을 어떤 순서로 초기화하는가?**

장바구니 clear → Jotai atom reset → queryClient.invalidateQueries 순서다. 장바구니를 먼저 비워야 이후 리렌더링이 발생할 때 항상 빈 장바구니를 보여준다. Jotai atom을 그다음 초기화해 파생 상태(totalAtom 등)가 빈 장바구니 기준으로 재계산되도록 한다. 마지막으로 React Query 캐시를 무효화해 주문 내역, 재고 등을 서버에서 새로 가져오도록 한다.

**SSR 환경에서 Zustand persist가 hydration 불일치를 일으키는 경우는 언제이고 어떻게 방지하는가?**

서버에서는 localStorage에 접근할 수 없으므로 persist store의 초기 상태는 비어있다. 클라이언트에서 hydration이 일어나면 localStorage에서 데이터를 읽어 store가 업데이트된다. 서버에서 렌더링한 HTML(빈 장바구니)과 클라이언트 hydration 결과(저장된 장바구니)가 달라 React가 경고를 내거나 UI가 깜빡인다. 방지 방법은 두 가지다. 첫째, `skipHydration: true` 옵션을 persist에 추가하고 컴포넌트 마운트 후 수동으로 `rehydrate()`를 호출한다. 둘째, 장바구니 렌더링 자체를 `useEffect` 이후로 미루어 서버 HTML에는 포함시키지 않는다.

```typescript
// SSR 환경에서 persist 안전하게 사용하기
// store/cart.ts
export const useCartStore = create<CartStore>()(
  persist(
    immer((set) => ({ /* 위와 동일 */ })),
    {
      name: 'cart',
      skipHydration: true,  // 서버에서 자동 hydration 건너뜀
    }
  )
)

// 클라이언트 진입 컴포넌트에서 명시적으로 rehydrate
// app/providers.tsx (클라이언트 컴포넌트)
'use client'
import { useEffect } from 'react'
import { useCartStore } from '@/store/cart'

export function Providers({ children }: { children: React.ReactNode }) {
  useEffect(() => {
    // 마운트 이후에 localStorage에서 읽음 — hydration 불일치 없음
    useCartStore.persist.rehydrate()
  }, [])

  return <>{children}</>
}
```

**상품 재고 확인을 5초마다 자동으로 갱신하려면 어떻게 하는가?**

4-2절의 `useStock`에서 구현했다. `refetchInterval: 5_000`을 쿼리 옵션에 추가하면 된다. React Query가 내부적으로 타이머를 관리하고, 컴포넌트가 언마운트되면 자동으로 정리된다. 주의할 점은 `staleTime`을 0으로 설정해야 한다는 것이다. `staleTime`이 `refetchInterval`보다 길면 폴링으로 데이터를 가져와도 신선한 것으로 취급해 UI가 갱신되지 않는다.

## 마무리

상태 관리 시리즈 6편을 통해 다음을 다뤘다.

- React Query로 서버 상태 캐시, 리페치, 뮤테이션, 낙관적 업데이트, 무한 스크롤
- Zustand로 전역 클라이언트 상태와 영속성
- Jotai로 파생 상태 그래프
- Redux Toolkit으로 예측 가능한 대규모 상태
- URL 상태로 필터와 페이지네이션

그리고 이 글에서 그 다섯 개를 하나의 실제 도메인에 배치하는 방법을 다뤘다.

핵심은 도구의 기능 목록이 아니라 상태의 성격에서 시작하는 것이다. 서버가 진실의 주체인가, 클라이언트가 진실의 주체인가. 단일 컴포넌트에서만 사는가, 여러 화면이 공유하는가. URL에 있어야 하는가, 메모리에 있어도 되는가. 이 질문에 답하면 도구 선택은 따라온다.

상태 경계를 잘 그으면 각 레이어가 서로의 문제를 떠안지 않는다. 서버 상태가 바뀌어도 장바구니는 영향받지 않는다. 필터를 바꿔도 체크아웃 계산은 흔들리지 않는다. 결제가 완료되면 세 레이어를 순서대로 정리하고 앱이 깨끗한 상태로 돌아간다.
