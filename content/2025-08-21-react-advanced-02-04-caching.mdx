# React Query 가이드 - 캐싱 전략

1~3편에서는 React Query를 설정하고, `useQuery`로 읽고, `useMutation`으로 쓰는 흐름을 다뤘다. 이제부터는 "가져온 데이터를 얼마나 믿고, 언제 다시 가져올 것인가"를 정해야 한다.

캐싱은 데이터를 오래 들고 있는 기술이 아니다. 사용자가 같은 화면을 다시 열었을 때 빠르게 보여주면서도, 서버의 최신 상태와 너무 멀어지지 않게 조절하는 기준이다.

---

## 왜 필요한가

`useQuery`를 쓰면 같은 `queryKey`의 요청은 캐시에 저장된다. 그래서 목록에서 상세로 갔다가 뒤로 돌아올 때 이전 목록이 즉시 보이고, 같은 데이터를 여러 컴포넌트가 요청해도 중복 요청이 줄어든다.

하지만 캐시는 항상 두 가지 질문을 만든다.

- 지금 화면의 데이터는 아직 믿을 수 있는가
- 서버에서 값이 바뀌었을 때 어떤 캐시를 다시 맞춰야 하는가

예를 들어 게시글 목록은 30초 정도 낡아도 큰 문제가 없을 수 있다. 반대로 결제 상태나 재고 수량은 탭을 다시 열 때마다 최신화하는 편이 안전하다. 같은 React Query를 쓰더라도 도메인에 따라 캐시 정책이 달라져야 한다.

캐싱 전략은 `staleTime` 하나를 고르는 문제가 아니다. 다음 네 가지를 함께 정하는 작업이다.

- `queryKey`로 캐시 주소를 어떻게 나눌 것인가
- 데이터가 fresh에서 stale로 바뀌는 시간을 어떻게 둘 것인가
- 화면을 떠난 캐시를 메모리에 얼마나 남길 것인가
- mutation 이후 어떤 범위를 무효화하거나 직접 수정할 것인가

---

## 원리: fresh, stale, inactive를 구분한다

React Query 캐시는 크게 세 상태로 이해하면 된다.

| 상태     | 의미                         | 화면 영향                                              |
| -------- | ---------------------------- | ------------------------------------------------------ |
| fresh    | `staleTime` 안에 있는 데이터 | 같은 키를 다시 써도 보통 새 요청을 보내지 않는다       |
| stale    | 신선도 시간이 지난 데이터    | 캐시를 보여줄 수 있지만 조건에 따라 다시 가져온다      |
| inactive | 구독 중인 컴포넌트가 없음    | 바로 삭제되지 않고 `gcTime` 동안 메모리에 남을 수 있다 |

여기서 중요한 점은 stale이 "사용하면 안 되는 데이터"가 아니라는 것이다. stale 데이터도 화면에 즉시 렌더링할 수 있다. 다만 React Query는 이 데이터를 오래된 사본으로 보고, 마운트나 포커스 복귀 같은 트리거에서 백그라운드 리페치를 시도할 수 있다.

---

## 기준: staleTime과 gcTime은 다른 질문이다

`staleTime`과 `gcTime`은 이름이 비슷해도 완전히 다른 축이다.

| 옵션        | 묻는 질문                                | 값이 길 때의 효과                             |
| ----------- | ---------------------------------------- | --------------------------------------------- |
| `staleTime` | 이 데이터를 얼마나 fresh로 볼 것인가     | 리페치가 줄지만 최신성은 늦게 맞춰질 수 있다  |
| `gcTime`    | 아무도 안 보는 캐시를 얼마나 남길 것인가 | 재방문이 빠르지만 메모리 사용 시간이 길어진다 |

```tsx
useQuery({
  queryKey: ["user", userId],
  queryFn: () => fetchUser(userId),
  staleTime: 1000 * 60,
  gcTime: 1000 * 60 * 10,
});
```

이 설정은 "사용자 상세는 1분 동안 fresh로 보고, 화면에서 사라져도 10분 동안 캐시를 메모리에 남긴다"는 뜻이다. 1분이 지나 stale이 되어도 10분 동안 메모리에 남아 있을 수 있다. 신선도와 보관 시간은 분리해서 생각해야 한다.

## 최소 예제: 목록 캐시 정책을 명시한다

게시글 목록은 자주 바뀌지만, 사용자가 몇 초 사이에 같은 목록을 여러 번 오갈 때마다 요청할 필요는 없다. 이 경우 짧은 `staleTime`과 적당한 `gcTime`을 줄 수 있다.

```tsx
import { useQuery } from "@tanstack/react-query";

interface Post {
  id: string;
  title: string;
}

async function fetchPosts(): Promise<Post[]> {
  const res = await fetch("/api/posts");
  if (!res.ok) throw new Error("failed");
  return (await res.json()) as Post[];
}

export function PostList() {
  const query = useQuery({
    queryKey: ["posts"],
    queryFn: fetchPosts,
    staleTime: 1000 * 30,
    gcTime: 1000 * 60 * 5,
  });

  if (query.isPending) return <div>loading...</div>;
  if (query.isError) return <div>error</div>;

  return (
    <ul>
      {query.data.map((post) => (
        <li key={post.id}>{post.title}</li>
      ))}
    </ul>
  );
}
```

이 코드는 다음 기준을 가진다.

- 30초 안에 같은 목록을 다시 열면 fresh 데이터로 본다.
- 목록 화면을 떠나도 5분 동안 캐시를 남긴다.
- 30초가 지나 stale이 된 뒤 다시 마운트되면 백그라운드 리페치 대상이 될 수 있다.

처음에는 이 정도로 충분하다. 모든 화면에 복잡한 정책을 넣기보다, 데이터 성격에 따라 기본값을 조금씩 조정한다.

## 실무 예제: 필터가 있는 목록의 캐시 주소

캐시는 `queryKey`로 나뉜다. 그래서 같은 게시글 목록이라도 태그나 정렬이 다르면 다른 캐시 주소를 써야 한다.

```tsx
import { useQuery } from "@tanstack/react-query";

interface Post {
  id: string;
  title: string;
}

interface PostFilters {
  tag?: string;
  sort: "latest" | "popular";
}

async function fetchPosts(filters: PostFilters): Promise<Post[]> {
  const url = new URL("/api/posts", window.location.origin);
  if (filters.tag) url.searchParams.set("tag", filters.tag);
  url.searchParams.set("sort", filters.sort);

  const res = await fetch(url.toString());
  if (!res.ok) throw new Error("failed");
  return (await res.json()) as Post[];
}

export function FilteredPostList({ filters }: { filters: PostFilters }) {
  const query = useQuery({
    queryKey: ["posts", "list", filters],
    queryFn: () => fetchPosts(filters),
    staleTime: 1000 * 30,
  });

  if (query.isPending) return <div>loading...</div>;
  if (query.isError) return <div>error</div>;

  return (
    <ul>
      {query.data.map((post) => (
        <li key={post.id}>{post.title}</li>
      ))}
    </ul>
  );
}
```

`["posts", "list", { tag: "React", sort: "latest" }]`와 `["posts", "list", { tag: "React", sort: "popular" }]`는 다른 캐시다. 필터를 키에 넣지 않으면 서로 다른 목록이 같은 캐시에 섞인다.

프로젝트가 커지면 query key 팩토리로 키 규칙을 고정한다.

```tsx
export const postKeys = {
  all: ["posts"] as const,
  lists: () => [...postKeys.all, "list"] as const,
  list: (filters: { tag?: string; sort: "latest" | "popular" }) =>
    [...postKeys.lists(), filters] as const,
  details: () => [...postKeys.all, "detail"] as const,
  detail: (postId: string) => [...postKeys.details(), postId] as const,
};
```

이렇게 만들면 무효화 범위도 읽기 쉬워진다.

```tsx
import { useQueryClient } from "@tanstack/react-query";

const queryClient = useQueryClient();

queryClient.invalidateQueries({ queryKey: postKeys.all });
queryClient.invalidateQueries({ queryKey: postKeys.detail(postId) });
```

첫 번째는 게시글 관련 캐시 전체를 stale로 만들 수 있다. 두 번째는 특정 상세 캐시만 대상으로 삼는다. mutation 이후 어디까지 다시 맞출지 판단하려면 query key 구조가 먼저 정리되어 있어야 한다.

---

## 무효화는 삭제가 아니라 다시 맞추기다

`invalidateQueries`는 보통 캐시를 바로 삭제하는 API로 이해하기 쉽다. 실제로는 해당 쿼리를 stale로 표시하고, 활성 관찰자가 있으면 리페치를 트리거하는 쪽에 가깝다.

```tsx
queryClient.invalidateQueries({ queryKey: ["posts"] });
```

mutation 이후에는 서버와 캐시를 다시 맞춰야 한다. 이때 선택지는 크게 두 가지다.

| 방식                | 적합한 상황                            | 주의점                                       |
| ------------------- | -------------------------------------- | -------------------------------------------- |
| `invalidateQueries` | 여러 목록과 상세가 함께 영향을 받을 때 | 네트워크 요청이 다시 발생한다                |
| `setQueryData`      | 단일 캐시의 작은 필드만 확실히 바뀔 때 | 관련 캐시를 빠뜨리면 화면 간 불일치가 생긴다 |

처음에는 `invalidateQueries`가 더 안전하다. 서버를 다시 읽어서 맞추기 때문이다. `setQueryData`는 사용자 경험을 크게 개선할 수 있지만, 어떤 캐시를 직접 고쳐야 하는지 명확할 때만 사용한다.

## 함정과 디버깅

### 1. `staleTime`을 캐시 삭제 시간으로 착각한다

`staleTime`이 지나도 캐시가 바로 사라지지는 않는다. 단지 fresh에서 stale로 바뀐다. 캐시가 메모리에서 언제 사라지는지는 `gcTime`이 담당한다.

### 2. 필터 값을 queryKey에 넣지 않는다

검색어, 태그, 정렬, 페이지 번호 같은 값이 서버 응답을 바꾸면 `queryKey`에도 들어가야 한다. 응답을 바꾸는 입력값이 키에 빠지면 캐시 오염이 생긴다.

### 3. 모든 mutation 뒤에 전체 캐시를 무효화한다

`queryClient.invalidateQueries()`처럼 범위 없는 무효화를 남발하면 화면 전체가 불필요하게 다시 가져와질 수 있다. query key 팩토리를 두고 도메인 단위, 목록 단위, 상세 단위로 범위를 좁힌다.

### 4. `setQueryData`로 일부 캐시만 고친다

상세 캐시는 바꿨는데 목록 캐시는 그대로면 같은 게시글의 제목이 화면마다 다르게 보인다. 직접 업데이트는 빠르지만 정합성 책임을 개발자가 가져간다.

### 5. 리페치 트리거를 화면마다 다르게 둔다

`refetchOnWindowFocus`, `refetchOnReconnect`, `refetchOnMount` 같은 옵션은 화면마다 제각각이면 사용자가 느끼는 동작이 달라진다. 전역 기본값을 정하고, 최신성이 중요한 화면에서만 예외를 둔다.

## Vue와 비교

Vue Query도 같은 캐시 모델을 쓴다. 차이는 React 컴포넌트에서 `useQuery`를 부르느냐, Vue 컴포저블에서 `useQuery`를 부르느냐다.

| 관심사      | Vue Query           | React Query         |
| ----------- | ------------------- | ------------------- |
| 캐시 주소   | `queryKey`          | `queryKey`          |
| 신선도      | `staleTime`         | `staleTime`         |
| 메모리 유지 | `gcTime`            | `gcTime`            |
| 무효화      | `invalidateQueries` | `invalidateQueries` |
| 키 규칙     | composable/모듈     | 모듈                |

Pinia에 서버 데이터를 직접 넣으면 `staleTime`, `gcTime`, 무효화 범위 같은 정책을 직접 만들어야 한다. Vue Query나 React Query를 쓰면 이 정책을 같은 어휘로 다룰 수 있다.

---

## 체크리스트

- `staleTime`은 신선도, `gcTime`은 메모리 유지 시간이라는 차이를 설명할 수 있다.
- stale 데이터가 화면에 즉시 렌더링될 수 있다는 점을 이해한다.
- 서버 응답을 바꾸는 입력값을 `queryKey`에 포함해야 하는 이유를 설명할 수 있다.
- mutation 이후 `invalidateQueries`와 `setQueryData` 중 무엇을 쓸지 판단할 수 있다.
- query key 팩토리로 무효화 범위를 좁히는 기준을 설명할 수 있다.
