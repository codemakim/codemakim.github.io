# Route Handlers

Server Action은 앱 내부 폼 mutation에 잘 맞는다. 하지만 모든 서버 로직을 Server Action으로 만들 수는 없다. 외부 서비스가 호출해야 하거나, HTTP 메서드와 헤더를 직접 다뤄야 하거나, 웹훅 서명을 검증해야 하면 Route Handler가 필요하다.

Route Handler는 App Router에서 HTTP 경계를 직접 구현하는 도구다.

---

## 왜 필요한가

다음 요구사항은 Server Action보다 Route Handler가 자연스럽다.

- GitHub 웹훅을 받아 글 캐시를 무효화한다.
- 모바일 앱이 호출할 JSON API를 만든다.
- 외부 도메인에서 접근할 수 있도록 CORS를 설정한다.
- Server-Sent Events로 진행 상태를 스트리밍한다.
- 결제 서비스의 서명을 검증한다.

이 요구사항은 “앱 내부 함수 호출”이 아니라 “HTTP 계약”이다.

---

## 최소 예제

Route Handler는 `app/api/.../route.ts`에 둔다.

```ts
// app/api/health/route.ts
export async function GET() {
  return Response.json({
    ok: true,
    checkedAt: new Date().toISOString(),
  });
}
```

메서드 이름이 HTTP 메서드가 된다.

```ts
// app/api/posts/route.ts
export async function POST(request: Request) {
  const body = (await request.json()) as { title?: string };

  if (!body.title) {
    return Response.json({ message: "title은 필수다." }, { status: 400 });
  }

  const post = await db.post.create({
    data: { title: body.title },
  });

  return Response.json(post, { status: 201 });
}
```

---

## `NextRequest`와 params

Next.js 기능이 필요하면 `NextRequest`를 쓴다.

```ts
// app/api/posts/[id]/route.ts
import type { NextRequest } from "next/server";

type RouteContext = {
  params: Promise<{ id: string }>;
};

export async function GET(request: NextRequest, context: RouteContext) {
  const { id } = await context.params;
  const preview = request.nextUrl.searchParams.get("preview") === "true";

  const post = await db.post.findUnique({
    where: { id },
  });

  if (!post) {
    return Response.json(
      { message: "게시글을 찾을 수 없다." },
      { status: 404 },
    );
  }

  if (!preview && !post.published) {
    return Response.json({ message: "공개되지 않은 글이다." }, { status: 403 });
  }

  return Response.json(post);
}
```

Route Handler의 404는 HTTP 응답이다. 페이지의 `notFound()`와 역할이 다르다.

## CORS

브라우저의 다른 origin에서 호출하려면 CORS 헤더가 필요하다.

```ts
// app/api/public-posts/route.ts
const corsHeaders = {
  "Access-Control-Allow-Origin": "https://example-client.com",
  "Access-Control-Allow-Methods": "GET, OPTIONS",
  "Access-Control-Allow-Headers": "Content-Type",
};

export async function OPTIONS() {
  return new Response(null, {
    status: 204,
    headers: corsHeaders,
  });
}

export async function GET() {
  const posts = await db.post.findMany({
    where: { published: true },
  });

  return Response.json(posts, {
    headers: corsHeaders,
  });
}
```

CORS는 보안 기능이라기보다 브라우저 접근 정책이다. 인증과 인가를 대체하지 않는다.

---

## 웹훅 검증

웹훅은 신뢰할 수 없는 외부 요청이다. body를 파싱하기 전에 서명 검증이 필요한 경우가 많다.

```ts
// app/api/webhooks/cms/route.ts
import { revalidateTag } from "next/cache";
import type { NextRequest } from "next/server";
import { verifySignature } from "@/app/lib/webhooks";

export async function POST(request: NextRequest) {
  const signature = request.headers.get("x-cms-signature");
  const rawBody = await request.text();

  if (!signature || !verifySignature(rawBody, signature)) {
    return Response.json({ message: "invalid signature" }, { status: 401 });
  }

  const event = JSON.parse(rawBody) as { type: string };

  if (event.type === "post.published") {
    revalidateTag("public-posts", "max");
  }

  return Response.json({ received: true });
}
```

Route Handler에서는 `updateTag`가 아니라 `revalidateTag(tag, "max")`를 쓴다. `updateTag`는 Server Action 전용이다.

## SSE와 streaming response

Route Handler는 스트리밍 응답도 만들 수 있다.

```ts
// app/api/progress/route.ts
export async function GET() {
  const encoder = new TextEncoder();

  const stream = new ReadableStream({
    async start(controller) {
      for (const step of ["queued", "running", "done"]) {
        controller.enqueue(encoder.encode(`data: ${step}\n\n`));
        await new Promise((resolve) => setTimeout(resolve, 500));
      }

      controller.close();
    },
  });

  return new Response(stream, {
    headers: {
      "Content-Type": "text/event-stream",
      "Cache-Control": "no-cache",
    },
  });
}
```

스트리밍은 연결을 오래 유지한다. 서버리스/Edge 비용, 동시 연결 수, 프록시 타임아웃을 같이 검토해야 한다.

## Server Action과 비교

| 기준      | Server Action         | Route Handler                   |
| --------- | --------------------- | ------------------------------- |
| 주 사용처 | 앱 내부 폼/mutation   | HTTP API, 웹훅, 외부 클라이언트 |
| 호출 방식 | form/action 함수 호출 | URL + HTTP 메서드               |
| CORS      | 직접 다루지 않음      | 직접 다룸                       |
| 상태 코드 | 주로 UI 흐름          | HTTP 계약의 일부                |
| 캐시 갱신 | `updateTag` 가능      | `revalidateTag` 사용            |

둘 중 하나가 더 고급인 것이 아니다. 경계가 다르다.

---

## Static export 제약

static export에는 요청을 받을 Node.js 서버가 없다. 따라서 Route Handler는 GitHub Pages 정적 배포에서 런타임 API로 동작하지 않는다. 외부 웹훅이나 API가 필요하면 별도 서버, 서버리스 함수, Supabase Edge Functions, Vercel 배포 같은 실행 환경이 필요하다.

---

## Nuxt와 비교

Nuxt의 Nitro server routes가 Route Handler와 가장 가깝다. 둘 다 파일 기반으로 서버 엔드포인트를 만든다.

| 기준          | Nuxt/Nitro        | Next.js                           |
| ------------- | ----------------- | --------------------------------- |
| 서버 API 파일 | `server/api/*.ts` | `app/api/**/route.ts`             |
| 요청 객체     | `event`           | `Request`, `NextRequest`          |
| 응답          | 반환값/h3 helpers | `Response`, `Response.json`       |
| 캐시 무효화   | Nitro/cache 계층  | `revalidateTag`, `revalidatePath` |

---

## 체크리스트

- Route Handler를 외부 HTTP 경계로 설명할 수 있다.
- `GET`, `POST`, `PATCH`, `DELETE`, `OPTIONS` 함수를 route 파일에 구성할 수 있다.
- `NextRequest`와 params를 사용할 수 있다.
- CORS가 인증/인가 대체재가 아니라는 점을 알고 있다.
- 웹훅에서 raw body와 서명 검증 순서를 설명할 수 있다.
- Route Handler에서 `updateTag`가 아니라 `revalidateTag(tag, "max")`를 써야 하는 이유를 알고 있다.
- static export에서 Route Handler가 런타임 API로 동작하지 않는다는 점을 설명할 수 있다.
