# 블로그 포스트 작성 규칙

> 이 파일은 `content/` 디렉토리 작업 시 추가로 로드됩니다.
> (구 `.cursor/rules/blog-writing.mdc`)

---

## 언어와 말투

- 한국어로 작성한다.
- 문장 종결은 **"~한다/~된다"** 체를 사용한다.
- 존댓말("~합니다/~하세요")을 쓰지 않는다.
- 주관적 완충 표현("~같다/~라고 생각한다")을 피한다.
- 느낌표, 이모지를 쓰지 않는다.

---

## 글의 구조 (기본 템플릿)

1. **왜 필요한가** — 실무에서 어떤 문제를 해결하는지 먼저 제시
2. **원리/기준** — 동작 원리, 설계 기준, 트레이드오프 명시
3. **최소 예제** — 핵심 API 1~2개로 동작을 보여줌
4. **실무 예제** — 에러 처리, 경계 조건, 운영 이슈 포함
5. **함정/디버깅** — 자주 틀리는 포인트, 원인 분해 방법
6. **Vue 비교** — Vue/Nuxt/Vue Router/Vue Query 등 대응 개념 비교
7. **체크리스트** — 학습 체크리스트 반드시 포함

이미 앞 글에서 다룬 내용은 짧게 연결하고, 이번 글의 핵심에 집중한다.

---

## 코드 예제 규칙

- "그대로 옮겨도 동작하는 형태"로 작성 (필요한 import 포함)
- 생략이 필요하면 생략 이유와 범위를 명시
- stale closure가 생기는 패턴(상태 업데이트, URL 업데이트 등)은 안전한 형태로 작성
- 경계 조건(빈 값, 404/401, 네트워크 실패) 최소 1개 이상 포함

---

## 테스팅/성능 글의 추가 목표

### 테스팅

- 테스트 레이어 구분: 단위/통합/컴포넌트/엔드투엔드
- 실패 메시지가 어떤 계약을 의미하는지 설명
- flaky를 "결정성 결여"로 정의하고 반복 실행/분류/처방을 적음
- mocking은 "격리"와 "관측 가능성" 관점에서 설명

### 성능

- Lab(재현)과 Field(실사용) 측정 구분
- 지표(LCP/CLS/INP 등)를 "점수"가 아니라 "원인 후보"로 해석
- 원인 분류 프레임: CPU(렌더/계산) / 네트워크(워터폴) / 레이아웃(CLS) / 입력 지연(INP)
- "측정 → 가설 → 변경 → 재측정" 루프 따름

## Frontmatter 규칙

```yaml
---
title: "포스트 제목"        # 필수, 이모지 금지
date: "2025-09-15"         # 필수
description: "포스트 설명"  # 권장
tags: ["Next.js", "블로그"] # 권장
series: "시리즈명"          # 시리즈인 경우
seriesOrder: 1             # 시리즈인 경우
---
```

- 시리즈 글 제목 형태: `[시리즈명 n/총] 제목`

---

## 수정 작업 후 확인사항

- `npm run build` 통과 여부
- `series/seriesOrder`가 같은 시리즈 내에서 일관되는지
- 앞 글과 중복 설명이 과도하지 않은지
