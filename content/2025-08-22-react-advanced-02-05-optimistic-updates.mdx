# React Query 가이드 - 낙관적 업데이트

4편에서는 캐시가 서버 상태의 사본이라는 점을 다뤘다. 이번 글은 그 사본을 서버 응답보다 먼저 바꾸는 경우를 다룬다.

낙관적 업데이트는 사용자가 버튼을 눌렀을 때 서버 응답을 기다리지 않고 UI를 먼저 바꾸는 패턴이다. 좋아요, 체크박스, 북마크처럼 결과가 거의 예측 가능하고 실패 시 원복이 쉬운 작업에서 체감 속도를 크게 높인다.

---

## 왜 필요한가

좋아요 버튼을 예로 든다. 가장 단순한 구현은 서버 응답이 끝난 뒤 캐시를 무효화한다.

```tsx
const mutation = useMutation({
  mutationFn: toggleLike,
  onSuccess: () => {
    queryClient.invalidateQueries({ queryKey: ["post", postId] });
  },
});
```

이 방식은 안전하다. 서버가 성공한 뒤 다시 읽기 때문이다. 하지만 네트워크가 느리면 사용자는 버튼을 눌렀는데 아무 변화가 없는 것처럼 느낀다.

낙관적 업데이트는 이 순서를 바꾼다.

1. 사용자가 클릭한다.
2. 캐시를 먼저 바꿔 UI를 즉시 업데이트한다.
3. 서버 요청을 보낸다.
4. 실패하면 이전 캐시로 되돌린다.
5. 마지막에는 서버 데이터로 다시 맞춘다.

핵심은 "빠르게 보이게 한다"가 아니라 "실패했을 때 되돌릴 수 있는 변경만 먼저 보여준다"이다.

---

## 기준: 언제 써도 되는가

낙관적 업데이트는 모든 mutation에 쓰는 패턴이 아니다. 다음 조건이 강할수록 안전하다.

| 적합한 작업          | 이유                                         |
| -------------------- | -------------------------------------------- |
| 좋아요 토글          | 변경 결과가 단순하고 롤백이 쉽다             |
| 체크리스트 완료 토글 | 성공 결과를 클라이언트가 거의 예측할 수 있다 |
| 북마크 추가/해제     | 실패해도 이전 상태로 되돌리면 자연스럽다     |
| 임시 댓글 추가       | pending 상태를 붙여 구분할 수 있다           |

반대로 다음 작업은 조심해야 한다.

- 결제, 환불, 정산처럼 실패 비용이 큰 작업
- 서버가 가격, 권한, 재고를 크게 보정하는 작업
- 실패 이유를 사용자가 반드시 확인해야 하는 승인 워크플로우
- 여러 사용자가 동시에 수정해 충돌 가능성이 큰 편집 화면

이 경우에는 낙관적 업데이트보다 `mutation.isPending`으로 진행 중 상태를 명확히 보여주는 편이 낫다.

---

## 원리: cancel, snapshot, update, rollback, invalidate

React Query의 낙관적 업데이트는 보통 다음 순서로 작성한다.

| 단계            | 역할                                                |
| --------------- | --------------------------------------------------- |
| `cancelQueries` | 진행 중인 리페치가 낙관적 변경을 덮어쓰지 않게 한다 |
| snapshot        | 실패 시 되돌릴 이전 캐시를 저장한다                 |
| `setQueryData`  | 캐시를 즉시 바꿔 UI를 먼저 업데이트한다             |
| rollback        | 요청 실패 시 snapshot으로 되돌린다                  |
| invalidate      | 성공/실패와 무관하게 서버 진실로 다시 맞춘다        |

이 순서를 지키면 "빠른 UI"와 "서버 정합성" 사이의 균형을 잡을 수 있다.

## 최소 예제: todo 완료 토글

체크박스 토글은 낙관적 업데이트를 설명하기 좋은 예제다. 서버 응답을 기다리지 않고 체크 상태를 먼저 바꾸되, 실패하면 이전 목록으로 되돌린다.

```tsx
import { useMutation, useQueryClient } from "@tanstack/react-query";

type Todo = { id: string; title: string; completed: boolean };

async function updateTodo(todo: Todo): Promise<Todo> {
  const res = await fetch(`/api/todos/${todo.id}`, {
    method: "PUT",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify(todo),
  });
  if (!res.ok) throw new Error("failed");
  return (await res.json()) as Todo;
}

export function TodoItem({ todo }: { todo: Todo }) {
  const queryClient = useQueryClient();

  const mutation = useMutation({
    mutationFn: updateTodo,
    onMutate: async (nextTodo) => {
      await queryClient.cancelQueries({ queryKey: ["todos"] });
      const previousTodos = queryClient.getQueryData<Todo[]>(["todos"]);

      queryClient.setQueryData<Todo[]>(["todos"], (old) => {
        const prev = old ?? [];
        return prev.map((t) => (t.id === nextTodo.id ? nextTodo : t));
      });

      return { previousTodos };
    },
    onError: (_err, _nextTodo, context) => {
      if (context?.previousTodos) {
        queryClient.setQueryData(["todos"], context.previousTodos);
      }
    },
    onSettled: () => {
      queryClient.invalidateQueries({ queryKey: ["todos"] });
    },
  });

  return (
    <label>
      <input
        type="checkbox"
        checked={todo.completed}
        onChange={() =>
          mutation.mutate({ ...todo, completed: !todo.completed })
        }
        disabled={mutation.isPending}
      />
      {todo.title}
    </label>
  );
}
```

`onMutate`가 반환한 값은 `onError`의 `context`로 전달된다. 이 예제에서는 이전 todo 목록을 저장해 두었다가 실패하면 그대로 복원한다.

`onSettled`에서 무효화하는 이유는 낙관적 업데이트가 임시값이기 때문이다. 서버가 `updatedAt`, 권한, 정렬 순서 같은 값을 보정할 수 있으므로 마지막에는 서버 결과로 다시 맞춘다.

## 실무 예제: 좋아요 버튼

좋아요는 토글과 카운트가 함께 바뀐다. 낙관적 업데이트를 적용하되, 요청 중에는 버튼을 비활성화해 연속 클릭 경쟁 상태를 줄인다.

```tsx
import { useMutation, useQuery, useQueryClient } from "@tanstack/react-query";

type Post = { id: string; likes: number; isLiked: boolean };

async function fetchPost(postId: string): Promise<Post> {
  const res = await fetch(`/api/posts/${postId}`);
  if (!res.ok) throw new Error("failed");
  return (await res.json()) as Post;
}

async function toggleLike(postId: string): Promise<void> {
  const res = await fetch(`/api/posts/${postId}/toggle-like`, {
    method: "POST",
  });
  if (!res.ok) throw new Error("failed");
}

export function LikeButton({ postId }: { postId: string }) {
  const queryClient = useQueryClient();
  const postQuery = useQuery({
    queryKey: ["post", postId],
    queryFn: () => fetchPost(postId),
    staleTime: 1000 * 30,
  });

  const mutation = useMutation({
    mutationFn: () => toggleLike(postId),
    onMutate: async () => {
      await queryClient.cancelQueries({ queryKey: ["post", postId] });
      const previous = queryClient.getQueryData<Post>(["post", postId]);

      queryClient.setQueryData<Post>(["post", postId], (old) => {
        if (!old) return old;
        const nextLiked = !old.isLiked;
        return {
          ...old,
          isLiked: nextLiked,
          likes: old.likes + (nextLiked ? 1 : -1),
        };
      });

      return { previous };
    },
    onError: (_err, _vars, context) => {
      if (context?.previous) {
        queryClient.setQueryData(["post", postId], context.previous);
      }
    },
    onSettled: () => {
      queryClient.invalidateQueries({ queryKey: ["post", postId] });
    },
  });

  if (postQuery.isPending) return <button disabled>loading...</button>;
  if (postQuery.isError) return <button disabled>error</button>;

  const post = postQuery.data;

  return (
    <button onClick={() => mutation.mutate()} disabled={mutation.isPending}>
      {post.isLiked ? "liked" : "like"} ({post.likes})
    </button>
  );
}
```

이 예제는 상세 캐시 `["post", postId]`만 직접 바꾼다. 만약 게시글 목록에도 좋아요 수가 표시된다면 목록 캐시도 함께 업데이트하거나, 목록을 무효화해야 한다. 직접 업데이트 범위를 확신하기 어렵다면 `invalidateQueries({ queryKey: ["posts"] })`처럼 서버에서 다시 읽는 편이 안전하다.

## 함정과 디버깅

### 1. `cancelQueries`를 생략한다

낙관적 업데이트 직후에 백그라운드 리페치가 끝나면 캐시가 이전 서버 응답으로 덮일 수 있다. `onMutate`에서 관련 쿼리를 먼저 취소한다.

### 2. rollback 스냅샷을 만들지 않는다

요청이 실패했는데 바뀐 UI가 그대로 남으면 사용자는 서버에 반영됐다고 믿게 된다. 낙관적 업데이트는 항상 되돌릴 기준이 있어야 한다.

### 3. 마지막 무효화를 생략한다

낙관적 업데이트가 성공했더라도 서버가 응답 값을 보정했을 수 있다. `onSettled`에서 관련 쿼리를 무효화해 서버 진실로 다시 맞춘다.

### 4. 목록과 상세 캐시 중 하나만 고친다

같은 데이터가 목록과 상세에 동시에 있으면 둘 중 하나만 바꿨을 때 화면 간 불일치가 생긴다. 직접 업데이트할 캐시 범위를 명확히 적거나, 범위가 넓으면 무효화를 선택한다.

### 5. 위험한 작업에 낙관적 업데이트를 쓴다

결제나 권한 변경처럼 실패 비용이 큰 작업은 빠른 착시보다 명확한 pending/failed 상태가 중요하다. 낙관적 업데이트는 사용자가 원복을 자연스럽게 받아들일 수 있는 작업에 제한한다.

## Vue와 비교

Vue Query도 같은 라이프사이클을 제공한다. React의 `useMutation`과 Vue Query의 mutation 옵션은 개념이 거의 같다.

| 관심사          | Vue Query                  | React Query                |
| --------------- | -------------------------- | -------------------------- |
| 낙관적 업데이트 | `onMutate`                 | `onMutate`                 |
| 실패 롤백       | `onError` + context        | `onError` + context        |
| 최종 재동기화   | `onSettled` + invalidate   | `onSettled` + invalidate   |
| 캐시 직접 수정  | `queryClient.setQueryData` | `queryClient.setQueryData` |

Pinia로도 비슷한 UI를 만들 수는 있다. 하지만 진행 중 요청 취소, 이전 스냅샷 저장, 실패 롤백, 최종 무효화 규칙을 모두 직접 설계해야 한다. Vue Query와 React Query를 쓰면 이 흐름을 같은 패턴으로 고정할 수 있다.

---

## 체크리스트

- 낙관적 업데이트가 적합한 작업과 위험한 작업을 구분할 수 있다.
- `cancelQueries → snapshot → setQueryData → rollback → invalidate` 흐름을 설명할 수 있다.
- `onMutate`가 반환한 context를 `onError`에서 rollback에 쓰는 구조를 이해한다.
- 목록과 상세 캐시가 함께 영향을 받을 때 직접 업데이트와 무효화 중 무엇을 고를지 판단할 수 있다.
- 연속 클릭 같은 경쟁 상태를 UI 비활성화나 요청 취소로 줄일 수 있다.
