# Plugins: 반복되는 하네스를 팀에 배포한다

한 프로젝트에서 좋은 Skill과 agent를 만들면 다른 프로젝트에도 쓰고 싶어진다. 처음에는 복사하면 된다. 하지만 팀 단위로 커지면 문제가 생긴다.

- 프로젝트마다 Skill 버전이 달라진다.
- reviewer agent의 출력 형식이 달라진다.
- hooks 정책이 누락된다.
- MCP 권한 기준이 사람마다 다르다.
- 새 팀원이 같은 하네스를 다시 만든다.

Plugin은 반복되는 하네스 구성요소를 배포 단위로 묶는 방법이다.

---

## 언제 필요한가

Plugin은 처음부터 만들 필요가 없다. 다음 조건이 생기면 고려한다.

- 같은 Skill을 3개 이상 프로젝트에서 쓴다.
- 팀 공통 코드 리뷰 기준이 필요하다.
- 승인 정책이나 hooks를 표준화해야 한다.
- 특정 MCP 연결을 같은 방식으로 배포해야 한다.
- 신규 프로젝트 셋업 때 반복 설정이 많다.

반대로 한 프로젝트에서만 쓰는 규칙은 plugin보다 프로젝트 Memory가 낫다. 너무 빨리 plugin으로 만들면 추상화가 앞서고 실제 작업에는 맞지 않는다.

---

## 무엇을 묶는가

Plugin 후보는 보통 네 가지다.

| 구성요소 | 예시                                   |
| -------- | -------------------------------------- |
| Skills   | bugfix, docs-rewrite, release-check    |
| Agents   | read-only reviewer, log explorer       |
| Hooks    | 완료 전 diff check, 금지 명령 차단     |
| MCP 정책 | GitHub read, docs search, DB read-only |

좋은 plugin은 도구 모음이 아니라 작업 방식 모음이다.

```txt
team-ai-harness-plugin
  skills/
    bugfix/
    code-review/
    docs-rewrite/
  agents/
    reviewer/
    explorer/
  hooks/
    pre-completion-check/
  policy/
    mcp.md
```

구성요소마다 목적과 권한이 분명해야 한다.

---

## Plugin으로 올리기 전 기준

프로젝트 안에서 검증되지 않은 절차를 바로 팀 plugin으로 올리면 안 된다. 먼저 로컬에서 반복 사용해 본다.

승격 기준은 다음 정도가 적당하다.

```txt
1. 같은 절차를 여러 번 사용했다.
2. 실패를 실제로 줄였다.
3. 프로젝트 특화 경로를 제거하거나 변수화했다.
4. 권한 경계가 명확하다.
5. 새 팀원이 읽어도 실행할 수 있다.
6. 업데이트 이력을 관리할 수 있다.
```

예를 들어 `docs-rewrite` Skill이 이 블로그의 `content/AGENTS.md` 문체에 강하게 묶여 있다면 그대로 plugin화하기 어렵다. 팀 공통 기준과 프로젝트별 기준을 분리해야 한다.

---

## 좋은 Plugin과 나쁜 Plugin

나쁜 plugin은 모든 것을 넣는다.

```txt
ai-everything-plugin
  모든 언어 규칙
  모든 프레임워크 규칙
  모든 테스트 전략
  모든 배포 절차
  모든 MCP 연결
```

이런 plugin은 컨텍스트를 크게 만들고, 프로젝트마다 충돌을 만든다.

좋은 plugin은 특정 문제를 해결한다.

```txt
code-review-plugin
  read-only reviewer agent
  review output format
  severity 기준
  forbidden review comments
```

또는:

```txt
docs-quality-plugin
  docs-rewrite skill
  frontmatter checker
  link/diff verification
```

범위가 좁을수록 유지보수와 도입이 쉽다.

---

## 버전과 변경 관리

Plugin은 팀에 배포되는 순간 제품이 된다. 업데이트가 기존 workflow를 바꿀 수 있다.

관리할 내용은 다음이다.

- 어떤 Skill이 추가되었는가
- 기존 agent 출력 형식이 바뀌었는가
- hook이 새 명령을 막는가
- MCP write 권한이 추가되었는가
- 프로젝트별 override가 가능한가

특히 권한이 넓어지는 변경은 명시적으로 리뷰해야 한다.

```txt
read-only GitHub tool
  → PR comment write 추가

이 변경은 기능 추가가 아니라 권한 확대다.
```

권한 확대는 편의 기능보다 보안 변경으로 취급해야 한다.

---

## 이 블로그 프로젝트에 적용한다면

이 프로젝트에서 plugin 후보가 될 만한 것은 다음이다.

- 한국어 기술 포스트 재작성 Skill
- MDX frontmatter 검증 hook
- code review용 read-only reviewer
- Next.js static export 검증 workflow

반대로 습관 기능의 날짜 처리 규칙이나 Spire 게임 데이터 흐름은 프로젝트 특화 규칙이다. plugin보다 `AGENTS.md`와 기능 문서에 남기는 편이 낫다.
