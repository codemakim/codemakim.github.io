# Sandbox와 Approval: 위험 행동은 실행 경로에서 막는다

AI에게 "위험한 명령은 실행하지 마"라고 말하는 것만으로는 부족하다. AI는 규칙을 읽어도 실수할 수 있고, 도구 실행은 실제 효과를 만든다.

파일 삭제, Git 리셋, 외부 DB 쓰기, GitHub push는 답변이 아니라 상태 변경이다. 안전장치는 말이 아니라 실행 경로에 있어야 한다.

---

## 왜 필요한가

코드 수정 중에는 다음 행동이 자주 등장한다.

- 파일 생성과 삭제
- 패키지 설치
- 테스트와 빌드 실행
- Git add, commit, push
- GitHub PR 작성
- DB 조회와 수정
- 브라우저 자동화

모든 행동이 같은 위험도를 갖지 않는다. `rg`로 검색하는 것과 운영 DB에 write하는 것은 완전히 다르다.

따라서 권한은 행동의 성격에 따라 나눈다.

| 등급        | 예시                            | 처리                  |
| ----------- | ------------------------------- | --------------------- |
| 안전한 읽기 | `rg`, `sed`, `git diff`         | 기본 허용             |
| 로컬 쓰기   | 작업트리 파일 수정              | 작업 범위 안에서 허용 |
| 검증 실행   | typecheck, test, build          | 보통 허용             |
| 외부 읽기   | GitHub, docs, browser           | 목적 확인 후 허용     |
| 외부 쓰기   | push, PR, DB write              | 승인 필요             |
| 파괴적 작업 | `git reset --hard`, 광범위 삭제 | 기본 금지             |

---

## Codex에서 먼저 볼 것

Codex에서는 sandbox와 approval이 1차 안전장치다. 파일 시스템 쓰기 범위, 네트워크 접근, 권한 상승 명령을 정책으로 다룬다.

작업 설계 시 먼저 묻는다.

```txt
이 명령은 읽기인가 쓰기인가?
로컬 상태만 바꾸는가 외부 상태도 바꾸는가?
되돌릴 수 있는가?
사용자 변경을 덮어쓸 수 있는가?
승인을 받아야 하는가?
```

예를 들어 `npm run build`는 로컬 검증 명령이다. 일반적으로 허용할 수 있다. 반면 `git push`는 외부 저장소 상태를 바꾼다. 사용자의 명시적 요청이나 승인 없이는 실행하지 않는 편이 맞다.

---

## Hooks의 위치

Hooks는 특정 시점에 자동으로 실행되는 검사나 차단이다. Claude Code에서는 hooks를 위험 명령 차단이나 완료 전 검증에 직접 연결할 수 있다. Codex에서도 hooks가 있지만, 처음 설계할 때는 sandbox와 approval을 먼저 보고 hooks는 그 위의 자동화 계층으로 둔다.

좋은 hooks 후보는 다음이다.

- 완료 전 `git diff --check`
- MDX 변경 후 frontmatter 검사
- 특정 위험 명령 차단
- 테스트 실패 로그 저장
- PR 작성 전 빌드 확인

나쁜 hooks는 너무 많은 일을 한다.

```txt
모든 작업 뒤에 전체 테스트, 전체 빌드, 배포까지 자동 실행
```

이런 hook은 느리고, 실패 원인을 흐리고, AI가 어떤 작업을 했는지보다 hook을 달래는 데 시간을 쓰게 만든다.

---

## 정책 예시

작은 프로젝트의 권한 정책은 이렇게 시작할 수 있다.

```md
# Tool Policy

## Allowed By Default

- Read files under the repository.
- Search with `rg`.
- Run typecheck, lint, test, and build commands.
- Edit files inside the requested scope.

## Requires Approval

- Install new packages.
- Push to remote branches.
- Create or modify GitHub PRs.
- Write to external databases.
- Run commands outside the workspace.

## Forbidden

- `git reset --hard`
- Reverting user changes without explicit request
- Deleting broad directories by pattern
- Writing secrets into source files
```

정책은 도구 이름보다 결과를 기준으로 써야 한다. 같은 `git`이라도 `git diff`는 읽기이고 `git push`는 외부 쓰기다.

---

## 검증도 권한 설계다

검증 명령은 안전장치다. 하지만 모든 변경에 항상 전체 검증을 돌리면 비용이 커진다.

작업 유형별로 나눈다.

| 작업                 | 최소 검증                                 |
| -------------------- | ----------------------------------------- |
| 문서만 수정          | `git diff --check`, frontmatter 확인      |
| MDX 포스트 수정      | format, Contentlayer/build 필요 여부 확인 |
| TypeScript 코드 수정 | `npx tsc --noEmit`                        |
| 런타임 코드 수정     | typecheck 후 `npm run build`              |
| UI 수정              | 빌드 + 브라우저 확인                      |

중요한 것은 "검증하겠다"가 아니다. 어떤 명령을 왜 실행하는지 정하는 것이다.

---

## 함정

가장 흔한 함정은 읽기와 쓰기를 섞는 것이다.

```txt
GitHub PR 상태를 봐줘.
```

이 요청은 읽기다. 그런데 AI가 바로 branch를 push하거나 PR을 수정하면 권한 경계가 깨진다.

또 다른 함정은 승인 피로다. 모든 명령마다 승인을 요구하면 사용자는 그냥 승인하게 된다. 승인은 위험한 행동에만 걸어야 의미가 있다.
