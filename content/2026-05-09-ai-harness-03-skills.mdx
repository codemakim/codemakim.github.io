# Skills: 반복 작업을 절차로 만든다

Memory는 항상 적용되는 규칙이다. 하지만 모든 절차를 Memory에 넣으면 금방 무거워진다.

버그 수정, 문서 재작성, 코드 리뷰, 릴리스 점검은 서로 다른 순서를 가진다. 이 절차를 매번 프롬프트로 길게 쓰면 빠뜨리기 쉽다. Skill은 이런 반복 작업을 필요할 때만 불러오는 작업 설명서로 만든다.

---

## 왜 필요한가

버그 수정을 예로 보자.

나쁜 흐름은 이렇다.

```txt
요청 받음
  → 관련 있어 보이는 파일 수정
  → 테스트 실행
  → 실패하면 다른 곳 수정
  → 완료라고 보고
```

문제는 재현이 없다. 원래 실패를 확인하지 않았기 때문에 수정이 맞는지 알 수 없다.

좋은 흐름은 다르다.

```txt
요청 받음
  → 증상 정리
  → 재현 명령 확인
  → 원인 후보 분리
  → 최소 수정
  → 원래 실패한 명령 재실행
  → 회귀 확인
```

이 순서는 매번 필요하지만 모든 작업에 필요한 것은 아니다. 그래서 Skill이 맞다.

---

## Skill에 들어갈 것

좋은 Skill은 짧고 강하다. 배경 설명보다 실행 순서가 중요하다.

```md
# Bug Fix Skill

Use when fixing a reported bug.

## Steps

1. Reproduce the bug or identify the missing reproduction.
2. Locate the smallest responsible code path.
3. Make one focused change.
4. Run the original failing command again.
5. Run nearby regression checks.

## Do Not

- Do not rewrite unrelated code.
- Do not claim fixed without verification output.
- Do not hide a failing test by changing the assertion first.

## Output

- Root cause
- Files changed
- Verification command and result
```

Skill은 "좋은 개발자가 되어라"라고 말하지 않는다. 어떤 순서로 움직일지 정한다.

---

## 문서 재작성 Skill 예시

이 글을 쓰는 작업도 Skill 후보다. 문서 재작성은 단순한 문장 수정이 아니라 구조 변경이다.

```md
# Docs Rewrite Skill

Use when rewriting a blog post or guide.

## Steps

1. Read the source note and target post rules.
2. Identify the reader and the job-to-be-done.
3. Replace summary-first structure with problem-first structure.
4. Keep examples runnable or clearly marked as templates.
5. Add pitfalls and checklist.
6. Verify frontmatter and formatting.

## Style

- Korean `~한다` tone.
- Avoid filler and motivational language.
- Prefer concrete commands, file paths, and decision criteria.

## Completion

- Run `git diff --check`.
- For MDX changes, run the project formatting/build command when required.
```

이 절차가 있으면 AI는 "분량을 늘리는 재작성"이 아니라 "독자가 바로 써먹는 재작성"을 목표로 삼기 쉽다.

---

## Skill과 Memory의 경계

구분 기준은 적용 빈도다.

| 질문                     | 둘 위치   |
| ------------------------ | --------- |
| 모든 작업에 필요한가     | Memory    |
| 특정 작업에서만 필요한가 | Skill     |
| 긴 배경 설명인가         | docs      |
| 한 번만 쓰는가           | 작업 노트 |

예를 들어 "사용자 변경을 되돌리지 말라"는 모든 작업의 안전 규칙이다. Memory에 둔다.

"버그 수정 전 반드시 재현한다"는 버그 수정 작업에서 강하다. Skill에 둔다.

"이 기능의 데이터베이스 스키마"는 긴 배경이다. docs에 둔다.

---

## Skill을 너무 크게 만들지 않는다

Skill이 커지면 AI가 절차보다 배경에 끌린다. 하나의 Skill은 하나의 작업 유형을 다루는 편이 낫다.

나쁜 묶음:

```txt
frontend-development-skill
  디자인
  접근성
  상태 관리
  테스트
  배포
  성능
  코드 리뷰
```

좋은 분리:

```txt
frontend-implementation
frontend-visual-qa
frontend-performance-debugging
code-review
release-check
```

작업자가 지금 어떤 절차를 따라야 하는지 분명해야 한다.

---

## 실무 적용 순서

처음 만들 Skill은 거창할 필요가 없다. 최근에 반복해서 실패한 작업 하나를 고른다.

추천 후보는 다음이다.

- bugfix
- code-review
- docs-rewrite
- release-check
- frontend-visual-qa

각 Skill은 다음 질문에 답하면 된다.

```txt
언제 쓰는가?
처음에 무엇을 읽는가?
무엇을 먼저 확인하는가?
무엇을 절대 하지 않는가?
완료 증거는 무엇인가?
출력 형식은 무엇인가?
```

이 여섯 질문에 답하지 못하면 Skill이 아니라 긴 프롬프트 조각일 가능성이 높다.

---

## 체크리스트

- [ ] Skill이 작업별 절차라는 점을 설명할 수 있다.
- [ ] Memory에 둘 규칙과 Skill에 둘 절차를 구분할 수 있다.
- [ ] 버그 수정 Skill에 재현, 원인 분리, 검증 단계를 넣을 수 있다.
- [ ] 문서 재작성 Skill에 독자, 구조, 예시, 검증 기준을 넣을 수 있다.
- [ ] 너무 큰 Skill을 작은 작업 유형별 Skill로 나눌 수 있다.
