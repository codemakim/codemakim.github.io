# Proxy

Next.js 16에서는 기존 Middleware로 설명되던 요청 앞단 훅이 Proxy라는 이름으로 이동하는 흐름이다. 이름이 바뀐 이유를 기능 목록보다 먼저 이해해야 한다. 이 계층은 애플리케이션 내부 로직이 아니라 네트워크 앞단의 얇은 분기 지점이다.

Proxy는 요청이 route에 도달하기 전에 실행된다. 인증 리다이렉트, locale rewrite, A/B 테스트처럼 빠르고 작은 결정을 내리는 데 맞다. 무거운 DB 조회나 비즈니스 로직을 넣는 곳이 아니다.

---

## 왜 필요한가

관리자 페이지에 접근하는 요청을 생각한다.

- 로그인하지 않은 사용자는 `/login`으로 보낸다.
- 이미 로그인한 사용자는 그대로 통과시킨다.
- `/admin` 아래에서만 실행한다.
- 정적 파일 요청에는 실행하지 않는다.

이런 처리는 페이지가 렌더링되기 전에 끝내는 편이 낫다. 하지만 여기서 DB를 조회하고 권한 정책 전체를 계산하면 앞단 훅이 병목이 된다.

---

## 최소 예제

프로젝트 루트 또는 `src` 아래에 `proxy.ts`를 둔다.

```ts
// proxy.ts
import { NextResponse, type NextRequest } from "next/server";

export function proxy(request: NextRequest) {
  const session = request.cookies.get("session")?.value;

  if (!session) {
    const loginUrl = new URL("/login", request.url);
    loginUrl.searchParams.set("next", request.nextUrl.pathname);
    return NextResponse.redirect(loginUrl);
  }

  return NextResponse.next();
}

export const config = {
  matcher: ["/admin/:path*"],
};
```

`matcher`는 비용 제어 장치다. 모든 요청에서 실행하지 말고 필요한 경로만 좁힌다.

## open redirect 방어

로그인 후 돌아갈 URL을 받을 때 외부 URL을 그대로 허용하면 open redirect가 된다.

```ts
function safeNextPath(value: string | null) {
  if (!value || !value.startsWith("/") || value.startsWith("//")) {
    return "/";
  }

  return value;
}
```

```ts
export function proxy(request: NextRequest) {
  const session = request.cookies.get("session")?.value;

  if (session) {
    return NextResponse.next();
  }

  const loginUrl = new URL("/login", request.url);
  loginUrl.searchParams.set("next", safeNextPath(request.nextUrl.pathname));

  return NextResponse.redirect(loginUrl);
}
```

Proxy는 보안 경계에 가까우므로 문자열 하나도 신중하게 다뤄야 한다.

## locale rewrite

사용자 언어에 따라 내부 경로를 rewrite할 수 있다.

```ts
import { NextResponse, type NextRequest } from "next/server";

const PUBLIC_FILE = /\.(.*)$/;

export function proxy(request: NextRequest) {
  const { pathname } = request.nextUrl;

  if (PUBLIC_FILE.test(pathname) || pathname.startsWith("/api")) {
    return NextResponse.next();
  }

  if (pathname.startsWith("/ko") || pathname.startsWith("/en")) {
    return NextResponse.next();
  }

  const locale = request.headers.get("accept-language")?.startsWith("ko")
    ? "ko"
    : "en";
  const url = request.nextUrl.clone();
  url.pathname = `/${locale}${pathname}`;

  return NextResponse.rewrite(url);
}
```

rewrite는 브라우저 주소를 바꾸지 않고 내부적으로 다른 route를 보여준다. redirect와 다르다.

## A/B 테스트

Proxy는 쿠키를 보고 실험군을 나누는 데 쓸 수 있다.

```ts
export function proxy(request: NextRequest) {
  const bucket = request.cookies.get("experiment-home")?.value;
  const selectedBucket = bucket ?? (Math.random() < 0.5 ? "a" : "b");

  const response =
    selectedBucket === "b"
      ? NextResponse.rewrite(new URL("/home-b", request.url))
      : NextResponse.next();

  if (!bucket) {
    response.cookies.set("experiment-home", selectedBucket, {
      path: "/",
      httpOnly: true,
      sameSite: "lax",
    });
  }

  return response;
}

export const config = {
  matcher: ["/"],
};
```

실험 배정은 가볍다. 결과 집계, 분석, 복잡한 개인화는 별도 계층에서 처리한다.

## Edge runtime 제약

Proxy는 요청 앞단에서 빠르게 실행되어야 한다. 일반 Node.js 서버 코드처럼 생각하면 안 된다.

피해야 할 것:

- 매 요청 DB 조회
- 큰 패키지 import
- 장시간 네트워크 요청
- 파일 시스템 접근
- 복잡한 비즈니스 규칙 계산

가능하더라도 비용과 지연 시간이 문제다. Proxy는 “결정에 필요한 최소 정보가 쿠키/헤더/URL에 이미 있는가”를 먼저 묻는다.

---

## DB 접근을 피해야 하는 이유

관리자 접근을 막기 위해 매 요청마다 DB에서 사용자 권한을 조회하면 모든 페이지 요청 앞에 DB 병목을 둔 셈이 된다. 대신 Proxy에서는 세션 토큰 존재 여부나 서명된 claim처럼 빠르게 확인할 수 있는 정보만 본다. 세부 권한 검사는 Server Component, Server Action, Route Handler에서 다시 수행한다.

Proxy는 첫 번째 문지기이지 유일한 보안 계층이 아니다.

---

## Static export 제약

GitHub Pages static export에는 요청을 가로챌 서버가 없다. 따라서 Proxy는 런타임에 동작하지 않는다. 정적 배포에서 인증이 필요한 경로를 보호하려면 클라이언트 인증과 외부 API 권한 검사를 함께 설계해야 한다. HTML 파일 자체를 비밀로 만들 수는 없다.

---

## Nuxt와 비교

Nuxt에는 route middleware와 server middleware가 있다. Next.js Proxy는 페이지 렌더링 전 요청 경계라는 점에서 server middleware에 가깝지만, Next.js에서는 `matcher`로 비용 범위를 명시하는 습관이 특히 중요하다.

| 기준                | Nuxt                | Next.js                 |
| ------------------- | ------------------- | ----------------------- |
| 라우트 진입 전 처리 | route middleware    | Proxy                   |
| 서버 앞단 처리      | Nitro middleware    | Proxy / 플랫폼 계층     |
| 이동                | `navigateTo`        | `NextResponse.redirect` |
| 내부 경로 변경      | route rules/rewrite | `NextResponse.rewrite`  |

---

## 체크리스트

- Next.js 16 기준으로 Proxy를 Middleware의 최신 흐름으로 설명할 수 있다.
- Proxy를 요청 앞단의 얇은 네트워크 훅으로 제한할 수 있다.
- `matcher`로 실행 범위를 좁힐 수 있다.
- auth redirect에서 open redirect를 방어할 수 있다.
- redirect와 rewrite의 차이를 설명할 수 있다.
- Proxy에서 무거운 DB/비즈니스 로직을 피해야 하는 이유를 설명할 수 있다.
- static export에서 Proxy가 동작하지 않는다는 점을 알고 있다.
