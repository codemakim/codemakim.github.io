# React Router 가이드 - 기본 라우팅

> React Router 최신 Declarative Mode 문서를 기준으로 정리한 학습 자료다.
> 예제는 기존 React SPA 프로젝트에서 많이 쓰는 `react-router-dom` import를 사용한다.

---

## 왜 필요한가

React 기초에서 배운 컴포넌트는 화면 조각을 만드는 방법이다. 실제 앱은 여기서 한 단계 더 나아가야 한다.

- `/`에서는 홈 화면을 보여준다.
- `/posts`에서는 글 목록을 보여준다.
- `/posts/42`에서는 42번 글 상세를 보여준다.
- 없는 주소로 들어오면 404 화면을 보여준다.

이 문제를 직접 풀려면 `window.location.pathname`을 읽고 조건문으로 컴포넌트를 바꿔야 한다. 처음에는 가능해 보이지만, 링크 클릭, 뒤로가기, 404, 중첩 레이아웃, 권한 처리까지 붙으면 금방 복잡해진다.

React Router는 이 복잡도를 “URL 패턴과 컴포넌트의 연결”로 정리한다.

```tsx
<Route path="/posts" element={<PostsPage />} />
```

이 한 줄은 “주소가 `/posts`이면 `PostsPage`를 렌더링한다”는 규칙이다.

---

## 원리: URL도 화면 상태다

라우팅을 어렵게 느끼는 가장 큰 이유는 URL을 단순한 문자열로 보기 때문이다. SPA에서 URL은 화면 상태다.

예를 들어 `/posts?page=2`는 다음 정보를 담는다.

- 현재 화면은 글 목록이다.
- 목록의 2페이지를 보고 있다.
- 새로고침해도 같은 화면으로 돌아와야 한다.
- 링크를 공유하면 상대도 같은 화면을 봐야 한다.

React Router는 브라우저의 History API를 사용해 페이지 전체를 새로고침하지 않고 URL만 바꾼다. URL이 바뀌면 라우터가 현재 URL과 라우트 목록을 비교하고, 맞는 컴포넌트를 렌더링한다.

```
사용자가 /about 링크 클릭
  ↓
React Router가 브라우저 기본 이동을 가로챔
  ↓
History API로 주소만 /about으로 변경
  ↓
라우트 목록에서 /about에 맞는 컴포넌트 탐색
  ↓
AboutPage 렌더링
```

그래서 React Router를 배운다는 것은 단순히 API 이름을 외우는 일이 아니다. “어떤 URL이 어떤 화면 상태를 의미하는가”를 설계하는 일이다.

## 최소 예제

가장 작은 라우터 앱은 네 가지로 구성된다.

- `BrowserRouter`: 라우터가 브라우저 URL을 읽고 쓸 수 있게 하는 바깥 껍데기
- `Routes`: 여러 `Route` 중 현재 URL에 맞는 것을 고르는 영역
- `Route`: URL 패턴과 컴포넌트의 연결
- `Link`: 새로고침 없이 다른 URL로 이동하는 링크

### 설치

```bash
npm install react-router-dom
```

### 앱을 `BrowserRouter`로 감싸기

```tsx
// main.tsx
import React from "react";
import ReactDOM from "react-dom/client";
import { BrowserRouter } from "react-router-dom";
import App from "./App";

ReactDOM.createRoot(document.getElementById("root")!).render(
  <React.StrictMode>
    <BrowserRouter>
      <App />
    </BrowserRouter>
  </React.StrictMode>,
);
```

`BrowserRouter` 밖에서는 라우터 훅과 링크가 동작하지 않는다. 라우터가 현재 주소를 알고 있어야 하기 때문이다.

### 주소와 컴포넌트 연결하기

```tsx
// App.tsx
import { Link, Route, Routes } from "react-router-dom";

function HomePage() {
  return <h1>Home</h1>;
}

function AboutPage() {
  return <h1>About</h1>;
}

function NotFoundPage() {
  return <h1>404</h1>;
}

export default function App() {
  return (
    <div>
      <nav>
        <Link to="/">Home</Link>
        {" | "}
        <Link to="/about">About</Link>
      </nav>

      <Routes>
        <Route path="/" element={<HomePage />} />
        <Route path="/about" element={<AboutPage />} />
        <Route path="*" element={<NotFoundPage />} />
      </Routes>
    </div>
  );
}
```

이 예제에서 `/`는 홈, `/about`은 소개 화면이다. 어떤 라우트에도 맞지 않는 주소는 `path="*"`에 걸려 404 화면을 보여준다.

## `Link`와 `NavLink`

React 앱 안에서 이동할 때는 보통 `<a>` 대신 `<Link>`를 쓴다.

```tsx
<Link to="/about">About</Link>
```

`<a href="/about">`는 브라우저가 서버에 새 문서를 요청하게 만든다. SPA에서는 이 과정에서 앱이 다시 로드될 수 있다. 반면 `Link`는 React Router를 통해 주소만 바꾸고 필요한 컴포넌트만 교체한다.

메뉴에서 현재 위치를 강조해야 한다면 `NavLink`를 쓴다.

```tsx
import { NavLink } from "react-router-dom";

export default function Navigation() {
  return (
    <nav>
      <NavLink
        to="/"
        end
        className={({ isActive }) => (isActive ? "active" : undefined)}
      >
        Home
      </NavLink>
      {" | "}
      <NavLink
        to="/about"
        className={({ isActive }) => (isActive ? "active" : undefined)}
      >
        About
      </NavLink>
    </nav>
  );
}
```

`end`는 “정확히 이 주소에서만 활성화한다”는 뜻이다. 루트 링크(`/`)에 `end`를 붙이지 않으면 `/about`에서도 Home이 활성화되는 것처럼 보일 수 있다.

## 실무 예제: 작은 블로그 라우팅

기초를 익혔다면 다음 수준은 “목록, 상세, 404”를 가진 작은 블로그다.

```tsx
// App.tsx
import { Link, Route, Routes } from "react-router-dom";

const posts = [
  { id: "react-state", title: "React state 이해하기" },
  { id: "react-router", title: "React Router 시작하기" },
];

function Layout({ children }: { children: React.ReactNode }) {
  return (
    <div>
      <header>
        <Link to="/">Home</Link>
        {" | "}
        <Link to="/posts">Posts</Link>
      </header>
      <main>{children}</main>
    </div>
  );
}

function HomePage() {
  return <h1>홈</h1>;
}

function PostsPage() {
  return (
    <section>
      <h1>글 목록</h1>
      <ul>
        {posts.map((post) => (
          <li key={post.id}>
            <Link to={`/posts/${post.id}`}>{post.title}</Link>
          </li>
        ))}
      </ul>
    </section>
  );
}

function PostDetailPage() {
  return <h1>상세 화면은 다음 글에서 다룬다</h1>;
}

function NotFoundPage() {
  return (
    <section>
      <h1>404</h1>
      <p>존재하지 않는 페이지다.</p>
      <Link to="/">홈으로 이동</Link>
    </section>
  );
}

export default function App() {
  return (
    <Layout>
      <Routes>
        <Route path="/" element={<HomePage />} />
        <Route path="/posts" element={<PostsPage />} />
        <Route path="/posts/:postId" element={<PostDetailPage />} />
        <Route path="*" element={<NotFoundPage />} />
      </Routes>
    </Layout>
  );
}
```

여기서 `/posts/:postId`는 동적 라우트다. `react-router`나 `react-state`처럼 바뀌는 값을 URL 일부로 받는다. 자세한 파라미터 처리는 3편에서 다룬다.

## 함정과 디버깅

### 1. 라우터 밖에서 `Link`를 쓴다

`Link`, `Routes`, `useNavigate`, `useParams` 같은 API는 라우터 컨텍스트 안에서만 동작한다. 오류가 난다면 `BrowserRouter`가 앱 최상단에 있는지 먼저 확인한다.

### 2. 정적 배포에서 새로고침 404가 난다

SPA 라우팅은 브라우저 안에서는 잘 동작한다. 하지만 사용자가 `/posts`에서 새로고침하면 서버는 실제 `/posts` 파일을 찾으려고 한다. 정적 호스팅에서는 이 경로를 앱의 진입 파일로 돌려주는 설정이 필요하다.

GitHub Pages 같은 정적 배포에서는 404 fallback이나 hash routing 같은 전략을 별도로 검토해야 한다.

### 3. 모든 상태를 URL에 넣으려 한다

URL에는 공유와 복원이 필요한 상태를 넣는다. 모달 열림 여부, 임시 입력값, 드래그 중인 좌표처럼 순간적인 UI 상태는 컴포넌트 state가 더 자연스럽다.

## Vue Router와 비교

Vue Router를 알고 있다면 대응 관계는 단순하다.

| 관심사      | Vue Router                     | React Router           |
| ----------- | ------------------------------ | ---------------------- |
| 라우터 등록 | `createRouter()`               | `<BrowserRouter>`      |
| 라우트 영역 | `<router-view />`              | `<Routes>`와 `<Route>` |
| 링크        | `<router-link>`                | `<Link>`               |
| 활성 링크   | `<router-link>`의 active class | `<NavLink>`            |
| 404         | catch-all route                | `path="*"`             |

React Router는 라우트 정의도 JSX 안에 둘 수 있다는 점이 다르다. 그래서 “URL과 컴포넌트의 관계”가 React 컴포넌트 트리 안에서 더 직접적으로 보인다.

---

## 체크리스트

- `BrowserRouter`가 왜 앱 최상단에 필요한지 설명할 수 있다.
- `Routes`와 `Route`가 URL과 컴포넌트를 연결한다는 것을 이해한다.
- `<a>`와 `<Link>`의 차이를 설명할 수 있다.
- 현재 메뉴를 강조할 때 `NavLink`를 사용할 수 있다.
- `path="*"`로 기본 404 라우트를 만들 수 있다.
- 정적 배포에서 새로고침 404가 별도 배포 설정 문제라는 것을 안다.
