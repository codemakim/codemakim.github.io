# 어떤 상태 관리를 선택하는가

## 1. 오늘 풀 문제

새 기능을 시작할 때마다 "이거 useState? Zustand? Redux?"를 매번 고민한다. 표로 비교해봐도 실제 결정이 어렵다. "팀이 크면 Redux"라는 조언은 너무 추상적이다.

문제의 핵심은 도구를 먼저 고르는 데 있다. 옳은 순서는 **상태의 성격을 먼저 판별하고**, 그 다음 도구를 고르는 것이다. 이 글은 그 판별을 위한 질문 순서를 제시한다.

앞선 글들에서 각 도구의 API와 작동 원리를 다뤘다. 이 글은 도구 자체를 설명하지 않는다. "어느 상황에 무엇을 쓰는가"에 집중한다.

---

## 2. 정신 모델 — 결정 트리

상태가 하나 생긴다. 아래 질문을 순서대로 따라가면 도구가 좁혀진다.

```
상태가 필요하다
  │
  ├─ 서버/외부 API 데이터인가?
  │     YES → React Query (서버 상태는 라이브러리 영역이 아니다)
  │
  ├─ URL에 반영해야 하는가? (필터, 검색어, 페이지 번호)
  │     YES → useSearchParams (URL = 공유 가능한 상태)
  │
  ├─ 이 컴포넌트 트리 안에서만 쓰이는가?
  │     YES → useState / useReducer (로컬이면 전역으로 올리지 않는다)
  │
  ├─ 여러 페이지/트리에서 공유, 구조 단순
  │     → Zustand (Provider 없이, 최소 보일러플레이트)
  │
  ├─ 변경 추적 / 복잡한 비즈니스 규칙 / 팀 규모
  │     → Redux Toolkit (구조 강제, devtools, action 로그)
  │
  └─ 파생 값 그래프 / 비동기 의존 / 조합 복잡도
        → Jotai (atom 조합, derived atom)
```

이 트리의 핵심은 **React Query와 useSearchParams가 가장 먼저 온다**는 점이다. 많은 팀이 이 두 가지를 건너뛰고 전역 store에 서버 데이터와 URL 상태를 넣다가 불필요한 동기화 비용을 만든다.

### Context는 어디에 있는가

Context는 위 트리에 없다. Context의 적절한 용처는 따로 있다.

- 테마, 로케일, 인증 사용자처럼 **앱 전체에서 읽기만 하는 설정**
- **변경 빈도가 매우 낮은** 값 (변경될 때마다 Provider 하위 전체가 리렌더된다)
- 라이브러리 의존 없이 소규모 트리에서 값 내려주기

Context에 자주 변하는 값을 넣으면 리렌더링 범위를 제어할 수 없다. `useMemo`/`useCallback`으로 어느 정도 억제할 수 있지만, 그 비용이 Zustand 같은 라이브러리를 쓰는 것보다 크다. Context는 "전역 상태 라이브러리의 대안"이 아니라 "설정 전달 채널"이다.

## 3. 최소 코드 — 시나리오별 상태 배치

하나의 페이지 컴포넌트에서 4종류의 상태가 공존하는 예제를 본다.

```typescript
// react-router-dom 기준 예제다.
// Next.js에서는 next/navigation의 useSearchParams를 사용한다.
import { useSearchParams } from 'react-router-dom'
import { useQuery } from '@tanstack/react-query'
import { useState } from 'react'
import { useCartStore } from '@/store/cart'

async function fetchProducts(category: string) {
  const res = await fetch(`/api/products?category=${category}`)
  if (!res.ok) throw new Error('상품 목록 로드 실패')
  return res.json() as Promise<Product[]>
}

function ProductPage() {
  // 1. URL 상태: 브라우저 공유/북마크 가능
  const [searchParams, setSearchParams] = useSearchParams()
  const category = searchParams.get('category') ?? 'all'

  // 2. 서버 상태: React Query가 캐시/리페치/에러를 관리한다
  const { data: products, isPending, isError } = useQuery({
    queryKey: ['products', category],
    queryFn: () => fetchProducts(category),
  })

  // 3. 로컬 상태: 이 컴포넌트 안에서만 필요한 UI 상태
  const [expanded, setExpanded] = useState(false)

  // 4. 전역 클라이언트 상태: 여러 페이지에서 공유하는 장바구니
  const addToCart = useCartStore(s => s.addItem)

  function handleCategoryChange(next: string) {
    setSearchParams(prev => {
      prev.set('category', next)
      return prev
    })
  }

  if (isPending) return <p>로딩 중...</p>
  if (isError) return <p>상품을 불러오지 못했다.</p>

  return (
    <div>
      <select
        value={category}
        onChange={e => handleCategoryChange(e.target.value)}
      >
        <option value="all">전체</option>
        <option value="clothing">의류</option>
        <option value="electronics">전자기기</option>
      </select>

      <button onClick={() => setExpanded(v => !v)}>
        {expanded ? '접기' : '펼치기'}
      </button>

      {products.map(product => (
        <div key={product.id}>
          <span>{product.name}</span>
          <button onClick={() => addToCart(product)}>담기</button>
        </div>
      ))}
    </div>
  )
}
```

4가지 상태가 각자 다른 도구로 관리된다. 이것이 정상이다. 하나의 라이브러리로 모든 상태를 처리하려 하면 어딘가에서 불필요한 비용이 발생한다.

## 4. 실무 코드 — 도구별 적용 시나리오

### 시나리오 표

| 시나리오                       | 선택                 | 이유                                             |
| ------------------------------ | -------------------- | ------------------------------------------------ |
| 모달 열림/닫힘 (전역)          | Zustand              | Provider 없이 3줄, 어느 컴포넌트에서도 제어 가능 |
| 장바구니 (단순 목록)           | Zustand + persist    | 아이템 목록 + 로컬스토리지 저장                  |
| 장바구니 (쿠폰/정책)           | Redux Toolkit        | action 추적, selector memoize, 정책 규칙 테스트  |
| 체크아웃 계산 (소계/할인/최종) | Jotai                | 파생 atom 그래프, 각 계산 단계가 독립 atom       |
| 상품 목록/상세                 | React Query          | 서버 상태, 캐시/리페치/무효화                    |
| 현재 필터 (카테고리, 정렬)     | useSearchParams      | URL = 공유 가능, 새로고침 후에도 유지            |
| 폼 입력 중 값 (아직 제출 전)   | useState             | 컴포넌트 로컬, 제출 전에는 서버와 무관           |
| 인증 토큰 / 현재 사용자        | Zustand 또는 Context | 앱 전역 공유, 서버 상태 아님, 변경 빈도 낮음     |

### 언제 Zustand에서 Redux Toolkit으로 전환하는가

Zustand는 가볍고 빠르다. 하지만 아래 신호가 나타나면 RTK로 전환을 검토한다.

**신호 1 — action 히스토리가 필요하다**
"결제 실패가 어느 단계에서 발생했는가"를 추적해야 한다면 Redux DevTools의 action 로그가 필수다. Zustand에는 이 기능이 없다.

**신호 2 — 비즈니스 규칙이 여러 slice에 걸쳐 있다**
"쿠폰 적용 → 장바구니 총액 재계산 → 배송비 정책 변경"처럼 하나의 이벤트가 여러 상태를 연쇄적으로 바꿔야 할 때, Zustand의 단일 store 또는 여러 store 분리 방식으로는 인과 관계를 명시하기 어렵다. RTK의 `extraReducers`와 `createListenerMiddleware`가 이 연쇄를 명시적으로 표현한다.

**신호 3 — 온보딩 비용이 높아졌다**
새 팀원이 store를 보고 "이 상태는 어디서 어떻게 바뀌는가"를 추적하는 데 30분 이상 걸린다면 구조 강제가 필요한 시점이다. RTK의 slice/reducer/selector 패턴은 파일만 열면 흐름이 보인다.

### 전환 시 최소 기준

위 3가지 신호 중 2개 이상이 해당되면 전환을 진행한다. 1개만 해당되면 Zustand에서 미들웨어(`devtools`, `subscribeWithSelector`)로 해결 가능한지 먼저 확인한다.

## 5. 망하는 패턴

### 모든 상태를 Redux에 넣기

```typescript
// 나쁜 예: 단순 모달 open 상태를 RTK slice로 만들기
const modalSlice = createSlice({
  name: "modal",
  initialState: { isOpen: false },
  reducers: {
    open: (state) => {
      state.isOpen = true;
    },
    close: (state) => {
      state.isOpen = false;
    },
  },
});
```

모달 하나를 위해 slice, reducer, action, selector, dispatch가 필요하다. `useState(false)` 한 줄이면 끝나는 상태에 이 비용을 쓰면 실제로 복잡한 상태를 다룰 때 RTK가 무거운 도구로 느껴진다.

### 라이브러리 하나로 모든 상태 처리하기

```typescript
// 나쁜 예: 서버 상태를 Zustand에 직접 넣기
const useProductStore = create<ProductStore>((set) => ({
  products: [],
  isLoading: false,
  error: null,
  fetchProducts: async (category) => {
    set({ isLoading: true });
    try {
      const data = await fetchProducts(category);
      set({ products: data, isLoading: false });
    } catch (e) {
      set({ error: e, isLoading: false });
    }
  },
}));
```

이 패턴은 캐시 무효화, 백그라운드 리페치, 중복 요청 방지, 스테일 상태 관리를 직접 구현해야 한다. React Query가 이미 해결한 문제를 다시 만드는 것이다.

### 트렌드 기반 선택

"요즘 Jotai가 핫하다"는 선택 기준이 아니다. Jotai는 파생 상태 그래프가 복잡할 때 빛난다. 단순한 전역 UI 상태에 Jotai를 쓰면 atom 파일이 늘어나고 디버깅이 어려워진다.

### 팀 크기만으로 RTK 결정하기

"팀이 5명 이상이면 Redux"는 틀린 기준이다. 실제 기준은 **추적 필요 여부**다. 팀이 2명이어도 결제/환불/정책처럼 감사 로그가 필요한 도메인이면 RTK가 맞다. 팀이 10명이어도 단순한 SaaS 대시보드라면 Zustand로 충분하다.

## 6. Vue/Nuxt 비교

각 도구의 대응 관계는 다음과 같다.

| 역할                    | Vue/Nuxt                    | React                   |
| ----------------------- | --------------------------- | ----------------------- |
| 클라이언트 전역 상태    | Pinia                       | Zustand / Redux Toolkit |
| 복잡한 도메인 상태 추적 | Pinia + actions             | Redux Toolkit           |
| 서버 상태               | `useFetch` / `useAsyncData` | React Query             |
| URL 상태                | `useRoute().query`          | `useSearchParams`       |
| 파생 상태               | `computed`                  | Jotai derived atom      |
| 설정/테마 전달          | `provide` / `inject`        | Context                 |

분리 기준 자체는 동일하다. Vue는 Pinia가 공식 표준으로 자리잡혀 있어 전역 클라이언트 상태 선택지가 하나지만, React는 상황에 따라 여러 라이브러리 중에서 고른다. 이 차이가 "React 상태 관리는 복잡하다"는 인식을 만드는 주원인이다.

핵심은 **서버 상태 / URL 상태 / 로컬 상태 / 전역 클라이언트 상태를 구분하는 기준**이 프레임워크와 무관하게 동일하다는 점이다. Vue에서 React로, 또는 React에서 Vue로 전환할 때 도구 이름만 바꾸면 된다.

## 7. 마스터 체크

다음 5개 상태를 어떤 도구로 관리하겠는가, 그 이유를 설명할 수 있어야 한다.

**1. 상품 목록 (서버에서 가져옴)**

React Query. 서버 상태다. 캐시 TTL, 백그라운드 리페치, 에러 재시도를 React Query에 위임한다. 전역 store에 넣으면 이 기능을 직접 구현해야 한다.

**2. 장바구니 아이템**

Zustand + persist 미들웨어. 클라이언트 상태이고 여러 페이지에서 공유된다. 쿠폰/정책 복잡도가 낮은 단계에서는 Zustand로 시작한다. 비즈니스 규칙이 복잡해지면 RTK로 이전한다.

**3. 검색 필터 (카테고리, 정렬)**

useSearchParams. URL 상태다. 사용자가 링크를 공유하거나 새로고침해도 필터가 유지돼야 한다. 전역 store에 넣으면 URL과 store가 동기화 상태를 맞춰야 하는 문제가 생긴다.

**4. 쿠폰 입력 중 값 (아직 제출 전)**

useState. 폼 로컬 상태다. 사용자가 입력 중인 값은 제출 전까지 서버와 무관하고 다른 컴포넌트와 공유할 필요가 없다. 제출 시점에 서버로 보내거나 전역 상태를 업데이트하면 된다.

**5. 사이드바 열림 여부**

Zustand 또는 useState. 사이드바를 여러 컴포넌트에서 제어해야 한다면 Zustand. 해당 레이아웃 컴포넌트 안에서만 제어한다면 useState. 전역 store에 넣기 전에 "다른 컴포넌트가 이 상태를 읽거나 써야 하는가"를 먼저 확인한다.

---

### Zustand에서 RTK로 마이그레이션해야 하는 신호 3가지

1. **action 히스토리 추적이 필요하다** — "이 상태가 언제, 왜 바뀌었는가"를 디버깅해야 한다면 RTK DevTools가 필요하다.
2. **비즈니스 규칙이 여러 상태에 걸쳐 연쇄적으로 작동한다** — 단일 이벤트가 여러 store 또는 slice를 함께 업데이트해야 할 때 RTK의 `extraReducers`와 listener가 그 관계를 명시한다.
3. **신규 팀원이 store 흐름을 이해하는 데 30분 이상 걸린다** — Zustand store가 거대해지면 읽기 어려워진다. RTK의 강제 구조(slice/action/selector 분리)가 온보딩 비용을 낮춘다.
