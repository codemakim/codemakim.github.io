# 상태 관리 - 상태 관리 라이브러리 비교

라이브러리 비교의 핵심은 “문법”이 아니라 “비용 구조”다. 실무에서 상태 관리는 다음 비용을 만든다.

- 리렌더링 비용(성능)
- 디버깅 비용(원인 추적)
- 구조 유지 비용(리팩터링, 온보딩)
- 동기화 비용(서버 상태와의 경계)

이 글은 선택 기준을 “규칙으로 고정”하는 것을 목표로 한다.

---

## 1. 먼저 결정해야 하는 것: 서버 상태인가, 클라이언트 상태인가

상태 관리 라이브러리는 클라이언트 상태를 다루는 도구다.

- 서버 상태: React Query가 담당한다.
- 클라이언트 상태: UI 흐름, 입력 임시값, 전역 UI 상태, 도메인 상태 머신이 담당한다.

서버 상태를 전역 store에 넣으면 캐시/무효화/재시도/경쟁 상태 비용을 store 쪽에서 다시 구현하게 된다.

---

## 2. 전역 상태가 진짜 필요한 신호

전역 상태는 “여러 컴포넌트가 공유해야 하는 상태”다. 다음이면 전역 후보가 된다.

- 페이지를 넘어가도 유지돼야 한다(예: 장바구니, 인증 토큰, 온보딩 진행도).
- 서로 멀리 떨어진 컴포넌트가 같은 상태를 읽거나 써야 한다(예: 헤더의 카운트와 본문의 리스트).
- 단일 이벤트가 여러 UI를 동시에 바꿔야 한다(예: 로그아웃 시 토큰 삭제 + 캐시 제거 + 라우팅).

반대로 한 화면 안에서만 쓰고 끝나는 상태는 컴포넌트 로컬 state가 기본이다.

---

## 3. 선택지 비교(요약)

| 선택지 | 가장 강한 점 | 가장 약한 점 | 적합한 상황 |
| --- | --- | --- | --- |
| Context | 의존성 없음, 단순 | 리렌더링/구조가 금방 무너짐 | 테마/언어처럼 느리게 변하는 설정 |
| Zustand | 가벼움, selector 구독 | 팀 규칙이 없으면 난립 | 중소 규모, 전역 UI 상태 |
| Redux Toolkit | 구조/규칙/디버깅 | 초기 설계 비용 | 대규모, 복잡한 도메인 상태 |
| Jotai | 파생 상태/조합 | atom 난립 가능 | 설정/에디터/폼처럼 상태 그래프가 중요한 화면 |

이 표는 “정답”이 아니라 “첫 번째 후보를 고르는 기준”이다.

---

## 4. 실무 의사결정 프레임

아래 질문 순서대로 답하면 선택이 좁혀진다.

### 1) 서버 상태인가

서버가 진실이면 React Query로 간다. 상태 관리 라이브러리를 선택하는 단계가 아니다.

### 2) 전역 상태가 필요한가

로컬 state로 끝나면 전역을 만들지 않는다.

### 3) 구조와 디버깅이 중요한가

복잡한 규칙, 많은 팀원, 회귀가 잦으면 Redux Toolkit이 유리하다.

### 4) 작은 전역 UI 상태가 중심인가

모달/토스트/드로어/필터 UI 같은 전역 UI 상태가 중심이면 Zustand가 유리하다.

### 5) 파생 상태가 상태 모델의 중심인가

계산된 값과 의존 관계 조합이 핵심이면 Jotai가 유리하다.

---

## 5. 대표 시나리오로 선택하기

| 시나리오 | 추천 | 이유 |
| --- | --- | --- |
| 테마/언어 | Context 또는 Jotai(atomWithStorage) | 변경 빈도 낮음, 구조 단순 |
| 장바구니 | Zustand 또는 Redux Toolkit | 전역 공유 + 여러 화면에서 갱신 |
| 폼 마법사 | Jotai | 단계별 atom + 파생 규칙이 자연스럽다 |
| 복잡한 주문 상태(승인/취소/환불) | Redux Toolkit | 상태 머신/규칙/디버깅 비용이 크다 |
| 검색 결과/목록/상세 | React Query | 서버 상태 문제다 |

---

## 6. Vue와 비교

Vue의 기본 선택지는 Pinia다. React는 “표준 전역 상태”가 하나로 고정되어 있지 않아, 상황에 맞게 선택해야 한다.

| 관점 | Vue | React |
| --- | --- | --- |
| 기본 전역 상태 | Pinia | Zustand/Redux/Jotai 중 선택 |
| 서버 상태 | Vue Query 또는 직접 구현 | React Query |
| 규칙 강제 | 프레임보다 팀 규칙 | Redux Toolkit이 규칙을 제공한다 |

---

## 체크리스트

- 서버 상태와 클라이언트 상태를 구분할 수 있다.
- 전역 상태가 필요한 신호를 설명할 수 있다.
- “Zustand vs Redux Toolkit vs Jotai”를 문법이 아니라 비용 구조로 비교할 수 있다.
