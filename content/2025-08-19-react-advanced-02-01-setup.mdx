# React Query 가이드 - 설치 및 설정

React 기초 시리즈에서는 `useEffect`와 `useState`로 데이터를 가져오는 패턴까지 다뤘다. 작은 화면에서는 그 정도면 충분하다.

문제는 이 패턴이 화면이 늘어날 때마다 비슷한 코드를 반복하게 만든다는 점이다. 같은 API를 두 컴포넌트가 따로 부르고, 탭을 다녀와도 데이터가 그대로이고, 글을 새로 쓰면 목록은 갱신되지 않는다. 이런 문제를 한 번 직접 해결해보면 React Query가 무엇을 대신 해주는 라이브러리인지 이해하기 쉽다.

이 글은 시리즈의 출발점이다. React Query를 왜 쓰는지, 무엇을 맡기고 무엇은 맡기지 않는지, 그리고 최소 설정은 어떤 모양인지를 잡는다.

---

## 왜 필요한가

React 컴포넌트에서 데이터를 가져오는 가장 단순한 코드는 다음과 같다.

```tsx
import { useEffect, useState } from "react";

interface User {
  id: string;
  name: string;
}

export function UserList() {
  const [users, setUsers] = useState<User[] | null>(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState<Error | null>(null);

  useEffect(() => {
    setLoading(true);
    fetch("/api/users")
      .then((res) => {
        if (!res.ok) throw new Error("failed");
        return res.json() as Promise<User[]>;
      })
      .then((data) => {
        setUsers(data);
        setLoading(false);
      })
      .catch((err) => {
        setError(err);
        setLoading(false);
      });
  }, []);

  if (loading) return <div>loading...</div>;
  if (error) return <div>error</div>;
  return (
    <ul>
      {users?.map((u) => (
        <li key={u.id}>{u.name}</li>
      ))}
    </ul>
  );
}
```

이 코드는 동작한다. 그런데 다음 요구가 들어오면 빠르게 무거워진다.

- 같은 사용자 목록을 사이드바와 본문이 모두 표시한다. 두 컴포넌트가 각각 `fetch`를 호출하면 같은 요청이 두 번 나간다.
- 사용자가 탭을 바꿨다가 돌아왔을 때 최신 데이터를 보여줘야 한다. 직접 만들려면 `visibilitychange` 이벤트를 붙이고, 마지막 페치 시간을 비교해야 한다.
- 사용자를 새로 만들면 목록을 다시 가져와야 한다. `setUsers([...users, newUser])`로 직접 추가하면 서버가 보정한 값이 누락된다.
- 입력값이 바뀌는 검색에서는 이전 요청을 취소해야 늦게 도착한 응답이 화면을 덮지 않는다.

이 항목들이 하나둘 추가되면 페치 코드는 컴포넌트 안에 흩어지고, 컴포넌트마다 캐시 정책과 재시도 정책이 조금씩 달라진다. 결국 같은 문제를 매번 다른 방식으로 푸는 상태가 된다.

React Query는 이 문제들을 라이브러리 수준의 규칙으로 묶는다. 캐시는 어떻게 관리되는지, 언제 다시 가져오는지, 실패와 취소는 어떻게 다루는지를 한 곳에서 정의하고, 컴포넌트는 "어떤 데이터를 쓰는가"만 선언한다.

## 원리: 서버 상태와 클라이언트 상태는 다르다

React Query를 이해하는 가장 중요한 출발점은 상태를 두 종류로 나누는 것이다.

| 종류            | 예시                                | 진실의 근원       | 만료 개념          |
| --------------- | ----------------------------------- | ----------------- | ------------------ |
| 클라이언트 상태 | 모달 열림, 탭 선택, 입력 폼 임시값  | 컴포넌트/브라우저 | 컴포넌트 수명      |
| 서버 상태       | 사용자 목록, 게시글 상세, 검색 결과 | 서버              | 시간이 지나면 낡음 |

클라이언트 상태는 `useState`나 Zustand 같은 도구로 충분하다. 누가 진실을 가지고 있는지가 명확하기 때문이다.

서버 상태는 진실이 외부에 있다. 화면에 떠 있는 데이터는 서버 데이터의 사본일 뿐이고, 시간이 지나면 사본이 낡는다. 그래서 서버 상태는 다음 질문을 항상 함께 풀어야 한다.

- 캐시를 얼마나 신뢰할 수 있는가
- 언제 다시 가져와야 하는가
- 같은 데이터를 여러 곳에서 쓸 때 요청을 중복하지 않으려면 어떻게 해야 하는가
- 데이터를 바꿨을 때 다른 화면이 어떻게 알아차리는가

React Query는 이 질문들에 대한 표준 답을 제공한다. 컴포넌트는 "이 데이터를 쓰겠다"고 선언만 하고, 캐시 조회와 백그라운드 갱신은 라이브러리가 처리한다.

## 최소 예제: useQuery로 같은 코드를 다시 쓴다

위에서 `useEffect`로 만들었던 사용자 목록을 React Query로 바꾸면 다음과 같다.

```tsx
import { useQuery } from "@tanstack/react-query";

interface User {
  id: string;
  name: string;
}

async function fetchUsers(): Promise<User[]> {
  const res = await fetch("/api/users");
  if (!res.ok) throw new Error("failed");
  return (await res.json()) as User[];
}

export function UserList() {
  const query = useQuery({
    queryKey: ["users"],
    queryFn: fetchUsers,
  });

  if (query.isPending) return <div>loading...</div>;
  if (query.isError) return <div>error</div>;
  return (
    <ul>
      {query.data.map((u) => (
        <li key={u.id}>{u.name}</li>
      ))}
    </ul>
  );
}
```

겉보기에는 코드 줄 수가 비슷하지만 동작이 다르다.

- 사이드바와 본문이 같은 `["users"]` 키로 `useQuery`를 호출하면 요청은 한 번만 나간다.
- 탭을 다녀오면 기본적으로 백그라운드에서 다시 가져온다.
- 컴포넌트가 사라져도 캐시는 잠시 유지되어, 사용자가 뒤로가기를 하면 이전 데이터가 즉시 보인다.
- 같은 키로 여러 번 호출하더라도 동일한 `data` 참조를 공유한다.

이 동작들은 `useQuery` 인자에서 명시한 적이 없다. `QueryClient`의 기본값과 캐시 매커니즘이 자동으로 적용한 결과다.

## 핵심 어휘 세 가지를 먼저 잡는다

본격적인 사용 전에 이 시리즈에서 반복적으로 등장할 어휘 세 개를 짧게 정리한다. 자세한 동작은 이후 글에서 다시 다룬다.

### queryKey는 캐시의 주소다

`useQuery({ queryKey, queryFn })`에서 `queryKey`는 캐시 항목의 식별자다. 같은 키로 여러 컴포넌트가 호출하면 같은 캐시를 공유한다.

- `["users"]`: 전체 사용자 목록
- `["user", userId]`: 특정 사용자
- `["posts", { page, tag }]`: 페이지/태그 필터가 적용된 목록

키가 다르면 다른 데이터로 간주된다. 그래서 입력값이 바뀌면 키도 바뀌어야 한다. 키 설계는 2편과 4편에서 더 자세히 다룬다.

### staleTime은 신선도 시간이다

데이터는 가져온 직후 fresh, 일정 시간이 지나면 stale로 표시된다. fresh인 동안 React Query는 같은 키에 대한 추가 요청을 보내지 않는다. stale이 되면 컴포넌트가 다시 마운트되거나 창에 포커스가 돌아올 때 백그라운드 페치를 트리거한다.

### gcTime은 메모리 유지 시간이다

캐시를 보고 있는 컴포넌트(observer)가 모두 사라진 뒤, 캐시를 메모리에 얼마나 더 유지할지 정한다. v5에서 `cacheTime`이 `gcTime`으로 이름이 바뀌었다.

`staleTime`과 `gcTime`은 서로 다른 축이다. "신선한 데이터인가"와 "메모리에 남겨두는가"는 다른 질문이다. 이 차이는 4편 캐싱에서 자세히 정리한다.

## 최소 설정: QueryClient와 Provider

React Query는 전역에 `QueryClient` 한 개를 둔다. 이 객체가 캐시 저장소이자 기본 정책의 중심이다.

```tsx
import { QueryClient, QueryClientProvider } from "@tanstack/react-query";
import { ReactQueryDevtools } from "@tanstack/react-query-devtools";
import { App } from "./App";

const queryClient = new QueryClient();

export function Root() {
  return (
    <QueryClientProvider client={queryClient}>
      <App />
      <ReactQueryDevtools initialIsOpen={false} />
    </QueryClientProvider>
  );
}
```

여기서 주의할 점이 두 가지 있다.

1. `QueryClient`는 모듈 최상단에서 한 번만 만든다. 컴포넌트 안에서 `new QueryClient()`를 호출하면 매 렌더마다 새 인스턴스가 만들어져 캐시가 초기화된다.
2. SSR이나 테스트에서는 요청마다 새 인스턴스를 만드는 패턴이 필요할 수 있다. 그 경우 `useState(() => new QueryClient())`로 컴포넌트 인스턴스 단위로 만든다.

## 실무 권장 설정: 기본값을 명시한다

`new QueryClient()`만 써도 동작한다. 다만 실무에서는 기본 정책을 명시해 두는 편이 디버깅과 협업에 유리하다.

```tsx
import { QueryClient } from "@tanstack/react-query";

export const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      staleTime: 1000 * 30,
      gcTime: 1000 * 60 * 5,
      retry: 1,
      refetchOnWindowFocus: false,
    },
    mutations: {
      retry: 0,
    },
  },
});
```

이 값들은 정답이 아니다. 의도를 명시하는 것이 핵심이다.

- `staleTime: 30초` — 짧은 시간 내 같은 화면을 다시 열어도 동일한 데이터로 간주한다. 페칭 폭증을 막는다.
- `gcTime: 5분` — 화면을 잠깐 떠났다 돌아올 때 즉시 이전 데이터를 보여줄 수 있도록 메모리에 유지한다.
- `retry: 1` — 일시적인 네트워크 문제에 한 번만 재시도한다.
- `refetchOnWindowFocus: false` — 탭 복귀 자동 페치를 끈다. 자동 페치를 켤지 끌지는 도메인에 따라 다르다. 결제 화면이라면 켜는 편이 안전하고, 정적 콘텐츠라면 꺼두는 편이 비용이 낮다.
- `mutations.retry: 0` — 쓰기 요청은 멱등성이 보장되지 않는 한 재시도하지 않는다.

핵심은 "이 값을 왜 이렇게 정했는지"를 팀이 설명할 수 있다는 점이다.

## queryFn 규칙: fetch는 실패해도 throw하지 않는다

React Query는 `queryFn`이 throw하면 실패로 인식한다. 그런데 `fetch`는 4xx/5xx 응답에서도 throw하지 않는다. `res.ok`를 직접 확인해 throw해야 React Query가 에러로 처리한다.

```tsx
type ApiError = Error & { status?: number };

export async function fetchJson<T>(
  input: RequestInfo,
  init?: RequestInit,
): Promise<T> {
  const res = await fetch(input, init);
  if (!res.ok) {
    const err: ApiError = new Error(`request failed: ${res.status}`);
    err.status = res.status;
    throw err;
  }
  return (await res.json()) as T;
}
```

이 함수를 모든 `queryFn`의 기본으로 두면 다음 이점이 있다.

- 4xx/5xx가 React Query 상태로 흘러간다. `query.isError`와 `query.error.status`로 분기할 수 있다.
- 컴포넌트마다 `if (!res.ok) throw` 코드를 반복하지 않는다.
- 인증 만료(401) 같은 공통 처리를 한 곳에 둘 수 있다.

`axios`나 `ky` 같은 클라이언트는 기본적으로 4xx/5xx에서 throw하므로 별도 래퍼가 필요 없을 수 있다. 단, 응답 본문 파싱과 타입 처리는 여전히 한 곳으로 모으는 편이 안전하다.

## 함정과 디버깅

### 1. `QueryClient`를 컴포넌트 안에서 만든다

`Root` 컴포넌트가 다시 렌더링될 때마다 `QueryClient`가 새로 만들어진다. 캐시가 초기화되기 때문에 디버깅이 어려워진다. 모듈 최상단에서 한 번 만들거나, 인스턴스 단위로 유지해야 한다면 `useState(() => new QueryClient())`로 만든다.

### 2. `Provider`를 빠뜨린다

`useQuery`가 트리에서 가까운 `QueryClientProvider`를 찾지 못하면 런타임 에러가 난다. 앱의 가장 바깥에서 감싼다.

### 3. `fetch`의 4xx 응답을 성공으로 본다

`res.ok` 검증을 빠뜨리면 서버가 404나 500을 줘도 React Query는 성공으로 간주한다. `data`에 에러 본문이 들어가 다음 단계가 모두 깨진다.

### 4. devtools를 확인하지 않는다

`ReactQueryDevtools`는 캐시 상태와 쿼리 라이프사이클을 직접 보여준다. 개발 환경에서는 켜두고 `queryKey`, stale 상태, 리페치 타이밍을 확인하는 편이 디버깅에 도움이 된다.

## Vue와 비교

Vue 생태계에는 거의 동일한 모델의 `@tanstack/vue-query`가 있다. 어휘는 그대로 옮겨도 된다.

| 관심사               | Vue 생태계            | React 생태계             |
| -------------------- | --------------------- | ------------------------ |
| 서버 상태 관리       | `@tanstack/vue-query` | `@tanstack/react-query`  |
| 클라이언트 전역 상태 | Pinia                 | Zustand, Redux, Jotai 등 |
| Provider             | `VueQueryPlugin`      | `QueryClientProvider`    |
| 캐시 키              | `queryKey`            | `queryKey`               |
| 신선도/메모리        | `staleTime`, `gcTime` | `staleTime`, `gcTime`    |

```html
<script setup lang="ts">
  import { useQuery } from "@tanstack/vue-query";

  const query = useQuery({
    queryKey: ["users"],
    queryFn: async () => {
      const res = await fetch("/api/users");
      if (!res.ok) throw new Error("failed");
      return res.json();
    },
  });
</script>
```

Pinia를 쓰던 팀이 서버 상태까지 Pinia store에 넣고 있었다면, Vue Query로 옮기면서 "store는 클라이언트 상태 전용"이라는 경계를 다시 잡는 것이 좋다. React 쪽도 마찬가지다. Zustand나 Redux에 서버 데이터 캐시를 직접 만들고 있었다면 React Query로 그 책임을 옮긴다.

## 체크리스트

- 클라이언트 상태와 서버 상태를 구분해 설명할 수 있다.
- `useEffect + useState`로 페칭을 직접 만들 때 늘어나는 비용을 예시로 들 수 있다.
- `QueryClient`와 `QueryClientProvider`의 역할을 설명할 수 있다.
- `staleTime`, `gcTime`, `retry`, `refetchOnWindowFocus`의 의미를 한 문장씩 정리할 수 있다.
- `fetch`는 4xx/5xx에서 throw하지 않으므로 직접 throw 규칙이 필요하다는 점을 이해한다.
- `queryKey`가 캐시의 주소라는 비유를 다른 사람에게 설명할 수 있다.
