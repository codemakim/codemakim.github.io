# Plugins: 팀에 배포 가능한 AI 작업 표준을 만든다

Plugin은 하네스를 팀이나 여러 프로젝트에 배포하는 단계다.

처음부터 plugin을 만들 필요는 없다. 먼저 프로젝트 안에서 충분히 반복해 보고, 안정화된 것만 plugin으로 뽑는다.

```txt
프로젝트 로컬 규칙
  아직 실험 중인 하네스

Plugin
  여러 프로젝트와 팀원에게 배포할 만큼 안정화된 하네스
```

Plugin은 "멋진 패키징"이 아니라 표준화다.

---

## Plugin으로 묶을 수 있는 것

Plugin 후보는 다음이다.

- 공통 Skill
- 공통 subagent
- 공통 hook
- MCP 설정
- 팀 표준 workflow
- 템플릿과 스크립트
- 권한 정책 문서

Claude Code plugin은 skills, agents, hooks, MCP servers를 묶을 수 있다. Codex에서도 plugin 또는 shared skill bundle에 가까운 방식으로 반복 절차를 배포할 수 있다.

구조 예시는 다음이다.

```txt
ai-harness-plugin/
  .claude-plugin/
    plugin.json
  skills/
    bug-investigation/
      SKILL.md
    mdx-post-writing/
      SKILL.md
    release-check/
      SKILL.md
  agents/
    code-reviewer.md
    test-failure-analyst.md
  hooks/
    block-dangerous-bash.js
  docs/
    MCP_POLICY.md
    EXTERNAL_WRITE_POLICY.md
```

Plugin으로 뽑기 전에는 반드시 물어야 한다.

```txt
여러 프로젝트에서 반복되는가?
팀원이 같은 방식으로 써야 하는가?
권한 정책이 합의됐는가?
업데이트 책임자가 있는가?
버전이 바뀌었을 때 영향을 추적할 수 있는가?
```

아니면 프로젝트 로컬 설정으로 충분하다.

---

## Claude Code에서는

Claude Code plugin은 `.claude-plugin/plugin.json` 같은 metadata와 함께 skills, agents, hooks, MCP 설정을 묶는 방식으로 생각할 수 있다.

좋은 plugin은 기능을 많이 넣는 것이 아니라 경계를 명확히 한다.

```json
{
  "name": "team-ai-harness",
  "version": "0.1.0",
  "description": "Shared AI workflow harness for the frontend platform team"
}
```

처음 버전에는 최소 구성만 넣는다.

```txt
skills/
  bug-investigation/
  code-review/
  release-check/

agents/
  code-reviewer.md

hooks/
  block-dangerous-bash.js

docs/
  MCP_POLICY.md
```

팀 전체에 배포하기 전에는 한 프로젝트에서 충분히 써보고, 실제로 반복되는 것만 남긴다.

---

## Codex에서는

Codex에서 plugin을 생각할 때도 핵심은 같다. 공통 `AGENTS.md` 패턴, 공통 Skill, 공통 approval 정책, 공통 MCP 정책을 재사용 가능한 묶음으로 만든다.

Codex 팀 표준은 이렇게 시작할 수 있다.

```txt
team-harness/
  AGENTS.template.md
  skills/
    bug-investigation/
      SKILL.md
    pr-review/
      SKILL.md
  policies/
    MCP_POLICY.md
    APPROVAL_POLICY.md
  examples/
    nextjs-blog/
    api-service/
```

각 프로젝트는 이 템플릿을 그대로 복사하지 말고 자기 맥락에 맞게 줄인다.

```txt
공통 template
  조직 표준

프로젝트 AGENTS.md
  실제 프로젝트 규칙

로컬 사용자 설정
  개인 선호
```

팀 표준이 프로젝트 현실보다 우선하면 하네스가 딱딱해진다. 공통 plugin은 기본값이고, 프로젝트 Memory가 실제 작업 규칙을 완성한다.

## 버전 관리가 중요하다

Plugin은 코드처럼 버전 관리해야 한다.

나쁜 방식은 이것이다.

```txt
팀 문서 어딘가에 최신 Skill을 복사해 둔다.
누가 고쳤는지 모른다.
어떤 프로젝트가 어떤 버전을 쓰는지 모른다.
문제가 생기면 되돌릴 수 없다.
```

좋은 방식은 이것이다.

```txt
plugin version
  0.1.0

changelog
  bug-investigation Skill에 재현 단계 추가
  dangerous bash hook에 git clean 차단 추가

adoption record
  blog project: 0.1.0
  admin project: 0.1.0
  api project: 0.0.3
```

하네스가 팀 표준이 되면 변경 자체가 제품 변경처럼 취급되어야 한다.

---

## 성숙도 모델

마스터하려면 한 번에 모든 것을 만들기보다 성숙도를 올린다.

```txt
Level 1. Memory
  AGENTS.md 또는 CLAUDE.md에 기본 규칙이 있다.

Level 2. Skills
  반복 작업이 절차화되어 있다.

Level 3. Permissions
  위험 명령과 외부 write가 차단 또는 승인 대상이다.

Level 4. Subagents
  조사, 리뷰, 구현 책임이 분리된다.

Level 5. Orchestration
  여러 agent가 gate와 ownership 안에서 조율된다.

Level 6. MCP
  외부 도구가 read-only부터 연결된다.

Level 7. Plugins
  팀 공통 하네스가 재사용 가능하게 배포된다.

Level 8. Audit
  실행 로그, 검증 결과, 권한 변경 이력이 남는다.
```

대부분의 개인 프로젝트는 Level 3만 되어도 안정성이 크게 올라간다. 팀 프로젝트는 Level 6 이상이 필요하다. 조직 표준으로 만들려면 Level 7과 audit이 필요하다.

## 전체 체크리스트

시리즈 전체를 마무리하며 확인할 항목은 다음이다.

- `AGENTS.md` 또는 `CLAUDE.md`가 있는가?
- Memory는 짧고 행동 중심인가?
- 반복 작업은 Skill로 분리되어 있는가?
- destructive command는 차단 또는 승인 대상인가?
- 외부 write 작업은 read와 분리되어 있는가?
- subagent는 파일 소유권과 출력 형식이 명확한가?
- 여러 agent를 조율하는 orchestration gate가 있는가?
- GitHub, DB, Gmail 같은 MCP 연결은 read-only부터 시작하는가?
- 완료 조건은 검증 명령으로 표현되는가?
- 배포 후에는 Actions 성공과 실제 산출물 존재를 모두 확인하는가?
- 여러 프로젝트에서 반복되는 절차만 plugin으로 묶는가?

## 초급은 어디서 끝나는가

이 시리즈의 초급 목표는 "AI agent platform을 만든다"가 아니다. 개인 또는 작은 팀 프로젝트에서 AI 코딩 도구를 안전하게 운영할 수 있는 최소 하네스를 만드는 것이다.

초급은 다음을 할 수 있으면 끝난다.

```txt
1. AGENTS.md 또는 CLAUDE.md를 작성한다.
2. 반복 작업 하나를 Skill로 만든다.
3. 위험 명령과 외부 write를 approval 대상으로 둔다.
4. subagent의 역할과 파일 소유권을 나눈다.
5. 간단한 fan-out / fan-in orchestration을 설명한다.
6. MCP는 read-only부터 연결한다.
7. 여러 프로젝트에서 반복되는 절차만 plugin 후보로 분리한다.
8. 완료 전 검증 결과를 보고한다.
```

중급부터는 "내 프로젝트에서 쓸 수 있다"를 넘어 "팀 workflow로 안정적으로 운영한다"가 목표가 된다. 고급은 권한 모델, audit, 평가, 비용, agent platform 설계로 넘어간다.

## 마지막 정리

AI 하네스 엔지니어링은 AI를 믿지 말자는 이야기가 아니다. AI를 믿을 수 있는 작업 환경 안에 두자는 이야기다.

```txt
Memory
  항상 필요한 규칙

Skills
  반복 절차

Hooks / Permissions
  위험 행동 차단

Subagents
  컨텍스트 격리

Orchestration
  여러 작업자의 순서, gate, 실패 처리

MCP
  외부 도구 연결

Plugins
  팀 배포

Verification
  완료 증거
```

Claude Code든 Codex든 목표는 같다. AI에게 더 좋은 말투로 부탁하는 것이 아니라, AI가 작업하고 실패하고 검증받는 구조를 설계하는 것이다.

프롬프트를 잘 쓰는 사람은 좋은 답변을 한 번 얻는다. 하네스를 잘 설계하는 사람은 좋은 작업 방식을 반복해서 얻는다.
