# Subagents: 일을 나누기보다 컨텍스트를 격리한다

Subagent를 쓰면 작업을 병렬로 나눌 수 있다. 하지만 핵심은 병렬성이 아니다. 더 중요한 목적은 컨텍스트와 책임을 격리하는 것이다.

큰 요청 하나에 조사, 구현, 리뷰, 검증을 모두 넣으면 AI는 쉽게 흔들린다. 읽기만 해야 할 작업자가 파일을 수정하고, 구현자가 검증까지 낙관적으로 해석하며, 리뷰가 변경 의도를 변호하는 문제가 생긴다.

---

## 언제 필요한가

작은 수정은 main agent 하나로 충분하다. Subagent는 다음 조건에서 쓴다.

- 조사해야 할 영역이 서로 독립적이다.
- 구현 파일 소유권을 분리할 수 있다.
- 리뷰를 구현자와 분리하고 싶다.
- 긴 로그나 큰 diff를 별도 컨텍스트로 읽히고 싶다.
- 외부 도구 조사와 로컬 구현을 분리하고 싶다.

반대로 다음 작업에는 굳이 쓰지 않는다.

- 바로 다음 단계가 subagent 결과에 막혀 있다.
- 수정 범위가 한 파일이고 작다.
- 요구사항이 아직 불명확하다.
- 여러 작업자가 같은 파일을 동시에 고쳐야 한다.

Subagent는 공짜 병렬성이 아니다. 결과를 통합하는 비용이 생긴다.

---

## 역할 모델

가장 안전한 시작은 읽기 전용 역할이다.

| 역할     | 권한           | 출력                         |
| -------- | -------------- | ---------------------------- |
| Explorer | 읽기 전용      | 관련 파일, 데이터 흐름, 근거 |
| Reviewer | 읽기 전용      | 버그, 회귀, 누락 테스트      |
| Verifier | 명령 실행 중심 | 검증 명령과 결과             |
| Worker   | 제한된 쓰기    | 맡은 파일 변경               |

역할 경계는 명시해야 한다.

```txt
Explorer는 파일을 수정하지 않는다.
Reviewer는 수정 제안을 하되 직접 고치지 않는다.
Worker는 지정된 파일만 수정한다.
Verifier는 결과를 해석하지 않고 명령 출력과 실패 지점을 보고한다.
Main은 통합과 최종 판단을 맡는다.
```

이 규칙이 없으면 subagent는 편의 기능이 아니라 예측 불가능한 변경자가 된다.

---

## 좋은 지시 예시

읽기 전용 explorer에게는 범위를 좁혀 준다.

```txt
Explorer로 조사해줘.

목표:
- `content/*.mdx` 포스트가 빌드에 포함되는 흐름을 확인한다.

범위:
- 읽기만 한다.
- 파일을 수정하지 않는다.
- 관련 파일 경로와 근거 줄을 보고한다.

출력:
- 수집 경로
- 필수 frontmatter
- 빌드 명령
- 확인한 파일 목록
```

worker에게는 파일 소유권을 준다.

```txt
Worker는 `content/2026-05-09-ai-harness-03-skills.mdx`만 수정한다.
다른 작업자가 같은 저장소에서 작업 중일 수 있으므로 unrelated diff를 되돌리지 않는다.
수정 후 변경 파일 목록과 검증 필요 명령을 보고한다.
```

소유 파일 범위가 없으면 충돌 가능성이 커진다.

---

## Fan-out과 Fan-in

Subagent를 여러 개 쓸 때는 fan-out / fan-in 구조로 본다.

```txt
Main
  → Explorer A: 콘텐츠 규칙 조사
  → Explorer B: 참조 문서 구조 조사
  → Explorer C: 기존 시리즈 frontmatter 조사
  ← 결과 수집
Main
  → 설계 통합
  → Worker 작업 배정
  → Reviewer 검토
  → Verifier 검증
```

fan-out은 독립 조사에 적합하다. fan-in에서는 main이 결과를 그대로 붙이지 않는다. 중복, 충돌, 근거 수준을 판단해야 한다.

---

## 리뷰용 subagent부터 시작한다

처음 도입할 subagent는 reviewer가 좋다. 읽기 전용이고, 기존 작업을 직접 망칠 가능성이 낮다.

리뷰 지시는 이렇게 쓴다.

```txt
최근 변경사항을 리뷰해줘.

관점:
- 버그
- 회귀 위험
- 누락된 검증
- 요구사항 불일치

제외:
- 취향성 리팩터링
- 포맷 취향
- 근거 없는 추측

출력:
- 심각도 순 findings
- 파일/라인 근거
- 남은 검증 리스크
```

리뷰 결과도 맹신하지 않는다. main이 근거를 확인하고 반영 여부를 결정한다.

---

## 함정

Subagent의 흔한 실패는 세 가지다.

첫째, 같은 일을 여러 agent에게 중복 배정한다. 결과가 길어지고 충돌만 늘어난다.

둘째, worker에게 파일 범위를 주지 않는다. 이 경우 unrelated refactor가 섞이기 쉽다.

셋째, reviewer에게 수정까지 시킨다. 리뷰와 구현이 섞이면 독립성이 사라진다.

Subagent는 책임을 흐리는 장치가 아니라 책임을 좁히는 장치로 써야 한다.
