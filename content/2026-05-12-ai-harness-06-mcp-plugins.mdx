# MCP와 Plugins: 외부 도구를 연결하되 최소 권한으로 시작한다

AI 코딩 도구는 로컬 파일만 다룰 때보다 외부 시스템과 연결될 때 훨씬 강력해진다.

- GitHub 이슈를 읽는다.
- PR 리뷰 코멘트를 반영한다.
- Actions 로그를 분석한다.
- 브라우저 screenshot을 확인한다.
- DB schema를 읽는다.
- Gmail thread를 요약한다.
- 배포 상태를 확인한다.

하지만 강력해지는 만큼 위험도 커진다. 외부 도구 연결은 생산성 기능이 아니라 권한 설계 문제다.

---

## MCP가 하는 일

MCP는 Model Context Protocol이다. AI 애플리케이션과 외부 도구를 연결하는 표준 인터페이스다.

MCP는 크게 다음 개념으로 이해하면 된다.

```txt
Tools
  AI가 호출할 수 있는 함수
  예: GitHub PR 읽기, 브라우저 screenshot, DB schema 조회

Resources
  AI가 읽을 수 있는 데이터
  예: 파일, 문서, schema, issue 내용

Prompts
  재사용 가능한 프롬프트 템플릿

Authorization
  어떤 도구에 어떤 권한으로 접근할지 정하는 계층
```

MCP의 핵심은 "AI에게 외부 세계를 보여준다"가 아니다. 어떤 외부 기능을 어떤 권한으로 보여줄지 정하는 것이다.

---

## read-only부터 시작한다

외부 도구 연결 순서는 보수적이어야 한다.

```txt
1. local docs read
2. GitHub issue/PR read
3. GitHub Actions log read
4. browser screenshot
5. database schema read
6. limited write with approval
7. broader write after audit
```

처음부터 write를 열면 위험하다.

```txt
GitHub write
  잘못된 댓글, 잘못된 label, 원치 않는 rerun

DB write
  데이터 손상, 개인정보 문제

메일 발송
  회수 어려움

cloud write
  비용, 장애, 삭제 위험
```

read-only는 학습과 조사에 충분하다. write는 workflow와 approval이 정해진 뒤에 열어야 한다.

---

## Claude Code에서는

Claude Code에서는 MCP 서버를 등록하고 상태를 확인할 수 있다.

```bash
claude mcp list
claude mcp get github
claude mcp remove github
```

MCP를 붙일 때는 정책 문서를 같이 둔다.

```md
# MCP Policy

## Default

All MCP integrations start as read-only.

## Allowed

- GitHub issue read
- GitHub PR read
- GitHub Actions read
- browser screenshot
- local docs read
- database schema read

## Requires Approval

- GitHub comment write
- PR branch push
- workflow rerun
- email send
- database write
- production resource access

## Forbidden

- printing secrets
- writing to production DB
- deleting remote resources
- running unreviewed shell scripts from MCP output
```

이 문서는 AI뿐 아니라 팀을 위한 문서다. 누가 어떤 권한을 열었는지, 어떤 작업이 승인 대상인지 남겨야 한다.

---

## Codex에서는

Codex에서도 MCP tools와 app connectors를 통해 외부 시스템을 다룰 수 있다. GitHub, Gmail 같은 connector가 대표적이다.

Codex에서 외부 도구를 쓸 때는 작업을 세 단계로 분리한다.

```txt
Read
  외부 정보를 읽는다.

Plan
  읽은 정보를 바탕으로 변경 계획을 만든다.

Write
  외부 상태를 바꾼다.
```

Read와 Plan은 비교적 안전하다. Write는 별도 승인 대상이다.

좋은 요청은 다음처럼 쓴다.

```txt
GitHub Actions 실패 원인을 확인해줘.
로그 읽기는 허용한다.
rerun, comment, push 같은 write 작업은 하지 마.
원인과 로컬에서 재현할 명령만 정리해줘.
```

쓰기 작업이 필요하면 분리한다.

```txt
PR 코멘트 답변 초안을 작성해줘.
바로 게시하지 말고 내가 승인하면 게시해.
```

이렇게 하면 Codex가 외부 시스템을 다룰 때도 감사 가능한 흐름이 된다.

## 예시: GitHub Actions 확인

배포 후 글이 보이지 않는 문제를 조사한다고 하자.

나쁜 흐름은 이렇다.

```txt
1. 배포가 실패했다고 가정한다.
2. 코드를 마구 수정한다.
3. 빌드를 돌리지 않는다.
```

좋은 흐름은 이렇게 나뉜다.

```txt
1. GitHub Actions run 상태를 읽는다.
2. 성공이면 배포 문제가 아니라고 판단한다.
3. Contentlayer 수집 대상과 라우트를 확인한다.
4. 정적 산출물 `out/posts/<slug>/index.html` 존재 여부를 확인한다.
5. 필요한 경우 게시용 MDX를 추가한다.
```

GitHub read만으로도 원인 분리가 가능하다. write 권한은 필요 없다.

---

## 예시: DB schema 조회

습관 관리 기능을 수정한다고 하자. AI가 직접 DB write를 하게 만들면 위험하다. 하지만 schema read는 도움이 된다.

```txt
Allowed:
  habit table schema 읽기
  RLS policy 읽기
  migration 파일 읽기

Requires approval:
  test row insert
  migration apply
  production data update

Forbidden:
  production record delete
  secret 출력
```

Codex에게는 이렇게 지시한다.

```txt
Supabase schema를 읽고 필요한 컬럼과 RLS 제약만 요약해줘.
DB write는 하지 마.
코드 수정이 필요하면 로컬 파일 변경 계획만 세워줘.
```

외부 도구는 정보 수집과 상태 변경을 분리해야 한다.

## Plugins는 언제 필요한가

Plugin은 하네스를 팀이나 여러 프로젝트에 배포하는 단계다.

처음부터 plugin을 만들 필요는 없다. 먼저 프로젝트 안에서 충분히 반복해 보고, 안정화된 것만 plugin으로 뽑는다.

Plugin 후보는 다음이다.

- 공통 Skill
- 공통 subagent
- 공통 hook
- MCP 설정
- 팀 표준 workflow
- 템플릿과 스크립트

Claude Code plugin은 skills, agents, hooks, MCP servers를 묶을 수 있다. Codex에서도 plugin 또는 shared skill bundle에 가까운 방식으로 반복 절차를 배포할 수 있다.

구조 예시는 다음이다.

```txt
ai-harness-plugin/
  .claude-plugin/
    plugin.json
  skills/
    bug-investigation/
      SKILL.md
    mdx-post-writing/
      SKILL.md
  agents/
    code-reviewer.md
  hooks/
    block-dangerous-bash.js
  docs/
    MCP_POLICY.md
```

Plugin으로 뽑기 전에는 반드시 물어야 한다.

```txt
여러 프로젝트에서 반복되는가?
팀원이 같은 방식으로 써야 하는가?
권한 정책이 합의됐는가?
업데이트 책임자가 있는가?
```

아니면 프로젝트 로컬 설정으로 충분하다.

## 하네스 성숙도 모델

마스터하려면 한 번에 모든 것을 만들기보다 성숙도를 올린다.

```txt
Level 1. Memory
  AGENTS.md 또는 CLAUDE.md에 기본 규칙이 있다.

Level 2. Skills
  반복 작업이 절차화되어 있다.

Level 3. Permissions
  위험 명령과 외부 write가 차단 또는 승인 대상이다.

Level 4. Subagents
  조사, 리뷰, 구현 책임이 분리된다.

Level 5. MCP
  외부 도구가 read-only부터 연결된다.

Level 6. Plugins
  팀 공통 하네스가 재사용 가능하게 배포된다.

Level 7. Audit
  실행 로그, 검증 결과, 권한 변경 이력이 남는다.
```

대부분의 개인 프로젝트는 Level 3만 되어도 안정성이 크게 올라간다. 팀 프로젝트는 Level 5 이상이 필요하다.

## 전체 체크리스트

시리즈 전체를 마무리하며 확인할 항목은 다음이다.

- `AGENTS.md` 또는 `CLAUDE.md`가 있는가?
- Memory는 짧고 행동 중심인가?
- 반복 작업은 Skill로 분리되어 있는가?
- destructive command는 차단 또는 승인 대상인가?
- 외부 write 작업은 read와 분리되어 있는가?
- subagent는 파일 소유권과 출력 형식이 명확한가?
- GitHub, DB, Gmail 같은 MCP 연결은 read-only부터 시작하는가?
- 완료 조건은 검증 명령으로 표현되는가?
- 배포 후에는 Actions 성공과 실제 산출물 존재를 모두 확인하는가?
- 여러 프로젝트에서 반복되는 절차만 plugin으로 묶는가?

## 마지막 정리

AI 하네스 엔지니어링은 AI를 믿지 말자는 이야기가 아니다. AI를 믿을 수 있는 작업 환경 안에 두자는 이야기다.

```txt
Memory
  항상 필요한 규칙

Skills
  반복 절차

Hooks / Permissions
  위험 행동 차단

Subagents
  컨텍스트 격리

MCP
  외부 도구 연결

Plugins
  팀 배포

Verification
  완료 증거
```

Claude Code든 Codex든 목표는 같다. AI에게 더 좋은 말투로 부탁하는 것이 아니라, AI가 작업하고 실패하고 검증받는 구조를 설계하는 것이다.

프롬프트를 잘 쓰는 사람은 좋은 답변을 한 번 얻는다. 하네스를 잘 설계하는 사람은 좋은 작업 방식을 반복해서 얻는다.
