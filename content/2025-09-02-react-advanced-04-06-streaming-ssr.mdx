# Streaming SSR와 Suspense

서버 렌더링은 HTML을 빨리 받을 수 있다는 장점이 있지만, 느린 데이터 하나가 전체 응답을 붙잡으면 장점이 사라진다. 게시글 본문은 빨리 준비되는데 추천 글 위젯이 느려서 전체 화면이 늦게 뜨는 상황이 대표적이다.

Streaming SSR은 화면을 한 덩어리로 기다리지 않게 만든다. 준비된 부분을 먼저 보내고, 느린 부분은 Suspense 경계 안에서 나중에 채운다.

---

## 왜 필요한가

대시보드에는 빠른 데이터와 느린 데이터가 섞인다.

- 사용자 이름: 빠름
- 오늘 매출: 보통
- 외부 분석 리포트: 느림
- 알림 목록: 실패 가능

모든 데이터를 `page.tsx` 최상단에서 `await`하면 가장 느린 데이터가 전체 화면의 첫 응답 시간을 결정한다. 사용자는 아무것도 보지 못한 채 기다린다.

---

## `loading.tsx`는 segment의 기본 대기 화면이다

route segment에 `loading.tsx`를 두면 해당 segment가 준비되는 동안 보여줄 UI를 정의할 수 있다.

```tsx
// app/dashboard/loading.tsx
export default function DashboardLoading() {
  return (
    <main>
      <h1>대시보드</h1>
      <section aria-label="loading summary">요약 정보를 불러오는 중</section>
    </main>
  );
}
```

`loading.tsx`는 페이지 전체가 늦을 때 유용하다. 하지만 화면 내부의 일부 위젯만 느린 경우에는 Suspense 경계를 직접 나누는 편이 낫다.

---

## nested Suspense

```tsx
// app/dashboard/page.tsx
import { Suspense } from "react";
import { AnalyticsPanel } from "./AnalyticsPanel";
import { AnalyticsSkeleton } from "./AnalyticsSkeleton";
import { RevenueSummary } from "./RevenueSummary";

export default function DashboardPage() {
  return (
    <main>
      <h1>대시보드</h1>
      <RevenueSummary />
      <Suspense fallback={<AnalyticsSkeleton />}>
        <AnalyticsPanel />
      </Suspense>
    </main>
  );
}
```

```tsx
// app/dashboard/AnalyticsPanel.tsx
async function getAnalytics() {
  const response = await fetch("https://example.com/api/analytics", {
    next: { revalidate: 300 },
  });

  if (!response.ok) {
    throw new Error("분석 데이터를 불러오지 못했다.");
  }

  return response.json() as Promise<{ visitors: number }>;
}

export async function AnalyticsPanel() {
  const analytics = await getAnalytics();

  return <section>방문자 {analytics.visitors}명</section>;
}
```

`AnalyticsPanel`이 느려도 `RevenueSummary`와 제목은 먼저 렌더링될 수 있다. Suspense는 데이터 fetching API가 아니라 렌더링 대기 경계다.

---

## 느린 데이터는 컴포넌트 아래로 내린다

나쁜 패턴은 페이지 최상단에서 모든 데이터를 기다리는 것이다.

```tsx
export default async function DashboardPage() {
  const revenue = await getRevenue();
  const analytics = await getAnalytics();

  return (
    <main>
      <RevenueSummary revenue={revenue} />
      <AnalyticsPanel analytics={analytics} />
    </main>
  );
}
```

느린 데이터는 해당 UI 컴포넌트 안으로 내린다.

```tsx
export default function DashboardPage() {
  return (
    <main>
      <Suspense fallback={<RevenueSkeleton />}>
        <RevenueSummary />
      </Suspense>
      <Suspense fallback={<AnalyticsSkeleton />}>
        <AnalyticsPanel />
      </Suspense>
    </main>
  );
}
```

이 구조에서는 각 영역이 자기 데이터 준비 상태를 가진다.

---

## skeleton은 레이아웃을 흔들지 않아야 한다

나쁜 skeleton은 실제 콘텐츠보다 훨씬 작거나 커서 CLS를 만든다.

```tsx
// app/dashboard/AnalyticsSkeleton.tsx
export function AnalyticsSkeleton() {
  return (
    <section aria-busy="true" aria-label="분석 데이터 로딩">
      <h2>분석</h2>
      <div style={{ minHeight: 160 }}>불러오는 중</div>
    </section>
  );
}
```

실무에서는 실제 패널과 비슷한 높이, 여백, 제목 구조를 유지해야 한다. Streaming을 넣었는데 레이아웃이 출렁이면 Core Web Vitals 관점에서는 실패다.

---

## 경계를 너무 많이 쪼갤 때의 비용

Suspense 경계가 많을수록 좋은 것은 아니다.

- fallback이 너무 많이 보이면 화면이 조각나 보인다.
- 데이터 요청이 의도치 않게 잘게 쪼개져 워터폴이 생길 수 있다.
- 각 영역의 오류/로딩 설계가 복잡해진다.

기준은 사용자 경험이다. 독립적으로 먼저 보여줄 가치가 있는 영역만 경계로 나눈다.

---

## 실무 예제: 게시글 + 느린 추천 글

```tsx
// app/posts/[slug]/page.tsx
import { Suspense } from "react";
import { RelatedPosts } from "./RelatedPosts";
import { RelatedPostsSkeleton } from "./RelatedPostsSkeleton";

type PageProps = {
  params: Promise<{ slug: string }>;
};

export default async function PostPage({ params }: PageProps) {
  const { slug } = await params;
  const post = await getPost(slug);

  return (
    <article>
      <h1>{post.title}</h1>
      <div>{post.body}</div>

      <Suspense fallback={<RelatedPostsSkeleton />}>
        <RelatedPosts slug={slug} />
      </Suspense>
    </article>
  );
}
```

본문은 핵심 콘텐츠라 먼저 보여준다. 추천 글은 있으면 좋지만 본문을 막을 이유는 없다.

## Static export에서의 의미

static export에서도 빌드 시점에 Server Component 결과를 만들 수 있다. 하지만 요청 시점 streaming 서버가 있는 것은 아니다. GitHub Pages에 배포된 정적 HTML은 이미 만들어진 파일이다.

따라서 Streaming SSR은 Vercel/Node 서버처럼 요청 시점 렌더링을 수행하는 배포에서 의미가 크다. 정적 블로그에서는 `loading.tsx`와 Suspense가 개발 구조에는 도움을 줄 수 있지만, 요청 시점 streaming UX를 기대하면 안 된다.

---

## Nuxt와 비교

Nuxt도 `<Suspense>`, lazy hydration, Nitro streaming을 통해 느린 영역을 분리할 수 있다. Next.js App Router에서는 Server Component와 Suspense가 결합되어 “느린 서버 데이터가 있는 컴포넌트”를 자연스럽게 경계로 만들 수 있다.

| 기준             | Nuxt                      | Next.js                           |
| ---------------- | ------------------------- | --------------------------------- |
| 페이지 대기 UI   | pending 상태, layout 처리 | `loading.tsx`                     |
| 부분 대기        | `<Suspense>`              | `<Suspense>`                      |
| 서버 데이터 경계 | `useAsyncData` 위치       | async Server Component 위치       |
| 배포 영향        | Nitro 배포 방식           | dynamic server/static export 차이 |

---

## 체크리스트

- Streaming SSR을 느린 데이터가 전체 화면을 막지 않게 하는 경계 설계로 설명할 수 있다.
- `loading.tsx`와 직접 쓰는 Suspense의 차이를 구분할 수 있다.
- 느린 데이터 fetching을 해당 Server Component 아래로 내릴 수 있다.
- skeleton이 CLS를 만들지 않도록 안정적인 크기를 줄 수 있다.
- Suspense 경계를 너무 많이 쪼갤 때의 비용을 설명할 수 있다.
- static export에서는 요청 시점 streaming 서버가 없다는 점을 알고 있다.
