# 상태 관리가 필요한 순간

"React Query를 쓰면 Redux가 필요 없나요?"

이 질문은 React 생태계에서 자주 등장한다. 대답은 "상황에 따라 다르다"가 아니라, "다루는 상태가 무엇이냐에 달려 있다"이다.

이 시리즈는 Zustand, Redux Toolkit, Jotai를 차례로 다룬다. 그 전에 이번 글에서 먼저 잡아야 할 것이 있다. 상태를 성격별로 구분하는 기준이다. 기준 없이 도구를 선택하면, 매 기능마다 "이걸 store에 넣어야 하나?"를 반복하게 된다.

---

## 오늘 풀 문제

다음 상황을 보자.

쇼핑몰 상품 목록 페이지다. 이 페이지는 세 가지 조건으로 상품을 필터링한다.

- 카테고리: URL 쿼리스트링 `?category=shoes`
- 페이지 번호: URL 쿼리스트링 `?page=2`
- 모달 열림 여부: 필터 패널이 열렸는지

상품 데이터는 API에서 가져온다. 장바구니 수량은 헤더와 상품 목록 두 곳에서 보여준다.

다음 질문이 생긴다.

1. 카테고리와 페이지 번호는 어디서 관리해야 하는가?
2. 상품 목록 데이터는 어디서 관리해야 하는가?
3. 필터 패널 열림 여부는 어디서 관리해야 하는가?
4. 장바구니 수량은 어디서 관리해야 하는가?

이 네 질문에 모두 "useState로 관리한다" 또는 "Redux store에 넣는다"로 통일하면 문제가 생긴다. 상태마다 진실의 근원이 다르기 때문이다.

---

## 정신 모델: 상태의 4가지 성격

상태를 분류하는 기준은 하나다. **진실의 근원이 어디 있는가.**

| 종류                 | 진실의 근원  | 예시                              | 대표 도구                     |
| -------------------- | ------------ | --------------------------------- | ----------------------------- |
| 서버 상태            | 서버 DB      | 상품 목록, 사용자 정보, 주문 내역 | React Query                   |
| URL 상태             | 브라우저 URL | 카테고리, 페이지, 검색어          | useSearchParams, useParams    |
| 로컬 상태            | 컴포넌트     | 모달 열림, 폼 입력값, 토글        | useState, useReducer          |
| 전역 클라이언트 상태 | 앱 메모리    | 장바구니, 인증 토큰, 테마         | Zustand, Redux Toolkit, Jotai |

이 분류가 중요한 이유는 각 종류마다 다루는 방식이 근본적으로 다르기 때문이다.

### 서버 상태: 서버 DB가 진실의 근원

상품 목록은 브라우저에 있는 데이터가 아니다. 서버 DB에 있고, 브라우저는 그 사본을 잠시 가지고 있을 뿐이다.

그래서 서버 상태는 다음 문제를 항상 함께 풀어야 한다.

- 사본은 언제 낡는가? (staleTime)
- 낡았을 때 누가 다시 가져오는가? (refetch 트리거)
- 같은 데이터를 두 컴포넌트가 쓰면 요청을 어떻게 합치는가? (dedup)
- 데이터를 바꿨을 때 화면이 어떻게 알아차리는가? (invalidation)

`useState` + `useEffect`로 이 문제를 직접 풀 수 있다. 하지만 화면이 늘어날수록 캐시 정책, 재시도 정책, 중복 요청 방지가 컴포넌트마다 조금씩 다르게 구현된다. React Query는 이 결정들을 라이브러리 수준에서 표준화한다.

### URL 상태: URL이 진실의 근원

카테고리와 페이지 번호를 `useState`로 관리하면 공유 링크가 깨진다. 사용자가 `shoes` 카테고리 2페이지에서 링크를 복사해 보내도, 받는 사람은 1페이지 전체 카테고리로 시작한다.

URL 상태는 URL이 진실의 근원이다. 브라우저 뒤로가기, 링크 공유, 새로 고침이 모두 URL에서 상태를 복원해야 한다.

`useSearchParams`는 URL 쿼리스트링을 읽고 쓰는 훅이다. 이 훅을 쓰면 상태와 URL이 항상 일치한다.

### 로컬 상태: 컴포넌트가 진실의 근원

필터 패널이 열렸는지 닫혔는지는 그 컴포넌트만 알면 된다. 다른 컴포넌트나 서버가 이 상태를 알 필요가 없다. 페이지를 새로 고침하면 초기 상태로 돌아가도 자연스럽다.

이런 상태는 `useState`나 `useReducer`로 컴포넌트 안에서 관리하는 것이 가장 적절하다. 전역으로 올릴 이유가 없다.

### 전역 클라이언트 상태: 앱 메모리가 진실의 근원

장바구니는 서버 상태와 다르다. 사용자가 "담기"를 눌렀을 때 즉시 UI에 반영되어야 한다. 서버가 실패하면 롤백하더라도, 일단 클라이언트에서 낙관적으로 업데이트한다.

장바구니는 헤더, 상품 목록, 사이드바 등 여러 곳에서 같은 데이터를 봐야 한다. 상위 컴포넌트에서 props로 내리면 prop drilling이 생긴다.

이런 상태는 앱 메모리, 즉 전역 스토어에 둔다. Zustand, Redux Toolkit, Jotai가 이 역할을 한다.

## 최소 코드: 4가지 상태를 한 컴포넌트에서

위 쇼핑몰 예제를 코드로 옮긴다. 각 상태가 어디서 오는지 import 포함해 작성한다.

```tsx
// ProductList.tsx
import { useState } from "react";
import { useSearchParams } from "react-router-dom";
import { useQuery } from "@tanstack/react-query";
import { useCartStore } from "@/store/cartStore"; // Zustand store

interface Product {
  id: string;
  name: string;
  price: number;
  category: string;
}

interface ProductsResponse {
  items: Product[];
  totalCount: number;
}

async function fetchProducts(
  category: string,
  page: number,
): Promise<ProductsResponse> {
  const params = new URLSearchParams({ category, page: String(page) });
  const res = await fetch(`/api/products?${params}`);
  if (!res.ok) throw new Error(`fetch failed: ${res.status}`);
  return (await res.json()) as ProductsResponse;
}

export function ProductList() {
  // 1. URL 상태 — URL이 진실의 근원
  const [searchParams, setSearchParams] = useSearchParams();
  const category = searchParams.get("category") ?? "all";
  const page = Number(searchParams.get("page") ?? "1");

  // 2. 로컬 상태 — 컴포넌트가 진실의 근원
  const [isFilterOpen, setIsFilterOpen] = useState(false);

  // 3. 서버 상태 — 서버 DB가 진실의 근원
  const productsQuery = useQuery({
    queryKey: ["products", { category, page }],
    queryFn: () => fetchProducts(category, page),
  });

  // 4. 전역 클라이언트 상태 — 앱 메모리가 진실의 근원
  const { items: cartItems, addItem } = useCartStore();

  function handleCategoryChange(next: string) {
    // URL을 업데이트하면 category 변수도 자동으로 바뀐다
    setSearchParams({ category: next, page: "1" });
  }

  function handlePageChange(next: number) {
    setSearchParams({ category, page: String(next) });
  }

  if (productsQuery.isPending) return <div>불러오는 중...</div>;
  if (productsQuery.isError)
    return <div>오류: {productsQuery.error.message}</div>;
  if (productsQuery.data.items.length === 0) return <div>상품이 없다.</div>;

  return (
    <div>
      {/* URL 상태에서 파생된 UI */}
      <select
        value={category}
        onChange={(e) => handleCategoryChange(e.target.value)}
      >
        <option value="all">전체</option>
        <option value="shoes">신발</option>
        <option value="bags">가방</option>
      </select>

      {/* 로컬 상태 */}
      <button onClick={() => setIsFilterOpen((prev) => !prev)}>
        {isFilterOpen ? "필터 닫기" : "필터 열기"}
      </button>
      {isFilterOpen && <div>필터 패널</div>}

      {/* 서버 상태에서 렌더링 */}
      <ul>
        {productsQuery.data.items.map((product) => {
          const inCart = cartItems.some((item) => item.id === product.id);
          return (
            <li key={product.id}>
              <span>{product.name}</span>
              <span>{product.price.toLocaleString()}원</span>
              {/* 전역 클라이언트 상태에서 파생된 UI */}
              <button onClick={() => addItem(product)} disabled={inCart}>
                {inCart ? "담김" : "담기"}
              </button>
            </li>
          );
        })}
      </ul>

      {/* 페이지네이션 — URL 상태 */}
      <div>
        <button disabled={page <= 1} onClick={() => handlePageChange(page - 1)}>
          이전
        </button>
        <span>{page}페이지</span>
        <button
          disabled={
            productsQuery.data.items.length <
            productsQuery.data.totalCount / page
          }
          onClick={() => handlePageChange(page + 1)}
        >
          다음
        </button>
      </div>
    </div>
  );
}
```

이 컴포넌트에서 상태마다 출처가 다르다는 점을 확인한다.

- `category`, `page`: URL에서 읽고 URL에 쓴다
- `isFilterOpen`: 컴포넌트 안에서만 존재한다
- `productsQuery.data`: React Query 캐시에서 온다
- `cartItems`: Zustand store에서 온다

## 실무 코드: Zustand cartStore 정의

위 예제에서 `useCartStore`를 쓰려면 store 정의가 필요하다. 간단한 장바구니 store다.

```ts
// src/store/cartStore.ts
import { create } from "zustand";
import { persist } from "zustand/middleware";

interface CartItem {
  id: string;
  name: string;
  price: number;
  quantity: number;
}

interface CartState {
  items: CartItem[];
  addItem: (product: { id: string; name: string; price: number }) => void;
  removeItem: (id: string) => void;
  clearCart: () => void;
  totalPrice: () => number;
}

export const useCartStore = create<CartState>()(
  persist(
    (set, get) => ({
      items: [],

      addItem(product) {
        set((state) => {
          const existing = state.items.find((i) => i.id === product.id);
          if (existing) {
            // 이미 담긴 상품이면 수량만 증가
            return {
              items: state.items.map((i) =>
                i.id === product.id ? { ...i, quantity: i.quantity + 1 } : i,
              ),
            };
          }
          return {
            items: [...state.items, { ...product, quantity: 1 }],
          };
        });
      },

      removeItem(id) {
        set((state) => ({
          items: state.items.filter((i) => i.id !== id),
        }));
      },

      clearCart() {
        set({ items: [] });
      },

      totalPrice() {
        return get().items.reduce(
          (sum, item) => sum + item.price * item.quantity,
          0,
        );
      },
    }),
    {
      name: "cart-storage", // localStorage 키
    },
  ),
);
```

`persist` 미들웨어를 쓰면 새로 고침해도 장바구니가 유지된다. 서버 상태와 달리 전역 클라이언트 상태는 로컬 스토리지에 저장해도 진실의 근원이 변하지 않는다. 앱이 그 데이터를 관리한다.

## 실무 코드: URL 상태의 타입 안전성 확보

`useSearchParams`를 직접 쓰면 값이 항상 `string | null`로 온다. 실무에서는 파싱과 기본값 처리를 한 곳에서 담당하는 훅으로 감싸는 편이 안전하다.

```ts
// src/hooks/useProductFilters.ts
import { useCallback } from "react";
import { useSearchParams } from "react-router-dom";

const VALID_CATEGORIES = ["all", "shoes", "bags", "accessories"] as const;
type Category = (typeof VALID_CATEGORIES)[number];

interface ProductFilters {
  category: Category;
  page: number;
  setCategory: (category: Category) => void;
  setPage: (page: number) => void;
  reset: () => void;
}

function parseCategory(raw: string | null): Category {
  if (raw === null) return "all";
  return VALID_CATEGORIES.includes(raw as Category) ? (raw as Category) : "all";
}

function parsePage(raw: string | null): number {
  const n = Number(raw);
  return Number.isFinite(n) && n >= 1 ? Math.floor(n) : 1;
}

export function useProductFilters(): ProductFilters {
  const [searchParams, setSearchParams] = useSearchParams();

  const category = parseCategory(searchParams.get("category"));
  const page = parsePage(searchParams.get("page"));

  const setCategory = useCallback(
    (next: Category) => {
      setSearchParams((prev) => {
        const next_ = new URLSearchParams(prev);
        next_.set("category", next);
        next_.set("page", "1"); // 카테고리 바뀌면 1페이지로 리셋
        return next_;
      });
    },
    [setSearchParams],
  );

  const setPage = useCallback(
    (next: number) => {
      setSearchParams((prev) => {
        const next_ = new URLSearchParams(prev);
        next_.set("page", String(next));
        return next_;
      });
    },
    [setSearchParams],
  );

  const reset = useCallback(() => {
    setSearchParams({});
  }, [setSearchParams]);

  return { category, page, setCategory, setPage, reset };
}
```

이 훅은 두 가지를 보장한다.

- `category`는 항상 `VALID_CATEGORIES` 중 하나다. URL에 잘못된 값이 들어와도 기본값으로 fallback한다.
- `page`는 항상 1 이상의 정수다.

URL 상태를 `useState`처럼 쓰는 API를 제공하되, 타입 안전성과 유효성 검사는 훅 안에 숨긴다.

## 망하는 패턴

### 패턴 1: 서버 상태를 Zustand store에 넣기

```ts
// 잘못된 방식
const useProductStore = create<{
  products: Product[];
  loading: boolean;
  error: Error | null;
  fetchProducts: (category: string) => Promise<void>;
}>((set) => ({
  products: [],
  loading: false,
  error: null,

  async fetchProducts(category) {
    set({ loading: true, error: null });
    try {
      const res = await fetch(`/api/products?category=${category}`);
      if (!res.ok) throw new Error("failed");
      const data = (await res.json()) as Product[];
      set({ products: data, loading: false });
    } catch (err) {
      set({ error: err as Error, loading: false });
    }
  },
}));
```

이 방식은 store 자체가 캐시, 로딩, 에러, 재시도, 만료를 직접 관리해야 한다. 화면이 여러 개가 되면 각 화면마다 `fetchProducts` 호출 시점을 조율해야 하고, 탭을 다녀온 후 최신 데이터를 보여주는 로직도 직접 만들어야 한다.

이 코드가 계속 늘어나면 결국 React Query가 하는 일을 수동으로 다시 만드는 꼴이 된다.

**원인**: 서버 상태는 "언제 낡는가"와 "어떻게 갱신하는가"라는 시간 개념이 함께 필요하다. Zustand store는 이 개념이 없다. React Query는 이 개념이 핵심 설계다.

**고치는 방법**: 서버 데이터는 React Query `useQuery`로 가져온다. store에는 장바구니처럼 순수하게 클라이언트가 소유한 상태만 둔다.

---

### 패턴 2: UI 상태를 전역 store에 넣기

```ts
// 잘못된 방식
const useUIStore = create<{
  isFilterOpen: boolean;
  toggleFilter: () => void;
  isModalOpen: boolean;
  modalType: "confirm" | "alert" | null;
  openModal: (type: "confirm" | "alert") => void;
  closeModal: () => void;
  activeTabIndex: number;
  setActiveTabIndex: (i: number) => void;
}>((set) => ({
  isFilterOpen: false,
  toggleFilter: () => set((s) => ({ isFilterOpen: !s.isFilterOpen })),
  isModalOpen: false,
  modalType: null,
  openModal: (type) => set({ isModalOpen: true, modalType: type }),
  closeModal: () => set({ isModalOpen: false, modalType: null }),
  activeTabIndex: 0,
  setActiveTabIndex: (i) => set({ activeTabIndex: i }),
}));
```

이 패턴이 퍼지면 store가 UI 상세 사항으로 가득 찬다. 모달 타입, 탭 인덱스, 각 컴포넌트의 열림 여부가 전역에 모인다.

**원인**: 이 상태들은 컴포넌트 안에서만 쓰인다. 다른 컴포넌트가 이 값을 공유할 이유가 없다. 그런데 store에 두면 컴포넌트가 사라진 후에도 store에 남아 있다. 메모리가 낭비되고, store 코드가 비대해진다.

**고치는 방법**: 컴포넌트가 독점으로 소유하는 상태는 `useState`로 컴포넌트 안에 둔다. 여러 컴포넌트가 실제로 공유해야 하는 상태만 store로 올린다.

### 패턴 3: URL 상태를 useState로 관리하기

```tsx
// 잘못된 방식
export function ProductListPage() {
  const [category, setCategory] = useState("all");
  const [page, setPage] = useState(1);

  const productsQuery = useQuery({
    queryKey: ["products", { category, page }],
    queryFn: () => fetchProducts(category, page),
  });

  // ...
}
```

이 코드는 동작한다. 하지만 다음 문제가 생긴다.

- 사용자가 URL을 복사해 보내면 상대방은 기본값(전체, 1페이지)을 본다.
- 뒤로가기를 누르면 이전 페이지가 아니라 초기 상태로 간다.
- 페이지를 새로 고침하면 필터가 초기화된다.

**원인**: URL 상태는 URL이 진실의 근원이어야 한다. `useState`로 관리하면 URL과 상태가 분리된다. URL에는 아무것도 기록되지 않는다.

**고치는 방법**: 공유되어야 하거나 새로 고침 후에도 유지되어야 하는 필터는 `useSearchParams`로 URL에 쓴다.

---

### 패턴 4: 상태 구분 없이 하나의 store에 모두 넣기

```ts
// 잘못된 방식 — 모든 것을 하나에
const useAppStore = create<{
  // 서버 상태 (React Query가 할 일)
  products: Product[];
  productsLoading: boolean;

  // URL 상태 (useSearchParams가 할 일)
  category: string;
  page: number;

  // 로컬 상태 (useState가 할 일)
  isFilterOpen: boolean;

  // 전역 클라이언트 상태 (store가 할 일)
  cartItems: CartItem[];
}>((set) => ({
  // ...
}));
```

이 방식은 초기에는 편리하지만 팀이 커지거나 기능이 늘어나면 다음 문제가 생긴다.

- 서버 데이터가 낡았는지 판단하는 기준이 없다.
- URL을 바꿔도 `category`와 동기화되지 않는다.
- 컴포넌트가 마운트될 때마다 store를 초기화해야 한다.
- store가 비대해져 어떤 상태가 어디서 쓰이는지 파악하기 어렵다.

**원인**: 성격이 다른 상태를 같은 도구로 관리하면 각 도구가 해결하는 문제를 직접 다시 풀어야 한다.

## Vue/Nuxt 비교

Vue 생태계도 동일한 분류가 적용된다.

| 상태 종류            | Vue/Nuxt 도구                     | React 도구                     |
| -------------------- | --------------------------------- | ------------------------------ |
| 서버 상태            | `@tanstack/vue-query`, `useFetch` | `@tanstack/react-query`        |
| URL 상태             | `useRoute`, `useRouter`           | `useSearchParams`, `useParams` |
| 로컬 상태            | `ref`, `reactive`, `computed`     | `useState`, `useReducer`       |
| 전역 클라이언트 상태 | Pinia                             | Zustand, Redux Toolkit, Jotai  |

### Pinia와 Zustand의 대응

Pinia의 defineStore와 Zustand의 create는 같은 역할을 한다.

```ts
// Pinia (Vue)
export const useCartStore = defineStore("cart", () => {
  const items = ref<CartItem[]>([]);

  function addItem(product: Product) {
    const existing = items.value.find((i) => i.id === product.id);
    if (existing) {
      existing.quantity += 1;
    } else {
      items.value.push({ ...product, quantity: 1 });
    }
  }

  function removeItem(id: string) {
    items.value = items.value.filter((i) => i.id !== id);
  }

  const totalPrice = computed(() =>
    items.value.reduce((sum, i) => sum + i.price * i.quantity, 0),
  );

  return { items, addItem, removeItem, totalPrice };
});
```

```ts
// Zustand (React)
export const useCartStore = create<CartState>()((set, get) => ({
  items: [],
  addItem(product) {
    set((state) => {
      const existing = state.items.find((i) => i.id === product.id);
      if (existing) {
        return {
          items: state.items.map((i) =>
            i.id === product.id ? { ...i, quantity: i.quantity + 1 } : i,
          ),
        };
      }
      return { items: [...state.items, { ...product, quantity: 1 }] };
    });
  },
  removeItem(id) {
    set((state) => ({ items: state.items.filter((i) => i.id !== id) }));
  },
  totalPrice() {
    return get().items.reduce((sum, i) => sum + i.price * i.quantity, 0);
  },
}));
```

### URL 상태: useRoute vs useSearchParams

```html
<!-- Vue: useRoute로 URL 쿼리스트링 읽기 -->
<script setup lang="ts">
  import { computed } from "vue";
  import { useRoute, useRouter } from "vue-router";

  const route = useRoute();
  const router = useRouter();

  const category = computed(() => (route.query.category as string) ?? "all");
  const page = computed(() => Number(route.query.page ?? "1"));

  function setCategory(next: string) {
    router.push({ query: { ...route.query, category: next, page: "1" } });
  }
</script>
```

```tsx
// React: useSearchParams로 URL 쿼리스트링 읽기
import { useSearchParams } from "react-router-dom";

const [searchParams, setSearchParams] = useSearchParams();
const category = searchParams.get("category") ?? "all";
const page = Number(searchParams.get("page") ?? "1");

function setCategory(next: string) {
  setSearchParams((prev) => {
    const next_ = new URLSearchParams(prev);
    next_.set("category", next);
    next_.set("page", "1");
    return next_;
  });
}
```

두 코드는 같은 개념이다. URL이 진실의 근원이고, 읽기와 쓰기가 모두 URL을 통한다.

### 서버 상태: useFetch vs useQuery

Nuxt의 `useFetch`는 React Query와 유사하지만 SSR 통합이 기본으로 제공된다.

```html
<!-- Nuxt: useFetch -->
<script setup lang="ts">
  const route = useRoute();
  const category = computed(() => route.query.category ?? "all");

  const { data, pending, error } = await useFetch("/api/products", {
    query: { category },
    // category가 바뀌면 자동으로 재요청
    watch: [category],
  });
</script>
```

```tsx
// React: useQuery
const { data, isPending, isError } = useQuery({
  queryKey: ["products", { category }],
  queryFn: () => fetchProducts(category),
  // queryKey의 category가 바뀌면 자동으로 재요청
});
```

두 도구 모두 "서버 데이터는 외부에 진실이 있다"는 개념을 구현한다. 의존값이 바뀌면 자동으로 재요청하고, 결과를 캐시한다.

Pinia를 쓰던 팀이 서버 데이터를 Pinia에 넣고 있었다면, Vue Query나 Nuxt의 `useFetch`로 옮기는 것이 좋다. React 팀도 마찬가지로 Zustand나 Redux에 서버 데이터를 넣고 있었다면 React Query로 옮긴다.

## 상태 배치 결정 흐름

코드를 작성할 때 상태를 어디에 둘지 결정하는 흐름이다.

```
새로운 상태가 필요하다
       │
       ▼
서버에서 오는 데이터인가?
       │ 예
       ▼
   React Query
(useQuery / useMutation)
       │ 아니오
       ▼
URL에 반영되어야 하는가?
(공유 링크, 새로 고침 유지)
       │ 예
       ▼
   useSearchParams
   (또는 useParams)
       │ 아니오
       ▼
여러 컴포넌트가 공유하는가?
Props로 내리기 불편한가?
       │ 예
       ▼
 전역 클라이언트 store
(Zustand / RTK / Jotai)
       │ 아니오
       ▼
    useState / useReducer
    (컴포넌트 로컬 상태)
```

이 흐름에서 핵심은 "가장 낮은 단계로 내리는 것"이다. 전역 store가 필요한 상태가 사실 컴포넌트 로컬로 충분한 경우가 많다.

## 도구 선택의 실제 기준

"React Query를 쓰면 Redux가 필요 없나요?"로 돌아간다.

정확한 질문으로 바꾸면 이렇다.

- "React Query로 서버 상태를 관리하면 서버 데이터를 Redux에 넣을 필요가 없다." → 맞다.
- "React Query를 쓰면 전역 클라이언트 상태 도구가 필요 없다." → 틀리다.

React Query는 서버 상태 전용이다. 장바구니처럼 클라이언트가 소유한 상태, 인증 토큰, 다크모드 설정처럼 여러 컴포넌트가 공유하는 상태는 여전히 전역 클라이언트 상태 도구가 필요하다.

단, 전역 클라이언트 상태가 예상보다 적을 때가 많다. 서버 데이터를 React Query로 빼고 나면, 남은 전역 상태는 몇 가지 되지 않는 경우가 흔하다.

도구 별 적용 기준을 정리한다.

| 도구            | 쓰는 기준                                                          |
| --------------- | ------------------------------------------------------------------ |
| React Query     | API 호출 결과, 페이지네이션, 무한 스크롤, 낙관적 업데이트          |
| useSearchParams | 필터, 페이지, 검색어, 정렬 — 공유 링크 또는 새로 고침 후 복원 필요 |
| useState        | 모달, 토글, 폼 임시값 — 그 컴포넌트만 아는 상태                    |
| Zustand         | 장바구니, 테마, 사이드바 열림 — 앱 전역, 단순한 구조               |
| Redux Toolkit   | 복잡한 상태 전이, 이력 추적, DevTools 필요                         |
| Jotai           | 컴포넌트 단위 원자, 파생 상태가 복잡한 경우                        |

이 시리즈에서는 Zustand, Redux Toolkit, Jotai를 각각 한 편씩 다룬다. 도구마다 해결하는 문제의 모양이 다르다.
