## 한 장 핵심

React Compiler는 React 코드를 빌드 타임에 분석해 불필요한 재계산과 re-render를 줄이는 최적화 도구다. 핵심 변화는 개발자가 모든 `memo`, `useMemo`, `useCallback` 지점을 손으로 맞추는 방식에서, 컴파일러가 안전하다고 판단한 코드를 자동으로 메모이제이션하는 방향으로 이동한다는 점이다.

그렇다고 성능 사고가 사라지는 것은 아니다. 오히려 더 중요한 기준은 React의 규칙을 지키는 코드다. render가 순수하고, props/state를 mutate하지 않고, Hook 규칙을 지키는 코드일수록 컴파일러가 안전하게 최적화할 여지가 커진다.

## Vue와 비교

Vue 개발자는 컴파일러와 런타임 반응성의 도움을 이미 많이 받는다. 템플릿 컴파일러가 정적 노드를 hoist하고, reactive dependency가 effect를 깨운다. 그래서 "프레임워크가 알아서 추적한다"는 감각이 강하다.

React는 오랫동안 개발자가 참조 안정성을 직접 관리하는 쪽에 가까웠다. `memo`로 컴포넌트를 감싸고, `useCallback`으로 함수를 안정화하고, `useMemo`로 계산 결과를 캐시했다. React Compiler는 이 부담을 줄이려는 방향이다.

하지만 Vue의 fine-grained reactivity와 같은 모델이 되는 것은 아니다. React Compiler는 React 컴포넌트 코드의 순수성과 데이터 흐름을 분석해 최적화한다. 따라서 컴파일러 시대의 React 학습은 "어디에 useMemo를 붙일까"보다 "컴파일러가 이해할 수 있는 순수한 컴포넌트를 어떻게 쓸까"에 가까워진다.

## React 내부 원리

공식 문서는 React Compiler가 기본적으로 분석과 휴리스틱에 따라 코드를 메모이즈한다고 설명한다. 예전에는 다음처럼 수동 memo가 많았다.

```tsx
import { memo, useCallback, useMemo } from "react";

const Item = memo(function Item({ label, onClick }: { label: string; onClick: () => void }) {
  return <button onClick={onClick}>{label}</button>;
});

export function List({ items }: { items: string[] }) {
  const sorted = useMemo(() => [...items].sort(), [items]);
  const handleClick = useCallback((label: string) => {
    console.log(label);
  }, []);

  return sorted.map((item) => (
    <Item key={item} label={item} onClick={() => handleClick(item)} />
  ));
}
```

여기에는 subtle한 함정도 있다. `handleClick`은 안정적이어도 `() => handleClick(item)`은 매 render마다 새 함수다. React Compiler는 이런 패턴을 더 넓은 문맥에서 분석해 자동 최적화를 시도한다.

수동 memo는 사라지는 것이 아니라 역할이 바뀐다. 공식 문서는 `useMemo`와 `useCallback`이 compiler 환경에서도 더 정밀한 통제가 필요한 escape hatch로 계속 사용될 수 있다고 설명한다. 기존 코드의 memo를 무작정 제거하는 것도 권장되지 않는다. 동작과 성능을 측정하면서 점진적으로 다뤄야 한다.

## 실무에서 터지는 문제

- **Compiler를 켜면 나쁜 코드가 빨라진다고 믿음**: mutation, impure render, 잘못된 Hook 사용은 최적화 대상 이전에 버그다.
- **기존 memo를 일괄 삭제**: 컴파일 결과와 성능 특성이 바뀔 수 있다. 작은 범위에서 측정하며 제거한다.
- **성능 측정을 생략**: compiler가 있어도 병목은 네트워크, layout, 큰 DOM, 무거운 계산 등 다양하다. Profiler와 브라우저 Performance를 같이 본다.
- **빌드 도구 지원을 확인하지 않음**: Next.js 등 프레임워크별 compiler 지원 방식과 버전을 확인해야 한다.
