## 한 장 핵심

React에서 **render**는 브라우저가 화면을 그리는 일이 아니다. React가 컴포넌트 함수를 호출해서 "지금 props와 state라면 UI는 어떤 모양이어야 하는가"를 계산하는 단계다. 이 계산 결과는 React element 트리이고, 실제 DOM 변경은 그 다음 commit 단계에서 필요한 만큼만 일어난다.

이 구분을 놓치면 "부모가 렌더되면 DOM도 전부 다시 그려진다", "render가 느리니 곧바로 DOM이 느리다" 같은 오해가 생긴다. 정확한 모델은 세 단계다. 업데이트가 render를 요청하고, React가 컴포넌트를 호출해 새 UI 스냅샷을 계산하고, 이전 결과와 비교해 필요한 DOM 변경만 commit한다.

## Vue와 비교

Vue는 템플릿과 반응형 의존성 추적을 중심에 둔다. 컴포넌트의 render effect가 실행될 때 읽힌 reactive 값이 추적되고, 그 값이 바뀌면 관련 effect가 다시 돈다. 개발자는 대개 "어떤 값이 바뀌었으니 그 값을 읽는 템플릿 부분이 갱신된다"에 가깝게 사고한다.

React는 의존성을 자동 추적하지 않는다. 컴포넌트 함수 전체가 다시 호출되고, 그 호출 결과로 새 React element 트리를 만든다. 그래서 React에서 중요한 질문은 "어떤 변수가 추적되는가"보다 "이 render 호출은 순수한가, 같은 입력이면 같은 UI를 반환하는가"다.

이 차이 때문에 React 컴포넌트 본문에는 네트워크 요청, 타이머 등록, DOM 직접 조작 같은 부수 효과를 두면 안 된다. Vue의 `watchEffect`처럼 함수 안에서 읽은 값이 자동으로 추적되는 모델이 아니며, React render는 언제든 다시 호출될 수 있는 순수 계산으로 유지해야 한다.

## React 내부 원리

공식 문서는 화면 업데이트를 **Trigger → Render → Commit**으로 설명한다. 최초 렌더는 `createRoot(...).render(...)`로 시작하고, 이후 렌더는 state 업데이트나 상위 컴포넌트의 렌더로 요청된다. render 단계에서 React는 컴포넌트 함수를 호출해 화면에 무엇이 있어야 하는지 계산한다.

```tsx
import { useState } from "react";

export default function Counter() {
  const [count, setCount] = useState(0);

  console.log("render", count);

  return (
    <button onClick={() => setCount(count + 1)}>
      count: {count}
    </button>
  );
}
```

버튼을 누르면 `setCount`가 DOM을 직접 바꾸는 것이 아니다. React에게 다음 render가 필요하다고 알리고, React는 `Counter()`를 다시 호출해서 `count: 1`이라는 새 결과를 얻는다. 이후 commit 단계에서 이전 결과와 다른 텍스트만 실제 DOM에 반영된다.

개발 모드의 Strict Mode에서는 순수성 문제를 드러내기 위해 컴포넌트 함수를 추가로 호출할 수 있다. 이 동작을 "React가 버그로 두 번 렌더한다"로 읽으면 안 된다. render 안에 부수 효과를 넣지 않았는지 확인하게 만드는 개발 도구적 동작이다.

## 실무에서 터지는 문제

- **render와 paint를 섞어 말함**: React render는 컴포넌트 호출이고, 브라우저 paint는 DOM 변경 후 화면을 칠하는 브라우저 작업이다. 성능 문제를 볼 때 둘을 분리해야 원인을 찾을 수 있다.
- **컴포넌트 본문에서 부수 효과 실행**: 함수 본문에서 `fetch`, `setInterval`, `localStorage.setItem` 같은 코드를 실행하면 render 횟수와 외부 작업 횟수가 묶인다. 외부 시스템 동기화는 event handler, framework loader, 또는 effect로 옮긴다.
- **부모 render가 곧 DOM 전체 교체라고 착각**: 부모가 다시 호출되면 자식 element도 다시 계산될 수 있지만, commit 단계에서 DOM은 필요한 부분만 바뀐다. 렌더 비용과 DOM 변경 비용은 같은 말이 아니다.
- **무조건 memo부터 적용**: render가 호출된다는 사실만으로 병목은 아니다. 먼저 Profiler나 로그로 실제 비용을 확인하고, 동일 props에서 비싼 계산이 반복될 때만 `memo`, `useMemo` 같은 도구를 검토한다.
