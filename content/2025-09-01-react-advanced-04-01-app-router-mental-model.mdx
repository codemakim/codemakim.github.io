# App Router의 정신 모델

App Router를 파일 기반 라우터 정도로 이해하면 실무에서 금방 막힌다. 페이지는 잘 나뉘는데 어느 코드를 서버에 둘지, 어디부터 클라이언트로 내려야 할지, 캐시가 왜 기대와 다르게 동작하는지 설명하기 어려워진다.

Next.js App Router는 라우팅 도구라기보다 화면 경계, 데이터 로딩, 캐시, 오류 처리, 배포 타깃을 한 번에 설계하게 만드는 프레임워크다. 이 시리즈의 목표는 API 이름을 외우는 것이 아니라 운영 가능한 Next 앱을 판단하는 기준을 갖는 것이다.

---

## 왜 필요한가

게시글 상세 페이지를 만든다고 가정한다.

- 제목과 본문은 DB에서 읽는다.
- 좋아요 버튼은 브라우저에서 클릭된다.
- 댓글 목록은 느릴 수 있다.
- 존재하지 않는 글은 404가 되어야 한다.
- GitHub Pages에 정적 배포하면 서버에서 동작하는 기능은 쓸 수 없다.

Pages Router식 사고로는 “페이지 컴포넌트 하나”에서 시작하기 쉽다. App Router식 사고는 다르다. 먼저 경계를 나눈다.

```txt
app/posts/[slug]/
  layout.tsx      공통 껍데기
  page.tsx        게시글 데이터와 본문
  loading.tsx     느린 구간의 대기 화면
  error.tsx       이 segment 안의 시스템 실패
  not-found.tsx   이 segment 안의 404
```

파일명은 규칙이지만, 실무에서는 화면 경계 설계 도구로 읽어야 한다.

---

## route segment와 nested layout

App Router의 URL은 segment 단위로 쪼개진다.

```txt
app/
  layout.tsx
  posts/
    layout.tsx
    page.tsx
    [slug]/
      page.tsx
```

`app/layout.tsx`는 전체 앱을 감싸고, `app/posts/layout.tsx`는 `/posts` 아래만 감싼다. 이 구조 덕분에 페이지 전환 시 유지할 UI와 다시 렌더링할 UI를 분리할 수 있다.

```tsx
// app/posts/layout.tsx
import type { ReactNode } from "react";

export default function PostsLayout({ children }: { children: ReactNode }) {
  return (
    <section>
      <nav aria-label="posts">
        <a href="/posts">전체 글</a>
        <a href="/posts/tags">태그</a>
      </nav>
      {children}
    </section>
  );
}
```

이 layout은 `/posts`와 `/posts/[slug]`에서 공유된다. 따라서 목록과 상세가 같은 사이드바, 탭, 필터 UI를 공유할 때 유리하다.

---

## Server Component가 기본값이다

App Router에서 컴포넌트는 기본적으로 Server Component다. 별도 지시어가 없으면 서버에서 실행되고, 브라우저 번들에 컴포넌트 코드가 들어가지 않는다.

```tsx
// app/posts/[slug]/page.tsx
import { notFound } from "next/navigation";
import { getPost } from "@/app/lib/posts";

type PageProps = {
  params: Promise<{ slug: string }>;
};

export default async function PostPage({ params }: PageProps) {
  const { slug } = await params;
  const post = await getPost(slug);

  if (!post) {
    notFound();
  }

  return (
    <article>
      <h1>{post.title}</h1>
      <p>{post.description}</p>
      <div>{post.body}</div>
    </article>
  );
}
```

여기서는 DB 접근, 파일 읽기, 비밀 키가 필요한 서버 SDK 호출을 서버에서 끝낼 수 있다. 클라이언트에는 결과로 만들어진 UI만 내려간다.

중요한 기준은 단순하다.

- 데이터 읽기, 권한 검사, 파일/DB 접근은 서버에 둔다.
- 클릭, 입력, 드래그, 브라우저 API는 클라이언트에 둔다.
- 클라이언트 컴포넌트는 가능한 leaf에 가깝게 둔다.

---

## `"use client"` 경계

상호작용이 필요한 컴포넌트는 파일 최상단에 `"use client"`를 둔다.

```tsx
// app/posts/[slug]/LikeButton.tsx
"use client";

import { useState } from "react";

type LikeButtonProps = {
  initialCount: number;
};

export function LikeButton({ initialCount }: LikeButtonProps) {
  const [count, setCount] = useState(initialCount);

  return (
    <button type="button" onClick={() => setCount((value) => value + 1)}>
      좋아요 {count}
    </button>
  );
}
```

Server Component는 Client Component를 렌더링할 수 있다.

```tsx
// app/posts/[slug]/page.tsx
import { LikeButton } from "./LikeButton";

export default async function PostPage() {
  const post = await getPost("app-router");

  return (
    <article>
      <h1>{post.title}</h1>
      <LikeButton initialCount={post.likeCount} />
    </article>
  );
}
```

하지만 Client Component에 넘기는 props는 직렬화 가능해야 한다. 함수, DB 커넥션, 클래스 인스턴스, `Date` 객체를 아무 생각 없이 넘기면 경계가 깨진다. 날짜는 ISO 문자열이나 `YYYY-MM-DD` 문자열처럼 명시적으로 직렬화된 값으로 넘기는 편이 안전하다.

## 파일 규칙은 화면의 계약이다

App Router의 특수 파일은 “자동으로 불리는 파일”이 아니라 route segment의 계약이다.

| 파일            | 역할                     | 설계 질문                               |
| --------------- | ------------------------ | --------------------------------------- |
| `layout.tsx`    | segment 공통 UI          | 어떤 UI가 페이지 이동 후에도 유지되는가 |
| `page.tsx`      | URL에 직접 대응되는 화면 | 이 경로의 핵심 데이터는 무엇인가        |
| `loading.tsx`   | 대기 UI                  | 어떤 느린 데이터가 전체 화면을 막는가   |
| `error.tsx`     | segment 오류 경계        | 시스템 실패를 어디까지 격리할 것인가    |
| `not-found.tsx` | segment 404              | 없는 리소스를 어떻게 보여줄 것인가      |

성공 경로만 만들면 데모는 된다. 운영 가능한 앱은 대기, 실패, 404, 권한 실패의 경계를 같이 설계한다.

## Static export 제약을 먼저 생각한다

이 블로그 프로젝트는 GitHub Pages 정적 배포를 사용한다.

```js
// next.config.js
const nextConfig = {
  output: "export",
  trailingSlash: true,
};

module.exports = nextConfig;
```

정적 배포에서는 빌드 결과물이 HTML/CSS/JS 파일로 생성된다. 이 방식은 빠르고 단순하지만 요청 시점의 Node.js 서버가 없다. 따라서 다음 기능은 그대로 사용할 수 없거나 별도 서버/플랫폼이 필요하다.

- Server Actions
- Route Handlers
- Proxy
- ISR
- 기본 `next/image` 서버 최적화
- 요청 시점의 쿠키/헤더 기반 동적 렌더링

이 제약은 단점만은 아니다. 공개 블로그처럼 빌드 시점에 대부분의 콘텐츠가 확정되는 앱은 static export가 잘 맞는다. 반대로 관리자 기능, 웹훅, 실시간 데이터, 서버 인증이 핵심이면 Vercel, Node 서버, 별도 API 서버 같은 배포 타깃을 먼저 정해야 한다.

## 최소 예제: 게시글 상세의 경계

```tsx
// app/posts/[slug]/page.tsx
import { notFound } from "next/navigation";
import { LikeButton } from "./LikeButton";

type PageProps = {
  params: Promise<{ slug: string }>;
};

async function getPost(slug: string) {
  const response = await fetch(`https://example.com/api/posts/${slug}`);

  if (response.status === 404) {
    return null;
  }

  if (!response.ok) {
    throw new Error("게시글을 불러오지 못했다.");
  }

  return response.json() as Promise<{
    title: string;
    body: string;
    likeCount: number;
  }>;
}

export default async function PostPage({ params }: PageProps) {
  const { slug } = await params;
  const post = await getPost(slug);

  if (!post) {
    notFound();
  }

  return (
    <article>
      <h1>{post.title}</h1>
      <p>{post.body}</p>
      <LikeButton initialCount={post.likeCount} />
    </article>
  );
}
```

이 예제에서 `page.tsx`는 서버에서 데이터를 읽고, `LikeButton`만 클라이언트로 내려간다. `404`는 `notFound()`로 보내고, 네트워크 실패는 `error.tsx`가 처리하도록 `throw`한다.

## Nuxt와 비교

Nuxt도 파일 기반 라우팅과 layout, 서버 렌더링, 서버 API를 제공한다. 차이는 기본 사고방식에 있다.

| 기준            | Nuxt                          | Next.js App Router              |
| --------------- | ----------------------------- | ------------------------------- |
| 라우팅          | `pages/` 기반 자동 라우팅     | `app/` segment 기반 라우팅      |
| 공통 UI         | `layouts/`                    | nested `layout.tsx`             |
| 데이터 로딩     | `useAsyncData`, server routes | Server Component의 async 렌더링 |
| 클라이언트 경계 | Vue 컴포넌트 hydration 중심   | `"use client"` 경계 명시        |
| 서버 API        | Nitro server routes           | Route Handlers / Server Actions |

Nuxt는 Vue 생태계의 데이터 훅 중심 사고가 강하고, App Router는 React Server Components를 중심으로 서버/클라이언트 경계를 더 노골적으로 드러낸다.

---

## 체크리스트

- App Router를 URL 매핑이 아니라 렌더링/데이터/캐시/배포 모델로 설명할 수 있다.
- `layout.tsx`, `page.tsx`, `loading.tsx`, `error.tsx`, `not-found.tsx`의 역할을 구분할 수 있다.
- Server Component가 기본값이라는 사실을 기준으로 컴포넌트 위치를 정할 수 있다.
- `"use client"`를 필요한 leaf 컴포넌트에만 둘 수 있다.
- Client Component props가 직렬화 가능해야 한다는 제약을 설명할 수 있다.
- static export에서 Server Actions, Route Handlers, Proxy, ISR, 기본 이미지 최적화가 제한된다는 점을 알고 있다.
