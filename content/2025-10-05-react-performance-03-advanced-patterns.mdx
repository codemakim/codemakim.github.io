# 대형 UI — 가상화와 입력 반응성

렌더링 비용을 줄이고 코드 분할을 해도, 화면 자체가 너무 크면 느리다. 1000개 상품 카드, 복잡한 테이블, 실시간 필터링, 드래그 가능한 보드처럼 많은 UI를 한 번에 렌더하면 DOM과 React 트리 모두 비용이 커진다.

이번 글은 대형 UI에서 INP를 지키는 방법을 다룬다. Concurrent React는 다음 글에서 따로 다루고, 여기서는 가상화와 long task 분해에 집중한다.

---

## 오늘 풀 문제

상품 관리 페이지에 5000개 상품이 있다. 사용자는 검색어를 입력하고, 정렬을 바꾸고, 스크롤한다. 처음에는 단순하게 구현했다.

```tsx
function ProductList({ products }: { products: Product[] }) {
  return (
    <div>
      {products.map((product) => (
        <ProductCard key={product.id} product={product} />
      ))}
    </div>
  );
}
```

문제는 곧 드러난다.

- 첫 렌더에서 DOM 노드가 너무 많다.
- 검색어 입력마다 5000개 항목을 다시 필터링한다.
- 스크롤이 끊긴다.
- 입력 직후 long task가 생겨 INP가 나빠진다.

이 경우 `memo`만 붙여서는 부족하다. 애초에 렌더할 노드 수를 줄여야 한다.

---

## 정신 모델: 보이는 만큼만 렌더한다

가상화는 전체 리스트를 모두 DOM에 만들지 않는다. 화면에 보이는 구간과 약간의 여유분만 렌더한다.

```txt
전체 데이터: 5000개
실제 DOM: 30개

스크롤 위치가 바뀜
  → 보이는 index 계산
  → 해당 항목만 렌더
  → 전체 높이는 spacer로 유지
```

가상화는 데이터 수를 줄이지 않는다. DOM과 React 렌더 노드 수를 줄인다. 따라서 초기 렌더, 스크롤, 필터 후 재렌더 비용에 효과가 있다.

---

## 최소 예제: `@tanstack/react-virtual`

```tsx
"use client";

import { useRef } from "react";
import { useVirtualizer } from "@tanstack/react-virtual";

type Product = {
  id: string;
  name: string;
  price: number;
};

function ProductRow({ product }: { product: Product }) {
  return (
    <article>
      <h2>{product.name}</h2>
      <p>{product.price.toLocaleString()}원</p>
    </article>
  );
}

export function VirtualProductList({ products }: { products: Product[] }) {
  const parentRef = useRef<HTMLDivElement>(null);

  const rowVirtualizer = useVirtualizer({
    count: products.length,
    getScrollElement: () => parentRef.current,
    estimateSize: () => 80,
    overscan: 5,
  });

  return (
    <div ref={parentRef} style={{ height: 600, overflow: "auto" }}>
      <div
        style={{
          height: rowVirtualizer.getTotalSize(),
          position: "relative",
        }}
      >
        {rowVirtualizer.getVirtualItems().map((virtualRow) => (
          <div
            key={virtualRow.key}
            style={{
              position: "absolute",
              top: 0,
              left: 0,
              width: "100%",
              height: virtualRow.size,
              transform: `translateY(${virtualRow.start}px)`,
            }}
          >
            <ProductRow product={products[virtualRow.index]} />
          </div>
        ))}
      </div>
    </div>
  );
}
```

핵심은 두 가지다.

- 바깥 컨테이너는 고정 높이와 `overflow: auto`를 가진다.
- 안쪽 컨테이너는 전체 리스트 높이를 유지하고, 실제 row만 absolute 위치에 놓는다.

---

## overscan은 안전장치이자 비용이다

`overscan`은 viewport 밖 항목을 미리 몇 개 더 렌더할지 정한다.

- 너무 작으면 빠른 스크롤에서 빈 영역이 보일 수 있다.
- 너무 크면 가상화 이점이 줄어든다.

기본값을 정하고, 실제 스크롤 기록으로 조정해야 한다. “넉넉하게 100개”는 대개 비용을 다시 키운다.

---

## 동적 높이 리스트

항목 높이가 고정이 아니면 추정값과 실제값 차이가 생긴다. 이때 스크롤 위치가 튈 수 있다.

```tsx
const rowVirtualizer = useVirtualizer({
  count: products.length,
  getScrollElement: () => parentRef.current,
  estimateSize: () => 96,
  measureElement:
    typeof window !== "undefined" &&
    navigator.userAgent.indexOf("Firefox") === -1
      ? (element) => element?.getBoundingClientRect().height
      : undefined,
});
```

동적 높이는 구현 비용이 올라간다. 가능하다면 row 높이를 디자인 시스템에서 안정적으로 제한하는 편이 낫다.

---

## INP: 입력 직후 긴 작업을 줄인다

INP는 사용자가 입력한 뒤 다음 화면 업데이트가 보이기까지의 반응성을 본다. 대형 UI에서 INP가 나빠지는 흔한 패턴은 input `onChange` 안에서 무거운 작업을 동기 실행하는 것이다.

```tsx
function BadSearch({ products }: { products: Product[] }) {
  const [query, setQuery] = useState("");
  const [filteredProducts, setFilteredProducts] = useState(products);

  return (
    <input
      value={query}
      onChange={(event) => {
        const value = event.target.value;
        setQuery(value);
        setFilteredProducts(
          products.filter((product) =>
            product.name.toLowerCase().includes(value.toLowerCase()),
          ),
        );
      }}
    />
  );
}
```

입력 값 반영과 무거운 필터링이 같은 이벤트 안에 묶여 있다. products가 크면 사용자는 키 입력 지연을 느낀다.

---

## 작업을 쪼개는 기본 패턴

React의 우선순위 분리는 5편에서 다룬다. 여기서는 브라우저에 제어권을 돌려주는 기본 패턴을 본다.

```tsx
"use client";

import { useCallback, useState } from "react";

type Product = {
  id: string;
  name: string;
};

function waitForNextTask() {
  return new Promise<void>((resolve) => {
    window.setTimeout(resolve, 0);
  });
}

export function SearchableProductList({ products }: { products: Product[] }) {
  const [query, setQuery] = useState("");
  const [filteredProducts, setFilteredProducts] = useState(products);

  const handleChange = useCallback(
    async (event: React.ChangeEvent<HTMLInputElement>) => {
      const value = event.target.value;
      setQuery(value);

      await waitForNextTask();

      const normalizedValue = value.toLowerCase();
      setFilteredProducts(
        products.filter((product) =>
          product.name.toLowerCase().includes(normalizedValue),
        ),
      );
    },
    [products],
  );

  return (
    <>
      <input value={query} onChange={handleChange} />
      <VirtualProductList products={filteredProducts} />
    </>
  );
}
```

이 코드는 입력 상태를 먼저 반영하고, 무거운 필터링을 다음 task로 미룬다. 완전한 해결책은 아니지만 long task를 끊어 입력 반응성을 확보하는 출발점이 된다.

---

## chunk 단위로 처리하기

필터링이나 계산이 매우 크면 한 번에 처리하지 않는다.

```tsx
async function filterInChunks(products: Product[], query: string) {
  const result: Product[] = [];
  const normalizedQuery = query.toLowerCase();
  const chunkSize = 500;

  for (let index = 0; index < products.length; index += chunkSize) {
    const chunk = products.slice(index, index + chunkSize);

    for (const product of chunk) {
      if (product.name.toLowerCase().includes(normalizedQuery)) {
        result.push(product);
      }
    }

    await new Promise<void>((resolve) => {
      window.setTimeout(resolve, 0);
    });
  }

  return result;
}
```

chunk 처리는 총 작업 시간을 줄이지 않는다. 대신 main thread를 오래 독점하지 않게 만든다. 사용자는 중간에 입력, 스크롤, paint를 받을 수 있다.

---

## Web Worker로 옮길 작업

계산이 크고 UI와 독립적이면 Web Worker를 검토한다.

```ts
// product-filter.worker.ts
self.onmessage = (
  event: MessageEvent<{ products: Product[]; query: string }>,
) => {
  const { products, query } = event.data;
  const normalizedQuery = query.toLowerCase();

  const filteredProducts = products.filter((product) =>
    product.name.toLowerCase().includes(normalizedQuery),
  );

  self.postMessage(filteredProducts);
};
```

Worker는 main thread를 비우는 데 좋지만, 데이터 직렬화 비용과 복잡도가 있다. 작은 배열 필터링에 Worker를 쓰면 오히려 비용이 크다.

---

## 드래그와 스크롤

드래그, resize, scroll 이벤트는 짧고 자주 발생한다. 이벤트마다 React state를 갱신하고 큰 트리를 렌더하면 입력 반응성이 무너진다.

실무 기준은 다음이다.

- pointer 위치 같은 초고빈도 값은 ref에 저장한다.
- 화면 반영은 `requestAnimationFrame`에 맞춘다.
- 드래그 중에는 비싼 파생 계산을 멈춘다.
- 드래그 종료 후 최종 상태만 React state로 확정한다.

모든 움직임을 React state로 저장하려는 습관이 대형 UI의 병목이 된다.

## 망하는 패턴

### 가상화 없이 큰 리스트를 렌더한다

1000개 DOM 노드는 단순 HTML이어도 비용이다. 각 row가 이미지, 버튼, tooltip, 이벤트 핸들러를 가지면 비용은 더 커진다.

### input `onChange`에서 큰 계산을 동기 실행한다

입력 이벤트는 urgent하다. 사용자가 입력한 값은 즉시 반영되어야 한다. 무거운 결과 계산과 묶으면 INP가 나빠진다.

### overscan을 과하게 키운다

가상화의 목적은 DOM 노드 수를 제한하는 것이다. overscan이 너무 크면 다시 큰 리스트를 렌더하는 것과 비슷해진다.

### row 높이가 계속 바뀐다

가상화는 위치 계산에 의존한다. row 높이가 이미지 로딩이나 동적 콘텐츠로 계속 변하면 스크롤 점프가 생긴다. 이미지 크기와 row 레이아웃을 안정화해야 한다.

## Vue와 비교

Vue에서도 대형 UI 문제의 처방은 거의 같다.

| 문제            | Vue                       | React                                   |
| --------------- | ------------------------- | --------------------------------------- |
| 긴 리스트       | Vue Virtual Scroller      | `@tanstack/react-virtual`, react-window |
| 큰 계산         | computed 최적화, Worker   | memoization, chunking, Worker           |
| 입력 지연       | watch/debounce, task 분리 | task 분리, transition                   |
| 초고빈도 이벤트 | ref/requestAnimationFrame | ref/requestAnimationFrame               |

프레임워크보다 중요한 것은 DOM 노드 수와 main thread 점유 시간이다. 가상화는 둘 다 줄이기 위한 공통 처방이다.

## 마스터 체크

**Q1. 가상화가 초기 렌더를 빠르게 하는 원리는 무엇인가?**

전체 데이터를 줄이는 것이 아니라 실제 DOM과 React 노드 수를 viewport 주변으로 제한한다. 렌더할 노드 수가 작아져 초기 렌더와 스크롤 비용이 줄어든다.

**Q2. INP와 FID의 차이는 무엇인가?**

FID는 첫 입력의 처리 시작 지연만 봤다. INP는 페이지 전체 상호작용에서 입력부터 다음 paint까지의 반응성을 본다. React 앱에서는 긴 렌더와 long task가 INP에 더 직접적으로 드러난다.

**Q3. Long Task를 쪼개는 방법 두 가지는 무엇인가?**

작업을 chunk로 나누고 task 사이에 브라우저에 제어권을 돌려준다. 또는 UI와 독립적인 큰 계산을 Web Worker로 옮긴다.

**Q4. 가상화를 적용해도 느릴 때 다음에 볼 것은 무엇인가?**

row 렌더 비용, overscan, 이미지 크기, scroll handler, 필터링 계산, state 위치를 본다. 가상화는 DOM 수를 줄일 뿐 row 하나의 비용을 없애지는 않는다.
