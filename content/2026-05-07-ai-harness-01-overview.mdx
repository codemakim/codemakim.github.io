# 프롬프트가 아니라 작업 환경을 설계한다

AI 코딩 도구를 처음 쓰면 프롬프트가 가장 중요해 보인다.

```txt
너는 10년차 시니어 개발자야.
이 코드를 깔끔하게 리팩터링해줘.
테스트도 잘 챙겨줘.
```

이 요청은 한 번의 답변에는 도움이 된다. 하지만 실무에서는 금방 한계가 온다.

- 어떤 문서를 먼저 읽어야 하는가
- 어떤 파일은 절대 건드리면 안 되는가
- 테스트 없이 완료라고 말하면 안 되는가
- GitHub나 DB 쓰기 권한은 어디까지 허용할 것인가
- 큰 작업을 언제 나누고, 언제 직접 처리할 것인가
- 이전 요청의 맥락이 섞였을 때 어떻게 멈출 것인가

이 문제는 문장 하나를 더 잘 쓰는 문제가 아니다. AI가 일하는 작업 환경을 설계하는 문제다.

이 글은 인프런의 [AI를 제어하는 능력 기르기: 하네스 엔지니어링 처음부터 마스터](https://www.inflearn.com/course/mastering-the-basics?cid=341814)에서 출발했지만, 강의 소개의 개념을 그대로 요약하지 않는다. Claude Code 공식 문서, OpenAI Codex 공식 문서, MCP 공식 스펙을 기준으로 다시 정리하고, Codex에서는 각 계층을 어떻게 대응시켜야 하는지까지 같이 다룬다.

---

## 하네스라는 말의 의미

하네스는 힘이 큰 대상을 사람이 다룰 수 있게 묶는 장치다. AI 코딩 도구에서도 같은 문제가 생긴다.

AI는 빠르다.

- 파일을 많이 읽는다.
- 코드를 빠르게 바꾼다.
- 테스트와 빌드를 실행한다.
- GitHub, Gmail, 브라우저, DB 같은 외부 도구와 연결된다.
- 긴 로그를 요약하고 여러 대안을 만든다.

빠르기 때문에 위험도 커진다.

- 사용자 변경을 되돌린다.
- 관련 없는 리팩터링을 섞는다.
- 재현 없이 버그를 고친다.
- 빌드 실패를 확인하지 않고 완료라고 한다.
- 외부 시스템에 넓은 권한으로 접근한다.
- 컨텍스트가 길어져서 이전 작업을 계속 수행한다.

하네스 엔지니어링은 AI를 완전히 결정론적으로 만드는 기술이 아니다. 목표는 더 현실적이다.

```txt
AI가 실수하지 않게 만든다 X

AI가 실수해도 작게 실패하게 한다 O
실패가 검증에서 드러나게 한다 O
사람이 원인을 추적할 수 있게 한다 O
반복 작업을 같은 절차로 수행하게 한다 O
위험한 행동은 승인 없이는 못 하게 한다 O
```

실무에서 중요한 것은 "AI가 똑똑한가"보다 "AI가 어떤 운영 구조 안에서 움직이는가"다.

## 5계층 하네스

하네스는 다섯 계층으로 나눠 설계하면 이해하기 쉽다.

```txt
Layer 1. Memory
  항상 적용되는 프로젝트 규칙

Layer 2. Skills
  필요할 때 불러오는 작업 절차

Layer 3. Hooks / Permissions
  도구 사용 전후 자동 검증과 차단

Layer 4. Subagents
  격리된 전문 작업자

Layer 5. MCP / Plugins
  외부 도구 연결과 팀 배포
```

도구마다 이름은 다르다. Claude Code는 `CLAUDE.md`, `.claude/skills`, hooks, subagents, MCP, plugins라는 이름을 쓴다. Codex는 `AGENTS.md`, skills, sandbox approval, subagents, MCP tools, app connectors, plugins에 가까운 구조를 쓴다.

역할을 맞춰 보면 이렇게 된다.

| 목적               | Claude Code                 | Codex                        |
| ------------------ | --------------------------- | ---------------------------- |
| 항상 적용되는 규칙 | `CLAUDE.md`                 | `AGENTS.md`                  |
| 작업별 절차        | `.claude/skills/*/SKILL.md` | Codex skills의 `SKILL.md`    |
| 위험 명령 차단     | hooks                       | sandbox, approval, 명령 정책 |
| 전문 작업자        | `.claude/agents/*.md`       | subagents                    |
| 외부 도구 연결     | MCP servers                 | MCP tools, app connectors    |
| 팀 배포            | plugins                     | plugins, shared skills       |

이 표에서 중요한 점은 도구 이름이 아니다. 같은 문제가 다른 도구에서 어떤 장치로 해결되는지 보는 것이다.

## 4단계 작업 흐름

계층은 부품이다. 실무에서는 workflow로 묶어야 한다.

```txt
1. Intake
   사용자의 말을 작업 단위로 바꾼다.

2. Planning
   읽을 문서, 수정 파일, 데이터 흐름, 검증 방법을 정한다.

3. Execution
   실제 수정과 도구 실행을 한다.

4. Verification
   타입 체크, 테스트, 빌드, diff, 배포 로그로 완료를 판단한다.
```

AI가 자주 실패하는 지점은 보통 `Planning`과 `Verification`이다. 바로 실행으로 뛰어들면 파일을 잘못 잡고, 검증 없이 완료하면 깨진 상태가 배포된다.

좋은 요청은 이렇게 생겼다.

```txt
이 버그를 고쳐줘.

조건:
- 먼저 재현 방법을 확인한다.
- 관련 파일을 찾기 전에는 수정하지 않는다.
- 수정 후 원래 실패하던 명령을 다시 실행한다.
- 검증 결과를 마지막에 요약한다.
```

더 좋은 구조는 이 요청을 매번 쓰지 않아도 되게 만드는 것이다. Memory와 Skill에 이 규칙을 넣고, 위험한 명령은 permission 계층에서 막는다.

## 왜 공식 문서를 먼저 봐야 하는가

AI 도구는 기능 변화가 빠르다. 특히 Claude Code와 Codex는 memory, hooks, sandbox, MCP, plugin 같은 기능이 계속 바뀐다. 블로그 글이나 강의는 좋은 출발점이지만, 구현할 때는 공식 문서를 기준으로 삼아야 한다.

이 시리즈에서 기준으로 삼는 1차 자료는 다음이다.

- Claude Code 공식 문서: memory, slash commands, hooks, subagents, MCP, plugins
- OpenAI Codex 공식 문서: `AGENTS.md`, sandbox, approval, MCP, hooks
- MCP 공식 스펙: tools, resources, prompts, authorization, security
- AGENTS.md 표준: 여러 AI 코딩 도구가 공유하는 프로젝트 지침 파일

강의나 블로그는 개념을 익히기 좋다. 하지만 팀에 도입할 하네스를 만들 때는 권한, 파일 위치, 실행 조건, 보안 모델을 공식 문서 기준으로 확인해야 한다.

## 예시: "배포했는데 글이 안 보인다"

이번 작업 자체가 좋은 예시다.

처음에는 하네스 관련 문서가 `docs/AI_HARNESS_LEARNING_PLAN.md`로 작성되어 있었다. GitHub Actions 배포도 성공했다. 그런데 사이트에는 글이 보이지 않았다.

원인은 배포 실패가 아니었다.

```txt
GitHub Actions
  성공

문서 위치
  docs/AI_HARNESS_LEARNING_PLAN.md

Contentlayer 수집 대상
  content/*.mdx

블로그 라우트
  /posts/[slug]

결론
  내부 문서는 있었지만 게시용 포스트는 없었다.
```

이 문제는 "AI가 글을 못 썼다"가 아니라 "작업 분류가 부족했다"에 가깝다.

```txt
학습 메모 작성
  docs/*.md 가능

블로그 게시용 포스트 작성
  content/*.mdx 필수
  title/date frontmatter 필수
  Contentlayer build 확인 필수
```

이 규칙이 `AGENTS.md`나 `mdx-post-writing` Skill에 들어가 있으면 같은 실수는 줄어든다.

## 좋은 하네스의 최소 형태

처음부터 완벽한 하네스를 만들 필요는 없다. 최소 형태는 이것이다.

```txt
1. 프로젝트 루트에 AGENTS.md 또는 CLAUDE.md를 둔다.
2. 그 안에 탐색 순서, 금지 명령, 검증 명령을 넣는다.
3. 자주 반복되는 작업 하나를 Skill로 만든다.
4. 외부 write 작업은 승인 없이는 하지 않는다고 정한다.
5. 완료 전 실행할 명령을 작업 유형별로 정한다.
```

예를 들어 이 블로그 프로젝트라면 다음만 있어도 효과가 크다.

```md
## Blog Post Rule

- 게시용 글은 `content/*.mdx`에 작성한다.
- 내부 문서는 `docs/*.md`에 작성한다.
- MDX frontmatter에는 `title`, `date`를 넣는다.
- 새 태그는 3개 이상 글에 쓸 때만 만든다.
- 작성 후 `npm run format`, `npx contentlayer2 build`, `npm run build`를 확인한다.
```

하네스는 거창한 플랫폼이 아니라 이런 작은 규칙에서 시작한다.

## Codex에서는 어떻게 시작하는가

Codex 기준으로는 다음 순서가 실전적이다.

```txt
Step 1. AGENTS.md 정리
  항상 적용할 규칙만 남긴다.

Step 2. Skill 하나 작성
  가장 자주 반복되는 작업을 절차화한다.

Step 3. Approval 기준 문서화
  destructive command와 외부 write 작업을 분리한다.

Step 4. Subagent 기준 정리
  조사, 리뷰, 구현의 책임을 나눈다.

Step 5. MCP read-only부터 연결
  GitHub read, docs read, browser screenshot부터 시작한다.

Step 6. 반복되면 plugin화
  여러 프로젝트에서 같은 절차를 쓸 때만 묶는다.
```

중요한 것은 순서다. Memory 없이 Skill을 만들면 절차가 프로젝트 규칙을 모른다. Approval 기준 없이 MCP를 붙이면 외부 시스템 권한이 위험해진다. Verification 없이 subagent를 늘리면 결과만 많아지고 완료 기준은 흐려진다.

## 체크리스트

- 프롬프트와 하네스의 차이를 설명할 수 있는가?
- AI가 항상 읽어야 하는 규칙과 필요할 때만 읽을 절차를 구분하는가?
- Claude Code의 `CLAUDE.md`와 Codex의 `AGENTS.md`를 대응시킬 수 있는가?
- hooks와 sandbox approval이 해결하는 문제가 무엇인지 아는가?
- subagent를 병렬 처리 도구가 아니라 컨텍스트 격리 도구로 보는가?
- MCP를 read-only부터 연결해야 하는 이유를 설명할 수 있는가?
- 완료를 메시지가 아니라 검증 결과로 판단하는가?
