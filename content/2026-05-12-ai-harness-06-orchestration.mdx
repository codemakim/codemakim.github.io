# Agent Orchestration: 여러 AI 작업자를 하나의 workflow로 묶는다

Subagent를 만들 수 있다고 해서 자동으로 좋은 workflow가 생기지는 않는다. 오히려 여러 agent를 동시에 쓰면 책임이 흐려지고, 같은 파일을 서로 다르게 고치고, 결과를 통합하는 비용이 커질 수 있다.

Orchestration은 여러 AI 작업자를 하나의 작업 흐름으로 묶는 설계다.

```txt
Subagent
  특정 역할을 가진 작업자

Orchestration
  여러 작업자를 어떤 순서로 실행하고,
  어디서 기다리고,
  어떤 결과를 통합하고,
  어디서 사람이 승인할지 정하는 구조
```

핵심은 병렬 실행이 아니다. 책임, 순서, gate, 실패 처리를 설계하는 것이다.

---

## 오케스트레이션이 필요한 순간

작업이 작으면 main agent 하나로 충분하다. 오케스트레이션은 다음 조건에서 필요하다.

- 조사해야 할 영역이 여러 개다.
- 구현 파일 소유권을 분리할 수 있다.
- 리뷰와 구현을 분리해야 한다.
- 외부 도구 read와 write 승인 gate가 필요하다.
- 실패한 agent 결과를 버리거나 재시도해야 한다.
- 여러 결과를 하나의 변경 계획으로 통합해야 한다.

예를 들어 "AI 하네스 시리즈를 강의 커리큘럼 기준으로 다시 짜라"는 작업은 오케스트레이션에 가깝다.

```txt
1. Course Mapper
  강의 페이지의 커리큘럼 항목을 정리한다.

2. Docs Researcher
  Claude Code, Codex, MCP 공식 문서를 확인한다.

3. Series Planner
  현재 글과 빠진 항목을 매핑한다.

4. Writer
  빠진 편을 작성하고 기존 편 번호를 수정한다.

5. Verifier
  Contentlayer, typecheck, build를 확인한다.
```

이 작업을 한 agent가 무작정 시작하면 빠진 항목이 생긴다. 실제로 이전 버전에서는 오케스트레이션이 독립 편으로 빠지지 않았다.

---

## 기본 역할 모델

오케스트레이션에서 가장 먼저 정할 것은 역할이다.

```txt
Main
  작업 전체를 소유한다.
  사용자와 대화하고 최종 결정을 내린다.

Explorer
  읽기 전용 조사자다.
  파일, 로그, 문서, 외부 상태를 조사한다.

Planner
  조사 결과를 작업 순서와 소유권으로 바꾼다.

Worker
  특정 파일 또는 모듈을 수정한다.

Reviewer
  변경 후 버그, 회귀, 누락 테스트를 찾는다.

Verifier
  검증 명령과 산출물을 확인한다.
```

작은 작업에서는 Main이 모든 역할을 직접 해도 된다. 큰 작업에서는 역할을 분리한다.

중요한 규칙은 이것이다.

```txt
Explorer는 수정하지 않는다.
Reviewer는 수정하지 않는다.
Worker는 자기 파일만 수정한다.
Verifier는 검증 결과만 보고한다.
Main은 통합과 최종 판단을 맡는다.
```

이 경계가 없으면 subagent는 편해지는 대신 위험해진다.

## Fan-out / Fan-in 패턴

가장 기본적인 오케스트레이션 패턴은 fan-out / fan-in이다.

```txt
Main
  ↓ fan-out
Explorer A: Contentlayer 흐름 조사
Explorer B: GitHub Actions 상태 조사
Explorer C: 기존 포스트 규칙 조사
  ↓ fan-in
Main: 결과 통합
  ↓
Worker: 필요한 파일 수정
  ↓
Verifier: 검증
```

fan-out은 병렬 조사에 적합하다. 서로 다른 파일이나 외부 시스템을 읽는 작업은 충돌하지 않는다.

fan-in에서 중요한 것은 결과를 그대로 합치지 않는 것이다. Main이 중복, 충돌, 근거 수준을 판단해야 한다.

```txt
Explorer A 결과
  Contentlayer는 content/*.mdx만 Post로 수집한다.

Explorer B 결과
  GitHub Actions는 성공했다.

Explorer C 결과
  content/AGENTS.md는 frontmatter와 문체를 요구한다.

Main 결론
  배포 실패가 아니라 게시 경로 누락이다.
  docs/*.md가 아니라 content/*.mdx 포스트가 필요하다.
```

오케스트레이션의 핵심은 많은 답을 받는 것이 아니라, 여러 근거를 하나의 결론으로 줄이는 것이다.

## DAG로 작업을 본다

복잡한 작업은 목록보다 DAG로 생각하면 좋다. DAG는 방향이 있는 의존성 그래프다.

```txt
요구사항 정리
  ├─ 공식 문서 조사
  ├─ 기존 글 구조 조사
  └─ 강의 커리큘럼 조사
        ↓
시리즈 목차 확정
        ↓
파일별 작성
  ├─ 1편 수정
  ├─ 5편 수정
  ├─ 6편 추가
  ├─ 7편 추가
  └─ 8편 추가
        ↓
검증
  ├─ format
  ├─ contentlayer build
  ├─ typecheck
  └─ static build
```

의존성이 없는 작업은 병렬화할 수 있다. 의존성이 있는 작업은 순차 처리해야 한다.

나쁜 병렬화는 이렇다.

```txt
Worker A: 5편 수정
Worker B: 5편 수정
Worker C: 5편 수정
```

좋은 병렬화는 이렇다.

```txt
Worker A: 6편 Orchestration만 작성
Worker B: 7편 MCP만 작성
Worker C: 8편 Plugins만 작성
```

파일 소유권이 분리되어야 병렬화가 안전하다.

## Gate를 둔다

Gate는 다음 단계로 넘어가기 전에 반드시 통과해야 하는 조건이다.

```txt
Planning Gate
  수정 범위와 파일 소유권이 정해졌는가?

Write Gate
  외부 write 또는 destructive command가 필요한가?

Merge Gate
  여러 worker 결과가 충돌하지 않는가?

Verification Gate
  필요한 검증 명령이 통과했는가?

Release Gate
  push, comment, deploy 같은 외부 write가 승인됐는가?
```

Gate가 없으면 workflow는 단순한 작업 목록이 된다. 하네스는 gate를 통해 작업을 통제한다.

예를 들어 GitHub PR 코멘트를 반영하는 workflow는 이렇게 둔다.

```txt
1. Read Gate
  unresolved comments를 읽는다.

2. Planning Gate
  actionable comment와 non-actionable comment를 나눈다.

3. Write Gate
  코드 수정 범위를 정한다.

4. Verification Gate
  테스트와 빌드를 통과한다.

5. External Write Gate
  댓글 작성 또는 push는 사용자 승인 후 실행한다.
```

Codex에서는 approval 요청이 gate 역할을 한다. Claude Code에서는 hooks와 permission decision이 gate 역할을 할 수 있다.

## 실패 처리: retry, discard, escalate

오케스트레이션에는 실패 처리가 필요하다.

```txt
Retry
  네트워크 일시 실패, flaky test, rate limit처럼 다시 시도할 가치가 있는 경우

Discard
  subagent 결과가 범위를 벗어났거나 근거가 약한 경우

Escalate
  권한, 보안, 외부 write, 요구사항 충돌처럼 사람 판단이 필요한 경우
```

모든 실패를 retry하면 위험하다.

```txt
빌드가 Google Fonts fetch로 실패
  네트워크 승인 후 retry 가능

테스트가 실제 회귀로 실패
  retry보다 원인 분석 필요

GitHub write 권한 필요
  retry가 아니라 user approval 필요

DB migration 실패
  자동 retry 금지, 사람 판단 필요
```

실패 처리도 하네스의 일부다.

## Codex에서 오케스트레이션하기

Codex에서 subagent를 쓸 때는 main agent가 먼저 critical path를 잡아야 한다.

```txt
1. 지금 당장 내가 해야 하는 blocking task는 무엇인가?
2. 병렬로 조사해도 되는 side task는 무엇인가?
3. 어떤 agent가 어떤 파일을 읽거나 쓸 것인가?
4. 결과를 언제 기다릴 것인가?
5. 결과를 어떻게 통합할 것인가?
```

좋은 Codex 오케스트레이션 지시는 이렇게 생겼다.

```txt
Explorer A:
강의 페이지 커리큘럼 항목만 정리한다.
코드는 수정하지 않는다.

Explorer B:
현재 content/ai-harness 시리즈의 제목, seriesOrder, 누락 항목을 정리한다.
코드는 수정하지 않는다.

Main:
두 결과를 비교해서 새 목차를 확정한다.

Worker:
확정된 목차에 따라 지정된 파일만 수정한다.

Verifier:
contentlayer build, typecheck, build 결과만 확인한다.
```

작업이 작을 때는 실제 subagent를 쓰지 않고도 이 구조를 main agent가 내부 workflow로 따라갈 수 있다. 중요한 것은 "누가 무엇을 소유하는지"를 먼저 정하는 것이다.

## Claude Code에서는

Claude Code에서는 subagents, hooks, skills를 조합해 오케스트레이션을 구성한다.

```txt
Skill
  workflow 절차를 정의한다.

Subagent
  조사자, 리뷰어, 작업자 역할을 정의한다.

Hook
  위험 명령과 gate를 강제한다.

MCP
  외부 read/write를 제공한다.

Plugin
  이 구조를 팀에 배포한다.
```

예를 들어 release workflow는 이렇게 구성할 수 있다.

```txt
release-check Skill
  릴리스 절차를 정의한다.

release-notes agent
  변경 요약을 만든다.

ci-log analyst
  실패한 Actions 로그를 분석한다.

PreToolUse hook
  위험한 git 명령을 막는다.

MCP GitHub read
  PR과 Actions 상태를 읽는다.

approval gate
  push와 release publish 전에 사람 승인을 받는다.
```

오케스트레이션은 단일 기능이 아니라 여러 하네스 계층을 묶는 운영 방식이다.

## 예시: 블로그 시리즈 재구성

이번 작업을 오케스트레이션으로 보면 이렇게 된다.

```txt
Input
  강의 커리큘럼에 있는 오케스트레이션이 빠진 것 같다.

Intake
  현재 시리즈가 강의 커리큘럼을 빠짐없이 반영했는지 확인한다.

Explore
  강의 페이지에서 커리큘럼 구조를 확인한다.
  현재 포스트의 제목과 seriesOrder를 확인한다.

Plan
  6편을 8편으로 재편한다.
  5편은 Subagents, 6편은 Orchestration, 7편은 MCP, 8편은 Plugins로 둔다.

Execute
  제목 번호를 수정한다.
  Orchestration 편을 추가한다.
  MCP/Plugins 편을 분리한다.

Verify
  Prettier
  Contentlayer
  TypeScript
  Build
```

이 흐름으로 진행하면 "빠진 항목이 있었는지"를 작업 전에 잡을 수 있다.

## 체크리스트

- 오케스트레이션을 단순 병렬 실행으로 착각하지 않는가?
- Main, Explorer, Worker, Reviewer, Verifier 역할을 구분했는가?
- 파일 write ownership을 agent별로 분리했는가?
- fan-out 후 fan-in에서 main agent가 결과를 판단하는가?
- 작업 DAG에서 병렬 가능한 작업과 순차 작업을 구분했는가?
- approval gate와 verification gate가 있는가?
- 실패를 retry, discard, escalate로 나누는가?
- 외부 write는 항상 사람 승인 뒤에 실행하는가?
