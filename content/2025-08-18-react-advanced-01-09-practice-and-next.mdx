# React Router 가이드 - 실전 예제 & 다음 단계

1~8편에서는 React Router의 핵심 조각을 따로 배웠다.

- URL과 컴포넌트 연결
- 중첩 라우트와 `<Outlet />`
- params와 query string
- `useNavigate()`와 router hooks
- protected route
- 로딩과 에러 상태
- route config와 구조 분리

이 글에서는 작은 블로그 앱을 기준으로 이 조각을 하나의 라우팅 구조로 묶는다. 목표는 화려한 기능이 아니라 "이 정도 구조면 실무 앱의 첫 버전으로 가져갈 수 있다"는 기준을 잡는 것이다.

---

## 왜 필요한가

라우팅 학습에서 가장 흔한 문제는 API는 아는데 실제 앱 구조가 그려지지 않는 것이다.

`Route`, `Link`, `useParams`, `useSearchParams`, `useNavigate`를 각각 이해해도 다음 질문이 남는다.

- 목록 필터는 state에 둘까, URL에 둘까
- 상세 화면의 없는 글은 404 페이지로 보낼까, 화면 안에서 처리할까
- 글 작성 페이지는 어디에서 보호할까
- 레이아웃과 본문 로딩은 어디서 나눌까
- 라우트 파일이 커지면 어떻게 나눌까

작은 블로그 앱은 이 질문을 모두 포함한다. 그래서 React Router의 기본기를 통합하기 좋다.

---

## 요구사항

만들 앱의 라우트는 다음과 같다.

| URL                   | 화면      | 특징                                       |
| --------------------- | --------- | ------------------------------------------ |
| `/`                   | 홈        | 최근 글 링크                               |
| `/posts`              | 글 목록   | `?tag=react&page=1` 같은 query string 사용 |
| `/posts/:postId`      | 글 상세   | params 사용, 리소스 404 처리               |
| `/posts/new`          | 글 작성   | 로그인 필요                                |
| `/posts/:postId/edit` | 글 수정   | 로그인 필요                                |
| `/login`              | 로그인    | 로그인 후 원래 경로 복귀                   |
| `*`                   | Not Found | 라우트 404                                 |

이 요구사항만 구현해도 앞선 글의 대부분을 실제로 사용한다.

---

## 라우트 구조부터 잡는다

먼저 URL 구조를 한 곳에 모은다. 예제는 `useRoutes()` 기반 route config를 사용한다.

```tsx
import { lazy, Suspense, type ReactNode } from "react";
import {
  BrowserRouter,
  Link,
  Navigate,
  Outlet,
  type RouteObject,
  useLocation,
  useRoutes,
} from "react-router-dom";

const HomePage = lazy(() => import("./pages/HomePage"));
const LoginPage = lazy(() => import("./pages/LoginPage"));
const PostListPage = lazy(() => import("./pages/PostListPage"));
const PostDetailPage = lazy(() => import("./pages/PostDetailPage"));
const PostCreatePage = lazy(() => import("./pages/PostCreatePage"));
const PostEditPage = lazy(() => import("./pages/PostEditPage"));
const NotFoundPage = lazy(() => import("./pages/NotFoundPage"));

function useAuth() {
  return { user: null as { id: string } | null, loading: false };
}

function RequireAuth({ children }: { children: ReactNode }) {
  const { user, loading } = useAuth();
  const location = useLocation();

  if (loading) {
    return <main aria-busy="true">인증 상태를 확인하는 중이다.</main>;
  }

  if (!user) {
    return <Navigate to="/login" replace state={{ from: location }} />;
  }

  return <>{children}</>;
}

function Layout() {
  return (
    <>
      <header>
        <Link to="/">홈</Link>
        <Link to="/posts">글 목록</Link>
      </header>
      <Suspense fallback={<main>본문을 불러오는 중이다.</main>}>
        <Outlet />
      </Suspense>
    </>
  );
}

const routes: RouteObject[] = [
  {
    element: <Layout />,
    children: [
      { index: true, element: <HomePage /> },
      { path: "login", element: <LoginPage /> },
      { path: "posts", element: <PostListPage /> },
      { path: "posts/:postId", element: <PostDetailPage /> },
      {
        path: "posts/new",
        element: (
          <RequireAuth>
            <PostCreatePage />
          </RequireAuth>
        ),
      },
      {
        path: "posts/:postId/edit",
        element: (
          <RequireAuth>
            <PostEditPage />
          </RequireAuth>
        ),
      },
      { path: "*", element: <NotFoundPage /> },
    ],
  },
];

function AppRoutes() {
  return useRoutes(routes);
}

export default function App() {
  return (
    <BrowserRouter>
      <AppRoutes />
    </BrowserRouter>
  );
}
```

이 구조에서 `Layout`은 공통 UI와 본문 로딩 경계를 담당한다. 페이지 컴포넌트는 자기 화면의 데이터와 상호작용에 집중한다.

## 목록 화면: query string을 단일 진실 소스로 둔다

글 목록의 태그와 페이지 번호는 URL에 두는 편이 좋다. 새로고침, 공유, 뒤로가기에서 같은 목록 상태가 복원되기 때문이다.

```tsx
import { Link, useSearchParams } from "react-router-dom";

const posts = [
  { id: "1", title: "React Router 시작", tag: "react" },
  { id: "2", title: "Vue와 라우팅 비교", tag: "vue" },
];

export function PostListPage() {
  const [searchParams, setSearchParams] = useSearchParams();
  const selectedTag = searchParams.get("tag") ?? "all";
  const page = Number(searchParams.get("page") ?? "1");

  const visiblePosts = posts.filter((post) => {
    return selectedTag === "all" || post.tag === selectedTag;
  });

  function updateTag(tag: string) {
    const next = new URLSearchParams(searchParams);

    if (tag === "all") {
      next.delete("tag");
    } else {
      next.set("tag", tag);
    }

    next.set("page", "1");
    setSearchParams(next);
  }

  function goToPage(nextPage: number) {
    const next = new URLSearchParams(searchParams);
    next.set("page", String(Math.max(1, nextPage)));
    setSearchParams(next);
  }

  return (
    <main>
      <h1>글 목록</h1>
      <button type="button" onClick={() => updateTag("all")}>
        전체
      </button>
      <button type="button" onClick={() => updateTag("react")}>
        React
      </button>
      <button type="button" onClick={() => updateTag("vue")}>
        Vue
      </button>

      <ul>
        {visiblePosts.map((post) => (
          <li key={post.id}>
            <Link to={`/posts/${post.id}`}>{post.title}</Link>
          </li>
        ))}
      </ul>

      <button type="button" onClick={() => goToPage(page - 1)}>
        이전
      </button>
      <span>{page}페이지</span>
      <button type="button" onClick={() => goToPage(page + 1)}>
        다음
      </button>
    </main>
  );
}
```

여기서 `selectedTag`와 `page`를 별도 React state로 또 저장하지 않는다. URL을 읽어서 화면을 만든다. URL과 state가 동시에 존재하면 둘 중 하나가 오래된 값이 되기 쉽다.

## 상세 화면: params와 리소스 404를 분리한다

상세 화면은 `postId` params를 읽는다. route는 맞지만 글이 없을 수 있으므로, 이 상태는 페이지 내부에서 처리한다.

```tsx
import { Link, useParams } from "react-router-dom";

const posts = [
  { id: "1", title: "React Router 시작", body: "라우팅의 기본을 배운다." },
  { id: "2", title: "Vue와 라우팅 비교", body: "두 라우터의 관점을 비교한다." },
];

export function PostDetailPage() {
  const { postId } = useParams();
  const post = posts.find((item) => item.id === postId);

  if (!post) {
    return (
      <main>
        <h1>글을 찾을 수 없다</h1>
        <p>주소는 맞지만 해당 id의 글이 없다.</p>
        <Link to="/posts">목록으로 돌아간다</Link>
      </main>
    );
  }

  return (
    <main>
      <h1>{post.title}</h1>
      <p>{post.body}</p>
      <Link to={`/posts/${post.id}/edit`}>수정한다</Link>
    </main>
  );
}
```

`/wrong-url`은 라우트 404다. `/posts/999`는 리소스 404다. 둘을 같은 Not Found 페이지로 보내도 되지만, 내부적으로는 원인을 구분해서 처리해야 디버깅과 UX가 좋아진다.

## 작성 화면: 작업 성공 후 이동한다

작성 화면은 링크 이동이 아니라 작업 결과에 따른 이동이 필요하다. 저장에 성공하면 새 글 상세로 이동하고, 실패하면 현재 화면에 에러를 보여준다.

```tsx
import { FormEvent, useState } from "react";
import { useNavigate } from "react-router-dom";

export function PostCreatePage() {
  const navigate = useNavigate();
  const [title, setTitle] = useState("");
  const [error, setError] = useState<string | null>(null);

  async function handleSubmit(event: FormEvent<HTMLFormElement>) {
    event.preventDefault();

    const trimmedTitle = title.trim();
    if (!trimmedTitle) {
      setError("제목을 입력해야 한다.");
      return;
    }

    try {
      const createdPost = { id: "3", title: trimmedTitle };
      navigate(`/posts/${createdPost.id}`, { replace: true });
    } catch {
      setError("글을 저장하지 못했다.");
    }
  }

  return (
    <main>
      <h1>글 작성</h1>
      <form onSubmit={handleSubmit}>
        <label>
          제목
          <input
            value={title}
            onChange={(event) => setTitle(event.target.value)}
          />
        </label>
        {error && <p role="alert">{error}</p>}
        <button type="submit">저장한다</button>
      </form>
    </main>
  );
}
```

저장 성공 후 `replace: true`를 쓰면 작성 화면이 히스토리에 남지 않는다. 뒤로가기를 눌렀을 때 작성 폼으로 돌아가는 것이 자연스럽지 않은 흐름이라면 `replace`가 맞다.

## 로그인 후 원래 화면으로 돌아간다

Protected Route에서 로그인 페이지로 보낼 때 `state.from`에 원래 위치를 담았다. 로그인 화면은 그 값을 읽어 성공 후 복귀할 수 있다.

```tsx
import { FormEvent } from "react";
import { useLocation, useNavigate } from "react-router-dom";

interface LocationState {
  from?: {
    pathname?: string;
    search?: string;
  };
}

function getSafeReturnPath(state: unknown) {
  const locationState = state as LocationState | null;
  const pathname = locationState?.from?.pathname;
  const search = locationState?.from?.search ?? "";

  if (!pathname || !pathname.startsWith("/") || pathname.startsWith("//")) {
    return "/";
  }

  return `${pathname}${search}`;
}

export function LoginPage() {
  const location = useLocation();
  const navigate = useNavigate();

  async function handleSubmit(event: FormEvent<HTMLFormElement>) {
    event.preventDefault();
    const returnPath = getSafeReturnPath(location.state);

    navigate(returnPath, { replace: true });
  }

  return (
    <main>
      <h1>로그인</h1>
      <form onSubmit={handleSubmit}>
        <button type="submit">로그인한다</button>
      </form>
    </main>
  );
}
```

복귀 경로는 반드시 내부 경로인지 확인한다. 외부 URL이나 `//evil.example` 같은 값을 그대로 이동에 쓰면 open redirect 문제가 된다.

## 함정과 디버깅

### 1. 목록 필터를 state와 URL에 동시에 둔다

둘 중 하나만 진실이어야 한다. 공유와 새로고침이 필요한 목록 상태라면 URL query string을 기준으로 둔다.

### 2. 작성 성공 후 무조건 `navigate(-1)`을 쓴다

사용자가 작성 URL로 직접 들어왔을 수도 있다. 작업 성공 후 목적지가 분명하다면 `/posts/:postId`처럼 명시적인 경로로 이동한다.

### 3. 보호 라우트에서 로딩 상태를 생략한다

인증 복구 중에 `user`가 잠깐 없다고 바로 로그인으로 보내면 화면이 깜빡인다. `loading`, `anonymous`, `authenticated` 상태를 분리한다.

### 4. 리소스 404를 라우트 404로 착각한다

`/posts/:postId`가 매칭됐다면 라우터는 할 일을 했다. 글이 없는 문제는 상세 페이지의 데이터 문제다.

## Vue Router와 비교

Vue Router로 같은 블로그 앱을 만들면 route config, route params, query, navigation guard를 먼저 떠올리게 된다. React Router도 같은 문제를 풀지만 컴포넌트 합성으로 표현하는 비중이 크다.

| 요구사항      | Vue Router                               | React Router                 |
| ------------- | ---------------------------------------- | ---------------------------- |
| 목록 필터     | `route.query` / `router.push({ query })` | `useSearchParams()`          |
| 상세 id       | `route.params.postId`                    | `useParams().postId`         |
| 코드 이동     | `router.push()`                          | `useNavigate()`              |
| 보호 라우트   | navigation guard / route meta            | `<RequireAuth>` 합성         |
| 공통 레이아웃 | nested routes + `<RouterView />`         | nested routes + `<Outlet />` |
| 라우트 404    | catch-all route                          | `path="*"`                   |

Vue Router의 guard 중심 사고에 익숙하다면 React Router에서는 "이 route element를 어떤 컴포넌트로 감쌀 것인가"로 바꿔 생각하면 된다.

## 다음 단계: Data Router로 넘어가는 이유

1~9편은 컴포넌트 라우터 관점에서 SPA 라우팅의 핵심을 다뤘다. 여기까지 이해하면 대부분의 기본 라우팅 요구사항을 만들 수 있다.

다음 10~12편에서는 Data Router로 넘어간다. Data Router는 라우트 정의에 데이터 읽기, 폼 제출, 에러 경계를 더 가깝게 붙인다.

- 10편: `createBrowserRouter`, `loader`, `redirect`, `errorElement`
- 11편: `action`, `<Form>`, `useNavigation`, `useFetcher`
- 12편: 정적 호스팅, basename, 테스트, 배포 이슈

Data Router가 필요한 이유는 단순하다. 페이지가 열리기 전에 필요한 데이터, 제출 중 상태, 라우트 단위 에러를 컴포넌트 안쪽에 흩어두지 않고 라우터 경계에서 다루기 위해서다.

---

## 체크리스트

- 작은 블로그 앱의 URL 구조를 직접 설계할 수 있다.
- 목록 필터와 페이지 번호를 query string으로 관리할 수 있다.
- 상세 화면에서 params와 리소스 404를 분리해 처리할 수 있다.
- 작성 성공 후 명시적인 목적지로 이동할 수 있다.
- protected route에서 로그인 후 원래 경로로 돌아갈 수 있다.
- 복귀 경로 검증으로 open redirect를 방지할 수 있다.
- Data Router로 넘어가는 이유를 설명할 수 있다.
